# 前端数据可视化与复杂交互架构

数据可视化不是“把数组传给图表库”。真正困难的是把业务语义、数学映射、浏览器渲染、连续交互、实时数据、可访问性和性能治理组合成一个长期可维护的系统。

你有 Vue 2 项目经验，应该已经用过 ECharts 等图表库。这一课不重复配置项，而是回答更底层的问题：图表为何会错、为何会卡、为何鼠标能用但键盘不能用，以及怎样设计出可替换渲染器、可测试、可观测的图表模块。

## 学习目标

完成本课后，你应该能够：

- 把数据可视化拆成领域数据、比例尺、几何、场景、渲染和交互层；
- 根据数据量、交互和可访问性要求选择 HTML、SVG、Canvas、WebGL 或混合方案；
- 正确处理坐标系、时间、单位、缺失值、尺寸与高分屏；
- 设计缩放、平移、命中测试、键盘操作和选择状态；
- 为实时流建立有界缓冲、背压和按帧调度；
- 用摘要、结构化表格和键盘路径补足 Canvas/WebGL 的语义缺口；
- 建立性能预算、视觉回归和资源生命周期治理。

## 从领域事实到可逆的屏幕几何

先不要选择图表库或渲染技术。一张图能否正确交互，取决于它是否保留了从领域值映射到屏幕、再从屏幕还原到领域值的完整链路；渲染器只是这条链路的最后一环。

### 先建立“语义到像素”的心智模型

一张折线图至少经过下面的流水线：

```text
业务记录
  → 校验与语义归一化
  → 领域模型（时间、数值、单位、缺失规则）
  → 比例尺（数据域 → 屏幕范围）
  → 几何图元（点、线、矩形、区域）
  → 渲染器（SVG / Canvas / WebGL）
  → 交互解释（屏幕坐标 → 数据语义）
  → 可访问替代与观测指标
```

图表库通常把这些步骤包在一个配置对象中，但步骤并没有消失。若领域层把秒级时间戳当成毫秒，渲染器仍会忠实地画出错误结果；若比例尺不可逆，指针位置就无法可靠地还原成数据值。

#### 数据图形不是领域事实

像素坐标只是领域数据在某个视口中的投影。不要把 `{ x: 123, y: 78 }` 当作业务状态持久化。窗口变化、缩放或字体变化后，它就失效了。

推荐分开保存：

- 领域事实：时间戳、金额、单位、分类、质量标记；
- 视图状态：可见时间域、选中项、焦点项、图例开关；
- 派生几何：屏幕坐标、路径字符串、刻度位置；
- 瞬时输入：指针当前位置、拖拽起点。

派生几何应从前两者重新计算，不应成为第二份事实来源。

#### “没有数据”和“数值为零”不同

下面几种输入不能都用 `0` 表示：

- 真实测量值是零；
- 设备未上报；
- 请求失败；
- 数值被隐私规则抑制；
- 当前筛选条件下没有记录。

将缺失值静默转成零会制造不存在的骤降。领域适配层必须定义：跳过、断线、插值、显示质量标记，还是拒绝渲染。示例选择拒绝非有限数值，避免错误被藏到像素层：

### 分层：让业务、数学与浏览器互不绑死

大型项目可采用以下边界：

| 层 | 负责 | 不负责 |
| --- | --- | --- |
| 领域适配 | 单位、时区、缺失值、权限、聚合语义 | DOM 和像素 |
| 数据模型 | 校验、排序、不可变快照、数据域 | 颜色和动画 |
| 比例尺/布局 | 数据域到像素范围、刻度、图例布局 | 网络请求 |
| 场景/几何 | 点线面、标签候选、命中索引 | 框架组件生命周期 |
| 渲染器 | SVG 节点、Canvas 指令或 GPU 资源 | 业务规则 |
| 交互控制器 | 焦点、选择、缩放、平移、手势解释 | 直接改业务数据 |
| 框架适配 | Vue/React props、生命周期、插槽 | 重写比例尺算法 |

这不是为了增加文件，而是为了获得三个能力：

1. 用纯函数测试最容易出错的数学逻辑；
2. SVG 性能不足时替换成 Canvas，而不重写领域层；
3. Vue 和 React 可以共享核心包，仅保留薄适配层。

### 坐标系与比例尺：图表的数学核心

#### 数据域与显示范围

线性比例尺把数据域 `[d0, d1]` 映射到像素范围 `[r0, r1]`：

```text
t = (value - d0) / (d1 - d0)
pixel = r0 + t × (r1 - r0)
```

反向映射则用于 tooltip、框选和缩放锚点：

```text
value = d0 + (pixel - r0) / (r1 - r0) × (d1 - d0)
```

完整实现包含有限值校验、零跨度保护、反向范围和刻度：

<<< ../../../examples/frontend/data-visualization/scales.ts

测试同时覆盖正常、反向坐标和失败路径：

生产项目通常使用 D3 scale 等成熟实现，因为“好看”的刻度还涉及 1、2、5 步长、对数域、时间间隔和本地化。本例的意义是理解不变量，而不是重新实现整个刻度系统。

#### 为什么 Y 轴通常反向

笛卡尔坐标中 Y 向上增大；浏览器 CSS 像素从左上角开始，Y 向下增大。因此把数据最小值映射到图框底部，把最大值映射到顶部：

<<< ../../../examples/frontend/data-visualization/chart-model.ts

图框 `PlotRect` 将坐标轴、标题和留白排除在实际绘图区外。指针坐标必须先减去元素边界，再与相同图框比较；混用页面坐标、视口坐标和 Canvas 内部坐标会导致滚动后 tooltip 偏移。

#### 不同数据需要不同比例尺

- 连续数值：线性、对数、幂或分段比例尺；
- 时间：保留时间语义的时间比例尺；
- 离散分类：band/point 比例尺；
- 颜色：顺序、发散或分类色阶；
- 地理数据：投影不是普通的经纬度线性缩放。

对数比例尺不能直接接受零和负数；发散色阶应有有意义的中点，例如盈亏的零。选择比例尺本身就是领域决策。

#### 时间不是普通数字

时间轴至少要明确：

- 输入是 Unix 秒还是毫秒；
- 聚合按 UTC 还是业务时区；
- “每天”是否受夏令时影响；
- 服务器返回的是瞬时时间还是没有时区的日历时间；
- tooltip 与刻度采用哪种 locale 和时区。

不要用固定 `24 * 60 * 60 * 1000` 推导所有“下一天”。遇到跨时区业务，应先用领域认可的时间边界聚合，再将确定的时间戳交给图表。

## 按真实瓶颈选择渲染器

领域模型、比例尺和几何保持独立后，渲染器才真正可替换。选择标准不是“点数超过某个神奇阈值”，而是当前设备上有多少图元、每帧改变多少、是否需要 DOM 语义，以及命中测试和像素填充各花多少成本。

没有一个固定点数阈值适用于所有设备。选择取决于图元数量、每帧更新量、样式复杂度、命中测试和目标硬件。

| 技术 | 优势 | 代价 | 常见场景 |
| --- | --- | --- | --- |
| HTML/CSS | 原生语义、布局和焦点 | 大量节点昂贵 | 表格、少量 KPI、图例 |
| SVG | 矢量、DOM 事件、样式和无损缩放 | 大量节点的布局/绘制成本高 | 中小规模图表、流程图 |
| Canvas 2D | 单画布批量绘制、API 直接 | 位图无节点语义，需手动命中 | 大量点线、波形、编辑器 |
| WebGL | GPU 并行、可处理大量图元 | 着色器、缓冲、上下文和兼容性复杂 | 大规模散点、3D、地图 |
| 混合 | 各层使用合适技术 | 坐标和生命周期要统一 | Canvas 数据层 + SVG/HTML 交互层 |

### SVG 的真实边界

SVG 元素进入 DOM，可用 CSS、事件和可访问名称；但“一万个点等于一万个 `<circle>`”会给样式计算、布局、绘制和事件处理施压。折线通常应是一条 `<path>`，而非每段一个节点。

### Canvas 是保留模式与立即模式的分界

DOM/SVG 大致是保留模式：你声明节点，浏览器保存场景。Canvas 2D 是立即模式：调用绘制命令后，浏览器只留下像素，不记得哪个像素属于哪个点。

因此 Canvas 应用必须自己保留：

- 当前数据与几何；
- 重绘触发条件；
- 命中测试索引；
- 焦点、选择和无障碍语义；
- resize、事件和动画帧的清理逻辑。

### WebGL 不只是“更快的 Canvas”

WebGL 的性能来自把顶点和属性批量上传到 GPU，再由着色器并行处理。它带来新的系统约束：

- GPU 缓冲、纹理、程序和帧缓冲需要显式创建与释放；
- draw call、状态切换和 CPU/GPU 同步都可能成为瓶颈；
- 设备限制和扩展能力不同，必须查询而非猜测；
- 上下文可能丢失，需要停止提交、重建资源并恢复；
- 高分屏会按像素面积放大显存和填充成本。

如果五千个点的 SVG 已满足性能目标，不要仅因“GPU 更高级”引入 WebGL。复杂性必须由测量结果支付。

### 混合架构通常最实用

常见组合是：

```text
底层 Canvas/WebGL：大批量数据图元
中层 SVG：坐标轴、标注、选择框
上层 HTML：tooltip、菜单、表格入口、可访问控件
```

三层必须共享同一个数据域、图框和比例尺。不要让 Canvas 与 SVG 各自测量边距，否则缩放后标注会漂移。

## 把交互与可访问性设计成同一套能力

渲染完成不等于用户能够理解和操作。Hover、键盘焦点、持久选择、视口和辅助技术播报必须来自同一份可推理状态；否则鼠标、键盘和数据表会各自形成不同事实。

### 交互不是事件回调，而是状态机

#### 区分 hover、focus 与 selection

- hover：指针短暂停留，设备相关、易消失；
- focus：键盘当前操作位置；
- selection：用户明确确认、应持续存在；
- viewport：当前缩放和平移后的数据域。

把它们都塞进一个 `activeIndex` 会造成鼠标移开即丢失选择，或键盘焦点被 pointermove 抢走。示例用可判别联合描述动作：

<<< ../../../examples/frontend/data-visualization/interaction-state.ts

框架组件只负责把 pointer、wheel、keydown 转换成 action，再根据新状态渲染。这样交互规则可以脱离 DOM 测试，也便于撤销、回放和协作同步。

#### 命中测试决定交互成本

SVG 可以利用目标节点，但 Canvas/WebGL 只有屏幕坐标。最直接的做法是每次 pointermove 扫描所有点，复杂度为 `O(n)`；在高频事件和大数据下会迅速超预算。

二维散点可使用网格、四叉树或 R-tree。还应设置像素命中半径：数学上最近不等于用户视觉上真的指向了它。

#### 缩放与平移应修改数据域

可靠做法是让交互更新 `visibleDomain`，然后从领域数据重新投影。只对最终像素做 CSS transform 虽然视觉上移动了，却可能导致坐标轴、tooltip 反算和命中索引不同步。

以指针对应的数据值 `anchor` 为锚点缩放：

```text
newMin = anchor + (oldMin - anchor) / factor
newMax = anchor + (oldMax - anchor) / factor
```

还要建立约束：最小/最大跨度、允许越界范围、恢复按钮、wheel 默认行为和触摸手势冲突。只有在图表确实处理手势时才阻止默认滚动。

#### 键盘路径必须独立可用

推荐至少提供：

- Tab 聚焦图表控制区，而非让成千上万个点全部进入 Tab 顺序；
- 左右键在时间序列中移动焦点；
- 上下键在系列间移动；
- Enter/Space 确认选择；
- `+`/`-` 缩放，方向键或明确控件平移；
- Escape 退出临时模式；
- “重置视图”和“查看数据表”按钮。

键盘操作后的当前值可通过一个节制更新的 live region 宣布。不要在每个动画帧播报，否则会淹没辅助技术用户。

### 不要把像素假装成语义

Canvas 和 WebGL 的像素本身不进入无障碍树。给 `<canvas>` 加一句 `aria-label="折线图"` 只能给出名称，不能让用户理解趋势、极值和每个点。

一个可用方案通常包含四层：

1. 图表前后的文字摘要：指标、时间范围、总体趋势、异常点；
2. 可访问名称和描述：说明图的目的，而非重复标题；
3. 键盘可操作的筛选、缩放、选择与数据浏览；
4. 等价数据表或下载方式，保留表头、单位和时间语义。

结构化表格示例显式接收 Locale、Time Zone、列标题和数字格式，避免“可访问替代”反而使用服务器或浏览器的隐式默认环境：

<<< ../../../examples/frontend/data-visualization/accessible-table.ts

不要为了“语义化”在屏幕外创建十万个隐藏按钮，这会把性能问题转移到无障碍树。大数据应提供聚合摘要、分页/虚拟化数据表或按需查询。

#### 颜色与感知

- 不以颜色作为唯一编码，同时使用标签、形状、纹理或线型；
- 文本、轴线、焦点指示满足对比度要求；
- 颜色顺序符合数据语义，分类色之间可区分；
- 动画尊重 `prefers-reduced-motion`；
- tooltip 不能只在 hover 出现，也要能由 focus/selection 触发；
- 放大或系统字体变化后，标签和控件仍可读。

## 让实时渲染管线保持有界

静态图只需正确地投影一次，实时图还要回答输入速度持续高于绘制速度时怎么办。核心不是“画得更快”，而是限制内存、合并更新，并让降采样规则保留业务真正关心的信号。

### 实时数据：先控制输入，再优化绘制

假设服务端每秒推送一万条数据，而屏幕每秒最多显示约 60 帧。为每条消息触发响应式更新和重绘，系统必然积压。

正确的实时管线是：

```text
网络消息 → 校验/解码 → 有界缓冲 → 聚合或降采样
                           ↓
                    下一动画帧取快照 → 投影 → 绘制
```

#### 有界内存

流式图表必须明确保留窗口：最近 N 点、最近五分钟，或分层聚合。无限 `array.push` 不是实时架构。

#### 背压与降采样不是丢掉正确性

应根据目的选择策略：

- 仪表盘趋势：按像素列聚合 min/max/first/last；
- 审计数据：原始数据保存在服务端，前端只取可见窗口；
- 告警：关键事件走独立通道，不可被普通采样覆盖；
- 波形：使用能保留峰谷的降采样，而非随意每 N 个取一个。

降采样规则属于数据语义，应可追踪。tooltip 必须说明展示的是原值还是聚合值。

#### 避免响应式系统代理巨量数据

在 Vue/React 中，百万点数组通常不需要每个元素深度响应：

- 保存不可变快照或浅引用；
- 每批次替换引用，而不是逐点触发更新；
- 将绘图数据放在框架渲染树之外；
- 框架状态只保存版本号、视口和选择等小状态。

这不是绕过框架，而是承认像素流水线和声明式 DOM 更新具有不同成本模型。

### Worker、OffscreenCanvas 与数据传输

当数据解码、布局或 Canvas 绘制形成主线程长任务时，可考虑 Worker。先用性能面板确认瓶颈，因为 Worker 会增加消息协议、复制和生命周期成本。

工程要点：

- 用 `ArrayBuffer`/TypedArray 表达紧凑数值列；
- 可转移对象应放入 transfer list，避免不必要复制；
- 转移后原线程不再拥有相应 buffer，不能继续读取；
- 消息携带版本号，丢弃迟到的旧渲染结果；
- Worker 报错要能回传，主线程应有降级 UI；
- 组件销毁时终止 Worker，释放 OffscreenCanvas 和 GPU 资源。

Worker 不会让算法自动变快，它主要把工作从交互关键路径移开。若每帧仍传输巨大对象，序列化本身就可能成为瓶颈。

### 性能工程：预算、测量与逐层优化

#### 先定义用户可感知预算

比“支持十万点”更有意义的指标是：

- 首次可见图形耗时；
- 拖拽和缩放帧耗时；
- pointermove 到高亮更新的延迟；
- 数据更新频率与丢帧比例；
- 峰值内存、GPU 资源和卸载后残留；
- 低端目标设备上的退化方式。

60 Hz 屏幕一帧约 16.7ms，但脚本不能独占全部时间，浏览器还要处理样式、布局、绘制与合成。高刷新率设备预算更紧。

#### 按流水线定位瓶颈

- 数据阶段：解析、排序、聚合是否重复？
- 布局阶段：文字测量、刻度和标签避让是否反复发生？
- 几何阶段：是否每次交互都重建全部路径？
- 绘制阶段：节点、draw call、状态切换或填充像素是否过多？
- 交互阶段：pointermove 是否线性扫描？
- 框架阶段：巨型数组是否进入深响应与虚拟 DOM diff？

优化必须针对证据。例如 tooltip 卡顿可能来自同步布局读取，而非折线绘制。

#### 常用优化顺序

1. 删除重复计算，缓存与尺寸/数据版本相关的派生结果；
2. 将多次输入合并到动画帧；
3. 限制可见数据并正确降采样；
4. 建立命中索引；
5. 减少 DOM 节点或 Canvas 状态切换；
6. 把已证实的 CPU 重任务移到 Worker；
7. 数据规模确实需要时采用 WebGL。

Canvas 可把重复复杂图形预渲染到离屏画布，再复用位图；坐标尽量对齐设备像素可避免不必要抗锯齿。WebGL 应批量 draw、避免阻塞式读取，并按像素预算管理显存。

## 用测试与平台契约保护长期演进

当图表数学、交互和资源生命周期已经分层，测试也应沿着同样的边界展开。像素截图只能覆盖最终表象，无法证明数据含义、反向映射、键盘路径或资源释放正确。

### 测试：像素快照远远不够

#### 纯逻辑测试

优先覆盖：

- 比例尺正向/反向映射与 invert；
- 空数据、单点、重复时间、乱序和非法数值；
- 坐标边界与裁剪；
- 缩放锚点、最小跨度和 reset；
- 命中半径和边界候选；
- 环形缓冲覆盖顺序；
- 时区、locale 和单位格式化。

这些测试快速、稳定，能准确指出数学错误。

#### DOM 与交互测试

验证可访问名称、表格表头、键盘顺序、选择保持、resize 清理和降级状态。不要只触发组件方法；关键路径要通过真实键盘/指针事件验证。

#### 视觉回归

截图适合发现轴标签错位、裁剪、颜色或字体变化，但要控制字体、时区、locale、视口、devicePixelRatio、动画和测试数据。否则快照噪声会掩盖真实回归。

视觉差异只能说明“像素变了”，无法判断数据语义是否正确，因此不能代替纯逻辑断言和可访问性检查。

#### 性能回归

使用固定数据规模和设备基线，记录初始化、更新和交互延迟。避免只在开发者高配电脑上测平均值；关注较慢分位数和长期运行后的内存。

### 组件 API 与平台治理

一个可复用图表组件的 API 应表达业务能力，而非泄漏底层库的所有配置：

```ts
interface TrendChartProps {
  series: readonly TrendSeries[];
  xDomain?: readonly [number, number];
  unit: 'ms' | 'percent' | 'currency';
  locale: string;
  timeZone: string;
  accessibleSummary: string;
  onSelectionChange?(selection: ChartSelection | null): void;
}
```

若直接暴露巨型 `option: Record<string, unknown>`，调用者会依赖任意底层细节，未来无法统一主题、升级库或替换渲染器。可以提供受控扩展点，但要区分稳定契约和逃生口。

平台层还应统一：

- 设计 token、分类/顺序/发散色板与对比度；
- 数字、时间、单位和空值格式化；
- loading、empty、error、partial 等状态；
- 埋点命名、性能测量和错误上报；
- 依赖版本、许可证、安全更新和浏览器基线；
- 数据导出、隐私脱敏和权限边界。

#### 不要把敏感数据藏在 tooltip 中

图表与导出仍受权限控制。隐藏系列、裁剪视口或不给坐标轴不等于数据不可访问；若完整数据已到浏览器，用户就可能从内存或网络读取。真正的行列级权限和聚合脱敏必须在服务端执行。

### 常见失败模式

#### 组件内混合请求、转换、绘图和事件

结果是无法单测数学逻辑、无法替换渲染器，组件销毁后还可能继续接收数据。应按领域适配、纯核心、渲染适配和生命周期拆分。

#### 只看点数选择技术

一万点静态折线可能只需一个 SVG path；一千个带阴影、动画和独立事件的节点却可能很重。应测量节点数、每帧变化量、填充面积和交互成本。

#### 每个 pointermove 都 setState 并全量重绘

输入事件频率可能高于刷新率。应保存最新输入、按帧处理，并用索引查找候选。只有视图真正变化时才提交框架状态。

#### 高分屏只放大 CSS

CSS 尺寸变大而 Canvas backing store 不变会模糊。反过来无上限地按 DPR 放大大型画布，又会消耗大量内存。应测量 CSS 尺寸、设置合理 DPR 上限，并在尺寸变化时重建。

#### tooltip 是唯一解释

触摸、键盘和辅助技术无法依赖 hover。必须同时提供选择状态、键盘路径、摘要和数据表。

#### 为了性能提前切 WebGL

没有基准就引入 GPU 管线，会把问题变成资源泄漏、上下文恢复和着色器兼容。先减少工作量，再用测量决定渲染器。

#### 组件卸载但资源仍活着

常见遗留包括 `ResizeObserver`、事件、RAF、定时器、Worker、WebSocket、Canvas backing store、WebGL buffer/texture。资源的创建者应拥有对称的 dispose 路径。

### 从现有 Vue 2 图表渐进改造

无需一次重写所有页面，可以按以下顺序演进：

1. 先记录现有输入/输出契约和关键截图；
2. 把单位、时区、缺失值和排序从组件抽成领域适配函数；
3. 把比例尺、几何和交互 reducer 抽成无框架 TypeScript；
4. 给旧组件补摘要、数据表入口和键盘选择；
5. 增加帧耗时、数据量、错误和资源清理观测；
6. 只在基准证明必要时，把数据层换成 Canvas/WebGL；
7. Vue 3/React 迁移时复用纯核心，只重写生命周期适配。

这条路径将“框架升级”和“图表重构”解耦，回滚范围更小。

### 落地检查清单

上线前确认：

- [ ] 单位、时区、缺失值、聚合和排序规则已写入契约；
- [ ] 空、单点、重复、乱序、极值和非法输入有确定行为；
- [ ] 数据域、图框、比例尺和反向映射只有一个事实来源；
- [ ] 渲染技术由测量与交互需求决定；
- [ ] hover、focus、selection 和 viewport 状态分离；
- [ ] Canvas/WebGL 有摘要、键盘操作和结构化数据替代；
- [ ] 实时流有容量、背压、降采样和迟到消息策略；
- [ ] resize、DPR、字体、locale、时区和 reduced motion 已验证；
- [ ] Observer、事件、RAF、Worker、连接和 GPU 资源可释放；
- [ ] 纯逻辑、交互、视觉、可访问性与性能回归各有覆盖；
- [ ] 服务端权限与脱敏不依赖前端隐藏；
- [ ] 性能指标在目标设备和代表性数据规模上达标。

## 回到主线：图表是一条可验证的语义到像素流水线

成熟的数据可视化系统不是某个图表库，而是一条可验证的语义到像素流水线：

- 领域层决定“数据意味着什么”；
- 比例尺和几何层决定“怎样映射”；
- SVG、Canvas 或 WebGL 决定“怎样绘制”；
- 状态机和命中索引决定“怎样交互”；
- 摘要、表格和键盘路径决定“不同用户怎样理解”；
- 有界流、性能预算和生命周期决定“能否长期稳定运行”。

掌握这些边界后，无论使用 ECharts、D3、Three.js，还是自研渲染器，你都能判断抽象是否可靠，而不只是不断追加配置项。
