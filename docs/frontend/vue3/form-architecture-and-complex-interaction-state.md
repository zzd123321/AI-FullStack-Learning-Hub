# Vue 3 表单架构与复杂交互状态

> 适用环境：Vue 3.5+、TypeScript、现代浏览器。本节不绑定特定表单库，先建立可迁移到任何库的状态与边界模型。

## 1. 学习目标

完成本节后，你应该能够：

- 区分 DOM 值、表单草稿、提交 DTO 与服务端实体。
- 设计 value、touched、dirty、pending、errors 等独立状态。
- 选择 input、blur、submit 等不同校验时机。
- 正确处理 `.number`、空字符串、IME 与输入规范化。
- 实现可取消、防竞态的异步字段校验。
- 合并客户端与服务端字段错误而不丢失上下文。
- 使用稳定 ID 管理动态字段数组。
- 设计安全、可迁移、可失效的本地草稿。
- 构建标签、提示、错误摘要和焦点管理完整的无障碍表单。
- 控制大型表单的响应式、渲染和校验成本。
- 判断手写 composable 与表单库的适用边界。

## 2. 表单不是一个 reactive 对象

真实表单至少包含以下并行状态：

```text
value       用户当前输入
initial     用于判断 dirty 的基线
touched     用户是否与字段完成过交互
dirty       当前规范化值是否偏离基线
errors      客户端或服务端错误
pending     异步校验/提交是否进行中
submitCount 用户尝试提交的次数
result      成功结果或表单级失败
```

只写一个 `reactive({ title, slug })` 能绑定输入，但无法回答：

- 何时显示错误？
- 是否允许离开页面？
- 旧异步校验能否覆盖新值？
- 保存失败后哪些输入要保留？
- 重置应回到空白、服务端值，还是最近保存值？

表单复杂度来自状态转换，而不是字段数量。

## 3. 四层数据模型必须分开

### DOM 输入值

浏览器输入常以字符串表达；空数字输入是 `''`，日期和多选也有各自 DOM 语义。

### 表单草稿

为编辑体验保留“不完整但合法存在”的值，例如：

```ts
estimatedHours: ''
```

用户删除旧数字准备输入新数字时，空字符串是合理的中间状态。

### 提交 DTO

只有校验通过后才转换：

```ts
estimatedMinutes: Math.round(Number(estimatedHours) * 60)
```

### 服务端实体

可能包含 ID、版本、权限、审计字段和服务端规范化结果，不应直接成为可随意编辑的表单对象。

不要强迫一套接口同时描述这四层，否则会出现大量错误断言和 `as number`。

## 5. `v-model` 的真实展开

不同原生控件对应不同属性和事件：

- 文本框和 textarea：`value` + `input`。
- checkbox/radio：`checked` + `change`。
- select：`value` + `change`。

Vue 以绑定的 JavaScript 状态为事实来源，模板上的初始 `value`、`checked`、`selected` 不会覆盖它。因此初值必须在模型中声明。

```vue
<input v-model="model.title" />
```

概念上接近：

```vue
<input
  :value="model.title"
  @input="model.title = ($event.target as HTMLInputElement).value"
/>
```

## 6. `.trim`、`.number`、`.lazy` 的边界

### `.trim`

同步时裁剪首尾空白。它适合 Slug 等明确不保留空白的字段，但不一定适合正文、密码或需要精确保留输入的内容。

### `.number`

使用类似 `parseFloat()` 的转换；空输入或无法解析时仍可能返回字符串。即使 TypeScript 推断看起来是数字，也不能假定运行时永远是 `number`。

复杂数字字段更稳妥的做法是保留原始字符串，校验后使用 `Number()` 或 `valueAsNumber` 明确转换。

### `.lazy`

把同步时机从 `input` 改为 `change`。这不是“自动防抖”，也不是 blur 校验策略。文本框的 change 一般在失焦且值变化后触发。

修饰符只是输入同步规则，不能替代表单状态机。

## 7. 中文 IME 组合输入

中文、日文、韩文输入法会经历 composition 阶段。Vue 的普通 `v-model` 不会在组合中的每个临时输入都更新，以免把未完成字符当成最终值。

因此：

- 不要假设每次键盘事件都对应最终文本。
- 搜索建议和字符计数要明确是否响应 composition。
- 自定义输入组件必须正确转发 composition 行为。
- 若确实需要组合过程中的值，应自行处理 `input` 与 composition 事件，并充分测试中文输入。

很多“英文正常、中文丢字”的组件问题来自忽略这一层浏览器语义。

## 8. 字段状态：touched 不等于 dirty

### touched

表示用户是否完成过交互，常在 blur 时设为 true。它主要控制错误何时出现，避免页面打开就满屏红色。

### dirty

表示当前值是否偏离初始基线。用户修改后再改回来，dirty 应恢复为 false。

### visited / focused

大型交互有时还会区分获得过焦点与完成过交互，但不要无需求增加状态。

一个字段可以：

- touched 但不 dirty：获得焦点后直接离开。
- dirty 但未 touched：程序或自动填充改变了值。
- 曾经 dirty，后来恢复初值。

不要用一个布尔值承担所有含义。

## 9. Dirty 比较要先定义语义

直接深比较对象会遇到：

- 首尾空白是否算修改？
- 数组顺序是否重要？
- 空字符串与 `null` 是否等价？
- 服务端自动生成字段是否参与？

示例先规范化，再序列化比较：

```ts
const dirty = computed(
  () => serializeDraft(model) !== baseline.value
)
```

这意味着标题末尾多一个空格不算有效修改。若富文本空白、标签顺序或大小写有业务意义，应调整规范化函数，而不是盲信 JSON 字符串。

大型模型可维护字段级 dirty 或结构化 diff，避免每次按键序列化整棵对象。

## 10. 校验规则应是纯函数

<<< ../../../examples/frontend/vue3-forms/validators.ts

纯同步规则有几个优势：

- 不依赖组件实例。
- 可直接单元测试。
- 可在 blur、submit、服务端共享概念。
- 不会隐式触发 Toast、网络请求或修改别的字段。

校验函数返回结构化错误，而不是立即展示错误。何时展示是表单交互策略，不是规则本身。

## 11. 校验时机不是越早越好

### 输入时

适合字符计数、格式提示和用户已经触碰过的错误恢复。若第一个字符刚输入就显示“长度不足”，体验通常很差。

### 失焦时

适合大多数字段的首次完整校验。用户完成一个输入单元后获得反馈。

### 提交时

必须执行全量校验，并把所有字段视为已触碰。键盘、自动填充和程序赋值可能绕过预期 blur 流程。

### 依赖字段变化时

确认密码、日期区间等跨字段规则，需要观察相关字段。错误归属应明确：结束日期错误放在结束日期，还是分组级错误？

推荐渐进策略：首次 blur 或 submit 后显示；一旦某字段有错，输入时及时重新校验以反馈修复。

## 12. 客户端校验不是安全边界

客户端校验用于快速反馈和减少无效请求。它可以被禁用、篡改或绕过，服务端必须重新校验：

- 类型、长度和格式。
- 唯一性。
- 权限与资源归属。
- 跨字段业务规则。
- 并发版本与状态转换。

HTML `required`、Vue 校验函数、第三方 schema 都不能替代服务器校验。

## 13. 原生 Constraint Validation 怎么用

浏览器支持 `required`、`minlength`、`pattern`、`min`、`max` 等约束，以及 `checkValidity()`、`reportValidity()`。

两种有效策略：

1. 使用原生校验 UI，保持简单、低成本。
2. 添加 `novalidate`，由应用统一显示错误摘要、字段错误和焦点。

最危险的是半接管：浏览器阻止 submit，但应用以为自己的 submit handler 会运行；或者同一错误弹原生气泡又显示自定义文本。

本课示例使用 `novalidate`，同时保留语义属性，让约束可被测试和渐进增强，但最终反馈由应用负责。

## 14. 异步校验是一类请求状态机

Slug 唯一性校验存在典型竞态：

```text
输入 vue      → 请求 A（慢）
输入 vue-form → 请求 B（快）
B 返回可用
A 晚到并返回不可用，错误覆盖新值
```

必须满足：

- 同步格式不合法时不发请求。
- 新值出现时取消或废弃旧请求。
- 取消不能显示成网络错误。
- 旧请求的 finally 不能结束新请求的 pending。
- submit 时仍需重新确认；字段校验结果可能已经过期。

示例使用 `watch` + `onWatcherCleanup()` + `AbortController`。清理注册必须在第一个 `await` 之前同步执行。

## 15. 防抖与取消解决不同问题

- 防抖减少请求数量：停止输入一段时间后再发。
- 取消停止旧工作：新值到来时中止旧请求。
- 请求序号阻止旧结果提交：底层不支持取消时仍有效。
- 缓存避免相同值重复请求：需要过期和失效策略。

只做防抖仍可能竞态，只做取消可能产生过多短命请求。生产实现可组合这些策略。

异步唯一性检查只能提供提示，不能预留名称。检查成功到正式提交之间，其他用户仍可能抢先创建，服务端提交必须再次检查并返回冲突。

## 16. 服务层与服务端字段错误

<<< ../../../examples/frontend/vue3-forms/lesson-service.ts

`FormSubmissionError` 同时携带：

- 表单级消息：例如“部分字段需要修改”。
- 字段错误映射：例如 `slug → 该 Slug 已被占用`。

服务层负责把 HTTP 422、409 或后端错误结构转换成稳定前端错误类型。组件不应到处判断 `error.response.data.errors`。

网络失败、服务不可用与字段错误不同：前者通常显示表单级提示并允许重试；后者必须关联到具体字段。

## 17. 客户端错误与服务端错误如何共存

常见策略有两种：

### 单一当前错误

每个字段只显示当前最高优先级错误。用户修改字段后清除旧服务端错误，再运行客户端规则。实现简单。

### 分来源保存

```ts
errors[field] = {
  client: '格式错误',
  server: '名称已存在'
}
```

适合需要审计或同时展示多个消息的复杂系统，但状态更多。

无论哪种策略，都要定义服务端错误何时失效。用户修改了 Slug 后，“Slug 已占用”针对旧值，不应永久保留。

## 19. `valid` 不等于“可以提交”

提交按钮状态常被误写为：

```vue
<button :disabled="!valid">提交</button>
```

问题包括：

- 用户还没交互时，无法通过提交发现错误。
- 异步校验 pending 时 valid 可能暂时为真。
- 服务端仍可能拒绝。
- disabled 控件不会获得焦点，也不易解释为什么不可用。

是否禁用取决于产品策略。通常必须禁用重复提交；字段不完整时可以允许点击，然后展示完整错误摘要。若禁用，应提供可感知原因。

## 20. 重复提交与幂等

客户端用 `submitting` 阻止双击只是第一层。以下情况仍可能重复：

- 用户刷新后重试。
- 网络超时但服务端已成功。
- 多标签页同时提交。
- 代理或客户端自动重试。

创建付款、订单等关键操作应使用服务端支持的幂等键。前端生成操作 ID，重试复用同一 ID，服务端返回同一结果。

普通表单也要明确：提交中是否允许继续编辑、是否可取消、超时后如何确认最终状态。

## 21. 动态字段数组

动态列表必须使用稳定业务 ID：

```vue
<div v-for="outcome in outcomes" :key="outcome.id">
```

错误也按 ID 保存：

```ts
outcomeById: Record<string, string>
```

不要按索引保存：删除第二项后，原第三项会成为第二项，索引错误和输入内部状态可能落到错误行。

删除字段时同步清理：

- 该字段错误。
- touched/dirty 元数据。
- 进行中的异步请求。
- 依赖它的跨字段错误。

## 22. 动态字段的键与服务端 ID

新建行还没有数据库 ID，可使用客户端临时 ID。保存后可以：

- 保留 client ID 仅用于 UI，再单独保存 server ID。
- 用服务端结果建立 client → server 映射。
- 替换 ID，但要小心 Vue key 导致组件重建和焦点丢失。

时间戳加数组长度只适合教学；生产项目应使用可靠 UUID/单调 ID，确保同一会话不碰撞。

## 23. 基础字段组件的职责

<<< ../../../examples/frontend/vue3-forms/BaseField.vue

字段壳组件负责稳定的表现协议：

- label 与 `for`。
- hint/error ID。
- `aria-describedby` 组合。
- `aria-invalid` 状态。
- 一致布局。

具体 input 通过 Slot 提供，避免基础组件猜测所有控件属性。Slot Props 本身是公共 API，应保持小而明确。

不要把校验规则、网络请求和整个表单上下文塞进通用 Input 组件；它会变成无法复用的业务组件。

## 24. Label、Hint 与 Placeholder

每个输入必须有可感知名称，优先使用显式 `<label for>`。Placeholder：

- 输入后会消失。
- 对比度可能不足。
- 不能稳定表达格式与要求。
- 不能替代 label。

帮助文本通过 `aria-describedby` 关联。错误出现时把错误 ID 加入同一个属性，让辅助技术同时读取提示与错误。

必填标记不能只靠颜色或星号。可见文案、`required` 属性或清晰的“必填”说明应共同表达。

## 25. 错误提示必须回答三个问题

好的错误应说明：

1. 哪个字段有问题。
2. 问题是什么。
3. 如何修复。

“输入非法”远不如“只能使用小写字母、数字和单个连字符”。

错误不能只靠红色边框。需要文字、程序化关联和必要时的错误摘要。

## 26. 错误摘要与焦点

提交失败后：

1. 渲染页面顶部错误摘要。
2. 等待 `nextTick()` 确保 DOM 更新。
3. 把焦点移动到摘要。
4. 摘要提供跳到具体字段的链接。
5. 字段旁仍显示就地错误。

摘要使用 `tabindex="-1"` 允许程序聚焦，但不进入常规 Tab 顺序。`role="alert"` 适合提交后新增的紧急错误，不应在每次按键时反复播报整段错误。

若产品更适合直接聚焦第一个错误字段，也要同时提供全局错误数量和上下文。避免页面在用户输入时自动抢焦点。

## 27. Status 与 Alert 的区别

- `role="status"`：非紧急状态，如“正在检查”“保存成功”。通常礼貌播报。
- `role="alert"`：需要立即注意的错误。通常强制播报。

不要把所有动态文字都标成 alert。异步输入校验每次字符变化都高优先级播报，会让读屏体验不可用。

Loading 文案还应与视觉按钮状态一致；只转圈但无可访问名称不够。

## 28. Fieldset 与分组语义

一组 radio、checkbox 或重复的学习成果应使用 `fieldset` + `legend` 表达组名。视觉上的标题和 div 不能自动提供同等语义。

组级错误可以关联到 fieldset 的说明，也可以在 legend 后展示。动态删除按钮的名称应包含行上下文，例如“删除成果 2”，而不只是多个同名“删除”。

## 30. 草稿持久化模型

<<< ../../../examples/frontend/vue3-forms/draft-storage.ts

从 `localStorage` 读取的内容属于不可信运行时数据，即使它是自己之前写入的：

- 用户或扩展可修改。
- 旧版本结构仍可能存在。
- JSON 可能损坏。
- 多标签页会互相覆盖。

因此需要解析、结构检查和版本号，不能直接：

```ts
Object.assign(model, JSON.parse(localStorage.getItem(key)!))
```

## 31. 草稿的产品决策

实现自动保存前要回答：

- 保存到本地还是服务端？
- 保存频率和最大大小？
- 多标签页冲突如何处理？
- 登录用户切换后是否隔离？
- 草稿何时过期？
- 服务端实体更新后如何合并？
- 保存成功、取消、退出登录时何时删除？

localStorage 是同步 API，大对象频繁序列化会阻塞主线程。大型草稿可使用 IndexedDB 或服务端草稿，并采用节流、增量更新与冲突版本。

## 32. 草稿安全与隐私

不要默认持久化：

- 密码、Token、银行卡安全码。
- 医疗、身份、隐私文本。
- 受合规限制的数据。
- 可被 XSS 读取的长期敏感内容。

任何同源脚本都可能读取 localStorage。加密若密钥也在同一前端，不能抵御 XSS。先最小化保存字段和保留时间，再讨论加密。

## 33. 服务端初值变化与草稿冲突

编辑页加载实体时，可能同时存在：

- 服务端最新版本。
- 用户本地草稿。
- 页面当前修改。

不能简单 watch Prop 后 `Object.assign(model, entity)`，这会覆盖未保存输入。常见策略：

- 未 dirty 时自动采用新服务端值。
- dirty 时提示“远端已更新”，让用户比较。
- 使用版本号/ETag 提交，服务端检测乐观锁冲突。
- 对结构化数据执行三方合并：基线、当前草稿、服务端新值。

表单基线是并发控制的一部分，不只是 dirty 计算工具。

## 34. Reset 的语义

“重置”可能表示：

- 新建页回到空表单。
- 编辑页回到最近服务端值。
- 回到最近成功保存值。
- 放弃当前步骤但保留其他步骤。

原生 `form.reset()` 只恢复 DOM 默认值，不会自动正确同步复杂 Vue 状态。受控表单应通过模型重置，并同时清理 touched、errors、pending、提交结果和草稿。

危险操作应明确文案，如“放弃所有修改”，必要时二次确认。

## 35. 多步骤表单

Wizard 不应把每一步拆成互不知情的独立事实来源。通常有一个顶层草稿和步骤级视图：

- 下一步：校验当前步骤。
- 上一步：保留当前草稿。
- 最终提交：再次执行全量校验。
- URL：可记录当前步骤，但不能泄露敏感表单值。

用户直接进入后续步骤时，要验证前置数据是否存在。若步骤可独立保存，服务端状态机必须定义允许的转换。

## 36. 跨字段校验

例子包括：

- 开始时间早于结束时间。
- 至少选择一种联系方式。
- 发布状态要求摘要和成果完整。
- 折扣价不能高于原价。

规则应接收完整 draft，返回明确错误位置。不要建立多个 watcher 互相修改字段和错误，容易形成环与瞬时不一致。

优先在纯 `validateDraft()` 中一次计算；只有需要即时联动 UI 时才观察精确依赖。

## 37. 条件字段

字段被隐藏后要决定：

- 值是否保留？
- 是否继续校验？
- 是否提交？
- touched/error 是否清除？

“看不见但仍阻止提交”是常见缺陷。规则应根据当前业务分支判断字段是否参与，不要只在模板用 `v-if` 隐藏。

若重新显示时要恢复用户输入，可以保留值但从 DTO 和校验中排除；若值不再有意义，应显式清除。

## 38. 文件上传

`<input type="file">` 不能像普通文本那样安全地通过 v-model 设置值。文件对象：

- 不应进入 JSON 草稿。
- 可能很大且生命周期短。
- 需要类型、大小和服务端内容验证。
- 预览 URL 要 `URL.revokeObjectURL()` 清理。

大型上传通常独立成上传状态机：选择、校验、上传、进度、成功资源 ID、失败重试、取消。最终表单 DTO 只提交已上传资源 ID。

`accept` 只影响文件选择提示，不是安全校验。

## 39. 富文本与第三方控件

富文本编辑器、日期选择器等组件必须定义：

- v-model 值格式。
- 空值语义。
- touched/blur 事件。
- disabled/readonly 差异。
- 错误与帮助文本关联。
- 键盘与读屏行为。
- SSR 和销毁清理。

外观看起来像 input 不代表具备原生控件语义。优先选择建立在原生元素上的组件；复杂 ARIA widget 需要完整键盘交互。

## 40. Disabled 与 Readonly

- disabled 控件不可交互、通常不可聚焦，原生表单提交时也不会包含。
- readonly 控件通常可聚焦和复制，仍可能参与提交；并非所有控件都支持。

提交中禁用整个 fieldset 可以阻止修改，但用户也无法复制内容或查看焦点位置。也可以只禁用提交按钮，同时让输入保持可读，服务端按提交快照处理。

权限不足时不要只用 disabled 隐藏安全问题，后端仍需拒绝写操作。

## 41. 大型表单性能

常见成本不是 `v-model` 本身，而是：

- 每次按键深度 watch 整个大对象。
- 每次按键运行全部复杂 schema。
- JSON 序列化巨大草稿。
- 数百字段同时渲染。
- 错误对象全量替换导致广泛更新。
- unstable key 导致字段组件重建。

优化顺序：

1. 测量实际慢点。
2. 校验精确字段，提交时再全量校验。
3. 防抖持久化和昂贵异步操作。
4. 拆分稳定子组件，保持 Props 稳定。
5. 动态列表使用稳定 key。
6. 超大列表考虑分步、折叠或虚拟化，但不能破坏键盘导航。

不要为了“性能”把所有字段改成非响应式 DOM 查询；这会失去可预测状态边界。

## 42. Watcher 设计

Watcher 用于副作用，不用于复制所有派生状态：

- 异步校验。
- 草稿持久化。
- 跨系统同步。

纯错误计算、dirty 和字符数优先 computed 或纯函数。

观察精确字段而非整个 route/store/form；深度 watcher 会遍历嵌套结构。异步 watcher 要同步注册 cleanup，并检查旧请求的 finally。

组件同步创建的 watcher 会随组件卸载停止；延迟异步创建的 watcher 可能需要手动停止。

## 43. 何时使用表单库

手写 composable 适合：

- 字段不多但业务状态特殊。
- 团队需要完全掌控交互。
- 不希望引入额外抽象。

表单库适合：

- 大量重复字段元数据。
- 动态数组、嵌套对象很多。
- 统一 schema、错误映射与提交生命周期。
- 团队已有成熟规范。

选择库时评估：

- Vue 版本和维护状态。
- TypeScript 推断质量。
- 原生与自定义控件接入方式。
- 异步校验取消与竞态。
- 动态数组稳定键。
- SSR、无障碍、包体和测试支持。

库能减少模板样板，但不能替你定义服务端错误、草稿冲突、权限和产品校验时机。

## 44. 测试分层

### 纯函数单测

- 每条字段边界。
- 跨字段规则。
- 草稿 → DTO 转换。
- 草稿版本解析和损坏数据。

### Composable 单测

- touched/dirty 状态转换。
- 异步校验乱序与取消。
- 提交成功后基线更新。
- 服务端字段错误映射。
- 重置清理。

### 组件测试

- label 与控件关联。
- blur/submit 后错误出现时机。
- 错误摘要焦点和链接。
- 动态添加删除保持正确字段。
- loading、disabled、status/alert。

### 端到端测试

- 键盘完成全流程。
- 浏览器自动填充。
- 刷新草稿恢复。
- 服务端冲突与重试。
- 导航离开保护。
- 中文 IME 和真实浏览器约束。

不要只断言内部 reactive 对象；优先验证用户可观察行为。

## 45. 常见反模式

### DTO 类型直接用于输入

数字字段无法表达空输入，只能到处断言。建立 Draft 类型。

### 页面加载立即显示全部错误

用户尚未交互就被惩罚。使用 touched 或提交尝试控制展示。

### 每次输入全量异步校验

请求风暴且产生竞态。先同步过滤，再防抖/取消。

### 服务端失败后重置表单

用户输入全部丢失。保留草稿，映射错误，并让用户修复或重试。

### 动态数组用 index 作 key

删除/重排后错误与组件状态错位。使用稳定 ID。

### Placeholder 代替 Label

输入后说明消失，辅助技术和认知体验都变差。

### 用 disabled 表示所有无效状态

用户无法触发并理解错误。至少保留可感知原因和提交反馈路径。

### watch 双向同步 Props 与本地草稿

远端更新覆盖用户输入，本地更新又回写父级。明确受控值、初始值与冲突策略。

## 46. Vue 2 迁移提示

- Vue 3 组件 `v-model` 默认是 `modelValue` / `update:modelValue`，并支持多个命名 model。
- 不再依赖 Vue 2 的 `.sync` 建立表单双向协议。
- Composition API 可把字段状态机抽为 composable，但不要把 mixin 的隐式依赖原样搬过去。
- 动态属性不再需要 `Vue.set`，但稳定键和错误清理仍然必要。
- Watcher cleanup 与 effect scope 更适合管理异步校验。
- 重新审查旧 UI 库的 label、error、ARIA 与 Vue 3 兼容性。

## 47. 工程检查清单

- 是否区分 Draft、DTO 与服务端 Entity？
- 数字/日期字段能否表达空和输入中间态？
- 初始基线和 reset 语义是否明确？
- touched 与 dirty 是否被混用？
- 校验展示时机是否符合用户交互？
- submit 是否总会执行全量校验？
- 异步校验是否取消旧请求并防止旧 finally？
- 服务端字段错误是否映射到具体控件？
- 网络错误是否保留全部输入？
- 动态字段是否使用稳定 ID？
- label、hint、error 是否程序化关联？
- 提交失败是否提供错误摘要和合理焦点？
- 草稿是否版本化、验证、过期并避免敏感数据？
- 多标签页和服务端版本冲突是否有策略？
- 大型表单是否避免深度全量 watch？
- 关键流程是否覆盖键盘、IME 与真实浏览器测试？

## 48. 概念辨析与因果回顾

### 为什么数字输入的模型常保留字符串？

输入过程需要表达空值和未完成状态；`.number` 也不能保证始终返回数字。校验通过后在 DTO 边界转换更安全。

### touched 和 dirty 有什么区别？

touched 表示用户完成过交互；dirty 表示当前值偏离基线。两者可以独立变化。

### 如何避免异步字段校验竞态？

新值出现时取消旧请求，或用序号阻止旧结果提交；清理必须在 await 前注册，并避免旧 finally 修改新 pending。

### 为什么检查 Slug 可用后提交仍可能冲突？

检查与提交不是原子操作，期间其他用户可能占用。唯一性必须由服务端在写入时最终保证。

### 为什么动态字段不能用索引作 key？

删除或重排会改变索引，Vue 复用错误的组件实例，字段错误、焦点和内部状态可能错位。

### 客户端校验和 HTML 约束能保证安全吗？

不能。客户端可被绕过，服务端必须验证输入、权限和业务约束。

### 表单库解决不了什么？

它不能替代业务校验时机、服务端错误协议、并发冲突、草稿安全、权限和产品交互决策。

## 49. 本节总结

- 表单是 value、baseline、touched、dirty、errors、pending 与 result 组成的状态机。
- Draft 应允许输入中间态，DTO 只在校验成功后生成。
- `v-model` 修饰符有明确运行时边界，IME 需要真实语言测试。
- 校验规则保持纯函数，展示时机由交互状态决定。
- 异步校验必须处理取消、竞态和提交时的最终确认。
- 服务端错误分为字段级与表单级，失败时应保留用户输入。
- 动态字段使用稳定 ID，删除时清理关联元数据。
- 完整无障碍反馈包括 label、hint、字段错误、摘要、焦点与 live region。
- 草稿是版本化、不可信且可能敏感的持久化数据。
- 大型表单先测量，再优化 watcher、校验、持久化和渲染范围。
