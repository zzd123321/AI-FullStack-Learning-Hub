# 数据类型、NULL、默认值与约束

::: tip 第一次学习只抓住四件事
- **必须理解**：类型描述值的业务含义；`NULL` 表示缺失，不等于空字符串或零；默认值不是前端兜底；约束负责拒绝非法状态。
- **必须会选**：标识、文本、精确金额、布尔值和时间的常用类型，并知道 JavaScript `number` 不能安全承载所有 `BIGINT`。
- **必须完成**：为一个创建接口写出包含主键、非空、唯一和检查约束的表结构。
- **可以后看**：类型的完整范围、数据库方言细节和复杂时间类型。不要背类型表。
:::

接口文档里的一个字段通常只有名称、示例和是否必填；数据库列还必须决定类型、长度、默认值、可空性和约束。这个转换不是机械地把 TypeScript 类型翻译成 SQL 类型，而是在定义系统最终允许保存什么数据。

## 本课目标与阅读路线

完成本课后，你应能从字段的业务语义、取值范围、精度、缺失含义和跨库差异五个角度选择列定义，并能说明为什么应用校验不能替代数据库约束。它把上一课建立的连接与会话落到可靠表结构上，下一课会在这些约束之上讨论安全且稳定的 `SELECT` 查询。

假设后端接收创建账号请求：

```json
{
  "email": "linxia@example.com",
  "displayName": "林夏",
  "status": "pending",
  "creditLimit": "1000.00",
  "marketingOptIn": false
}
```

我们会把它设计成一张 `learning_accounts` 临时表，并分别给出 MySQL 8.4 与 PostgreSQL 18 的版本。重点不是记住两段 DDL，而是理解每个决定保护了什么。

## 表结构是最后一道数据边界

一次写入通常会经过多层：

```text
页面校验 → 接口参数校验 → 业务规则 → 数据库类型与约束
```

这些层不是重复劳动：

- 页面校验提供及时反馈，但请求可以绕过页面直接发送。
- 后端校验能表达复杂业务规则，但可能存在多个服务、脚本或导入程序同时写库。
- 数据库类型和约束位于最终存储边界，负责拒绝任何入口提交的不合法状态。

数据库也不适合承载全部业务规则。比如“试用账号只能在注册后 30 天内升级”涉及时间、权限和流程，更适合由业务服务处理；而“主键不能重复”“信用额度不能为负数”非常适合由数据库约束兜底。

## 先从业务含义选择类型

不要先问“这个值现在长什么样”，而要问：

1. 它表示数量、标识、金额、文本还是时间点？
2. 允许的范围和精度是什么？
3. 是否参与计算、排序、唯一性或关联？
4. 缺失值与空字符串是否具有不同含义？
5. 未来的变化是否会超出当前示例？

例如手机号看起来全是数字，却不应该用整数保存：它不参与算术，可能有前导零、国家码和格式符。订单金额参与精确计算，才应该使用精确数值类型。

## 整数：标识与计数不是一回事

MySQL 和 PostgreSQL 都有 `SMALLINT`、`INTEGER`/`INT`、`BIGINT` 等整数类型。位数越大，可表示范围和存储成本通常也越大。

### 主键

本课使用 64 位整数主键：

```sql
-- MySQL
id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT PRIMARY KEY
```

```sql
-- PostgreSQL
id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY
```

两种写法都能在插入时生成新值，但语法和机制不同：

- MySQL 的 `AUTO_INCREMENT` 是整数列属性。
- PostgreSQL 的 identity 列由隐式序列生成值；`BY DEFAULT` 允许显式提供值，`ALWAYS` 默认拒绝普通显式值。
- 自动生成不等于唯一。PostgreSQL identity 定义仍应搭配 `PRIMARY KEY` 或 `UNIQUE`；这里直接声明主键。
- 生成的编号只应该被视为身份，不应该假设连续无缺口。事务回滚、并发写入和数据迁移都可能留下间隔。

### 与前端联调的隐藏问题

JavaScript 的 `number` 不能精确表示所有 64 位整数。如果数据库 `BIGINT` 主键可能超过 `Number.MAX_SAFE_INTEGER`，API 应考虑把 ID 序列化为字符串：

```json
{ "id": "9223372036854775807" }
```

ID 是不透明标识，用字符串传输不会妨碍后端用整数建立主键和关联。前端也不应该对 ID 做加减运算。

### 非负计数

MySQL 支持 `UNSIGNED`，PostgreSQL 没有等价的通用无符号整数类型。为了明确业务规则，可以使用约束：

```sql
login_count INTEGER NOT NULL DEFAULT 0,
CONSTRAINT chk_learning_accounts_login_count CHECK (login_count >= 0)
```

即使 MySQL 使用 `UNSIGNED`，保留有业务名称的 `CHECK` 也能让结构意图更清楚，并为跨数据库迁移提供线索。

## 精确数值：金额不要使用浮点数

`FLOAT`、`REAL`、`DOUBLE PRECISION` 是近似数值。二进制浮点无法精确表示许多十进制小数，因此不适合作为需要精确相等和加总的金额存储。

MySQL 的 `DECIMAL` 与 PostgreSQL 的 `NUMERIC` 都可定义精度和小数位数：

```sql
credit_limit DECIMAL(12, 2) NOT NULL DEFAULT 0.00
```

```sql
credit_limit NUMERIC(12, 2) NOT NULL DEFAULT 0.00
```

`(12, 2)` 表示总共最多 12 位十进制数字，其中 2 位在小数点后，整数部分最多 10 位。它只是本课示例，不是所有金额字段的固定答案：

- 不同币种的小数位数可能不同。
- 汇率和计费中间值往往需要高于展示金额的精度。
- 金额通常还需要单独的币种列，不能只存一个裸数字。
- PostgreSQL 的 `money` 类型受本地化设置影响，跨环境系统通常更容易用 `NUMERIC` 明确控制精度和格式。

接口示例把 `creditLimit` 写成字符串，是为了避免 JSON 数字进入 JavaScript 浮点表示后提前丢失精度。后端应使用语言中的十进制定点类型，例如 Java `BigDecimal`，并由驱动绑定到数据库精确数值类型。

## 文本：长度、字符集和排序规则

本课使用：

```sql
email VARCHAR(255) NOT NULL,
display_name VARCHAR(80) NOT NULL,
status VARCHAR(20) NOT NULL
```

`VARCHAR(n)` 的 `n` 表示允许的字符长度上限，而不是“页面当前看起来最多有多长”。长度应该来自领域规则、外部协议或明确的产品限制。

### `CHAR`、`VARCHAR` 与 `TEXT`

- `CHAR(n)` 是定长字符类型，通常只适合真正固定长度的值；不要仅因为状态码“比较短”就使用它。
- `VARCHAR(n)` 适合有明确最大长度的普通文本。
- `TEXT` 适合不希望用列类型强制短上限的文本，但仍应在接口层设置合理请求大小。

MySQL 与 PostgreSQL 对这些类型的存储、尾部空格、索引限制等细节并不完全相同。跨库设计应依赖业务需要，而不是假设同名类型内部实现一致。

### 字符集不等于排序规则

- **字符集（character set）** 决定能表示哪些字符及其编码。MySQL 应优先使用完整 Unicode 的 `utf8mb4`，不要把历史上的 `utf8mb3` 当作完整 UTF-8。
- **排序规则（collation）** 决定字符串如何比较和排序，包括大小写、重音和语言规则。

`UNIQUE (email)` 是否把大小写不同的邮箱视为重复，可能受到数据库产品和列排序规则影响。不要假设 MySQL 默认排序规则与 PostgreSQL 默认文本比较得到相同结果。

可靠做法通常是先定义产品规则：写入时如何规范化邮箱、是否保留原始展示形式、唯一性基于哪个规范值。表达式索引、函数索引和排序规则会在索引课程继续讨论。

### 状态字段为什么暂用 `VARCHAR + CHECK`

MySQL 与 PostgreSQL 都有枚举相关能力，但定义和迁移方式不同。本课先使用可移植性更高的组合：

```sql
status VARCHAR(20) NOT NULL DEFAULT 'pending',
CONSTRAINT chk_learning_accounts_status
  CHECK (status IN ('pending', 'active', 'disabled'))
```

这能拒绝拼写错误，又允许以后通过迁移调整集合。状态转换是否合法，例如能否从 `disabled` 直接回到 `pending`，仍是业务流程规则，不能只靠值域检查表达。

## 布尔值：API 一样，数据库实现不同

接口中的 `true`/`false` 可以直接映射为布尔概念，但产品实现不同：

```sql
-- MySQL：BOOLEAN 是 TINYINT 的同义写法
marketing_opt_in BOOLEAN NOT NULL DEFAULT FALSE
```

```sql
-- PostgreSQL：原生 boolean 类型
marketing_opt_in BOOLEAN NOT NULL DEFAULT FALSE
```

为了让 MySQL 端也只接受布尔范围，本课额外加入：

```sql
CONSTRAINT chk_learning_accounts_marketing_opt_in
  CHECK (marketing_opt_in IN (FALSE, TRUE))
```

后端应该通过驱动绑定语言布尔值，不要依赖字符串 `'true'`、整数 `1` 和数据库隐式转换在所有产品上碰巧得到相同结果。

## 时间：先区分日期、当地时间和时间点

时间字段最常见的问题不是语法，而是把不同业务含义塞进同一种类型。

| 业务含义 | 典型类型 | 例子 |
| --- | --- | --- |
| 只有日历日期 | `DATE` | 生日、账单日期 |
| 不含时区的当地日期时间 | `DATETIME` 或 `timestamp without time zone` | 门店每天当地 09:00 开门 |
| 全球唯一时间点 | MySQL `TIMESTAMP`；PostgreSQL `timestamptz` | 创建时间、登录时间 |

本课的 `created_at` 和 `last_login_at` 表示事件发生的时间点：

```sql
-- MySQL
created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
last_login_at TIMESTAMP NULL
```

```sql
-- PostgreSQL
created_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
last_login_at TIMESTAMPTZ NULL
```

必须理解它们不是完全等价的同名实现：

- PostgreSQL 中单写 `TIMESTAMP` 表示 **without time zone**；`TIMESTAMPTZ` 是 `timestamp with time zone` 的缩写。带时区时间点内部按 UTC 处理，输出时转换到当前会话时区。
- MySQL `TIMESTAMP` 会依据会话时区在 UTC 与会话值之间转换；`DATETIME` 不做同样的时区转换，而且两者范围等细节也不同。
- 类型通常不会替你永久保存用户输入的 IANA 时区名称。若业务需要“每周一上海时间 09:00”这类未来计划，还要保存 `Asia/Shanghai` 一类时区规则。

后端系统应明确统一策略：数据库会话时区、应用时区、API 输出格式和前端展示时区分别是什么。API 通常用包含 `Z` 或偏移量的 ISO 8601/RFC 3339 字符串传输时间点，例如 `2026-07-13T09:30:00Z`。

### `CURRENT_TIMESTAMP` 的边界

把 `created_at` 默认设为 `CURRENT_TIMESTAMP` 能保证遗漏该列的写入仍有时间值。`updated_at` 则要更谨慎：

- MySQL 支持列级自动更新语法，但它是产品特性。
- PostgreSQL 通常由 SQL、触发器或应用显式维护更新时间。
- “任何行变化都更新”与“业务内容变化才更新”不是同一规则。

本课不添加 `updated_at`，避免用一个看似方便的默认值掩盖更新语义。

## `NULL`：未知或缺失，不是空字符串

`NULL` 表示值未知或不存在。它不是 `0`、`FALSE`、空字符串，也不是字符串 `'null'`。

本课允许 `last_login_at` 为 `NULL`，因为新账号确实可能从未登录：

```sql
SELECT id, email
FROM learning_accounts
WHERE last_login_at IS NULL;
```

不能写：

```sql
-- 错误的 NULL 判断
WHERE last_login_at = NULL
```

SQL 使用三值逻辑：比较结果除了 `TRUE`、`FALSE`，还可能是 `UNKNOWN`。`NULL = NULL` 的结果也不是 `TRUE`，所以必须用 `IS NULL` 或 `IS NOT NULL`。

是否允许 `NULL` 应逐列决定：

- `email NOT NULL`：账号没有邮箱就不是合法记录。
- `display_name NOT NULL`：本系统要求注册时提供显示名。
- `last_login_at NULL`：从未登录是合理状态。

如果空字符串与缺失在业务上没有区别，后端应先规范化输入，不要让 `NULL`、`''` 和只含空格的字符串长期表达同一种状态。

## `DEFAULT`：列被省略时采用的值

默认值减少重复输入，但不等于“数据库会修复所有空值”。例如：

```sql
status VARCHAR(20) NOT NULL DEFAULT 'pending'
```

省略 `status` 时会使用默认值：

```sql
INSERT INTO learning_accounts (email, display_name)
VALUES ('linxia@example.com', '林夏');
```

显式传入 `NULL` 不会因为存在 `DEFAULT` 就自动改用默认值；因为列同时声明了 `NOT NULL`，在 PostgreSQL 和本课程要求的 MySQL 严格 SQL 模式下，这次写入应失败：

```sql
INSERT INTO learning_accounts (email, display_name, status)
VALUES ('linxia@example.com', '林夏', NULL);
```

MySQL 的非严格模式可能对部分非法值执行隐式转换并给出警告，这会让开发、测试和生产表现不一致。团队应把 `sql_mode` 纳入环境基线并优先使用严格模式，不能把自动纠错当作数据校验策略。

数据库默认值也不一定等于 API 默认值。接口层是否允许客户端省略字段，是接口契约；数据库默认值负责所有写入入口最终省略该列时的存储行为。两者应有意保持一致，而不是偶然相同。

## 约束分别保护什么

### `NOT NULL`

保证列必须有值。它比在每个写库调用前手工判断更接近最终数据边界。

### `UNIQUE`

保证一列或一组列的值不重复：

```sql
CONSTRAINT uq_learning_accounts_email UNIQUE (email)
```

唯一约束不是“先查询再插入”。并发请求可能同时发现邮箱不存在，只有数据库中的唯一约束能在最终写入竞争中可靠裁决。后端仍需捕获约束冲突并转换为合适的接口错误，例如 HTTP `409 Conflict`。

### `CHECK`

保证每一行满足布尔表达式：

```sql
CONSTRAINT chk_learning_accounts_credit_limit CHECK (credit_limit >= 0)
```

命名约束能让错误日志和迁移脚本更容易定位。`CHECK` 适合行内、确定性的规则；不要期待它跨表查询或调用任意业务逻辑。MySQL 8.4 与 PostgreSQL 18 都会执行本课使用的基础 `CHECK`，但表达式允许范围和细节仍有产品差异。

还要注意：`CHECK (value > 0)` 对 `NULL` 的判断可能得到 `UNKNOWN`，并不等价于禁止空值。若列必须存在，应同时写 `NOT NULL`。

### `PRIMARY KEY`

主键同时表达唯一与非空，并成为其他表引用该行的稳定目标。一张表只能有一个主键，但主键可以由多列组成。

### `FOREIGN KEY`

外键保证子表中的引用能在父表找到，例如订单的 `account_id` 必须对应真实账号：

```sql
CONSTRAINT fk_orders_account
  FOREIGN KEY (account_id)
  REFERENCES accounts (id)
```

删除父记录时采用 `RESTRICT`、`CASCADE` 还是 `SET NULL` 是业务语义，不应为了“省代码”随意选择。本课可执行示例使用临时单表；MySQL 临时表不能参与外键约束，因此外键会在后续关系设计课程用安全的独立环境完整演示。

## MySQL 8.4 完整表定义

```sql
CREATE TEMPORARY TABLE learning_accounts (
  id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  email VARCHAR(255) NOT NULL,
  display_name VARCHAR(80) NOT NULL,
  status VARCHAR(20) NOT NULL DEFAULT 'pending',
  credit_limit DECIMAL(12, 2) NOT NULL DEFAULT 0.00,
  login_count INTEGER UNSIGNED NOT NULL DEFAULT 0,
  marketing_opt_in BOOLEAN NOT NULL DEFAULT FALSE,
  created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  last_login_at TIMESTAMP NULL,
  PRIMARY KEY (id),
  CONSTRAINT uq_learning_accounts_email UNIQUE (email),
  CONSTRAINT chk_learning_accounts_status
    CHECK (status IN ('pending', 'active', 'disabled')),
  CONSTRAINT chk_learning_accounts_credit_limit CHECK (credit_limit >= 0),
  CONSTRAINT chk_learning_accounts_login_count CHECK (login_count >= 0),
  CONSTRAINT chk_learning_accounts_marketing_opt_in
    CHECK (marketing_opt_in IN (FALSE, TRUE))
);
```

这里没有写死存储引擎、字符集和排序规则，因为临时表会继承当前服务器/数据库配置。真实迁移必须先检查团队的 MySQL 基线配置，并明确是否需要在表级覆盖，不能靠开发机默认值决定生产行为。

## PostgreSQL 18 完整表定义

```sql
CREATE TEMPORARY TABLE learning_accounts (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY,
  email VARCHAR(255) NOT NULL,
  display_name VARCHAR(80) NOT NULL,
  status VARCHAR(20) NOT NULL DEFAULT 'pending',
  credit_limit NUMERIC(12, 2) NOT NULL DEFAULT 0.00,
  login_count INTEGER NOT NULL DEFAULT 0,
  marketing_opt_in BOOLEAN NOT NULL DEFAULT FALSE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
  last_login_at TIMESTAMPTZ NULL,
  PRIMARY KEY (id),
  CONSTRAINT uq_learning_accounts_email UNIQUE (email),
  CONSTRAINT chk_learning_accounts_status
    CHECK (status IN ('pending', 'active', 'disabled')),
  CONSTRAINT chk_learning_accounts_credit_limit CHECK (credit_limit >= 0),
  CONSTRAINT chk_learning_accounts_login_count CHECK (login_count >= 0)
);
```

PostgreSQL 有原生布尔类型，因此不需要再用 `CHECK` 限定为 `0`/`1`。这里显式使用 `TIMESTAMPTZ`，表明两个时间列代表时间点。

## 安全运行配套脚本

两个脚本都会：

1. 创建当前连接可见的临时表。
2. 插入两行非敏感演示数据。
3. 查询最终值与默认值。
4. 使用 `IS NULL` 找出从未登录的账号。

连接断开后，临时表自动消失。脚本不包含 `DROP`、`DELETE`、`UPDATE` 或对永久表的操作。

```bash
mysql \
  --host=127.0.0.1 \
  --port=3306 \
  --user=app_writer \
  --password \
  app_learning \
  < examples/database/03-mysql-types-and-constraints.sql
```

```bash
psql \
  --host=127.0.0.1 \
  --port=5432 \
  --username=app_writer \
  --dbname=app_learning \
  --file=examples/database/03-postgresql-types-and-constraints.sql
```

`CREATE TEMPORARY TABLE` 仍需要相应权限。不要为了运行示例改用生产管理员账号；应让管理员在隔离的学习数据库中授予最小必要权限。

## 从演示 DDL 走向真实迁移

真实项目不应把 `CREATE TABLE` 随意粘贴到生产控制台。表结构变更应该进入版本化迁移：

1. 记录目标数据库产品与最低版本。
2. 明确字符集、排序规则、时区和 SQL 模式等环境基线。
3. 在独立数据库验证向前迁移、应用兼容性和回滚/修复策略。
4. 评估大表加列、建约束时的锁和扫描成本。
5. 由受控发布流程执行并记录结果。

本课脚本为了安全而使用临时表，它验证的是列定义、默认值和约束的基本行为，不替代生产迁移评审。

## 本课小结

- 数据类型应根据业务含义、范围、精度和运算方式选择，不根据示例外观猜测。
- `BIGINT` ID 可能超出 JavaScript 安全整数范围，API 可以用字符串传输不透明标识。
- 金额使用 `DECIMAL`/`NUMERIC` 等精确数值，不能使用浮点数碰运气。
- 字符集决定可表示字符，排序规则决定比较和排序；唯一文本的语义必须明确。
- PostgreSQL 原生支持布尔值，MySQL 的 `BOOLEAN` 映射为整数类型。
- 日期、当地日期时间和全球时间点是不同概念；MySQL 与 PostgreSQL 时间类型语义不能只看名字。
- `NULL` 表示未知或缺失，必须用 `IS NULL` 判断。
- `DEFAULT` 在列被省略时生效，不会自动替代显式 `NULL`。
- `NOT NULL`、`UNIQUE`、`CHECK`、主键和外键分别保护不同的数据不变量。
- 真实表结构变化应使用版本化迁移，不应手工修改生产库。
