# Cache-Aside 与缓存一致性

::: tip 第一次学习只抓住四件事
- **必须理解**：数据库是事实来源，缓存是可过期、可删除、可重建的副本；命中、未命中、回源和失效必须形成完整协议。
- **必须会做**：读时先查缓存，未命中回源并设置 TTL；写成功数据库后删除缓存，而不是把缓存当第二个事实来源。
- **必须完成**：为一个热点只读接口实现 Cache-Aside，并验证缓存不可用时仍能受控回源。
- **可以后看**：并发填充、延迟双删、版本键和热点保护。业务没有明确热点时，完全可以先不加缓存。
:::

Redis 最常见的角色，是保存 MySQL 或 PostgreSQL 数据的可过期副本。接口先读 Redis，命中就快速返回；未命中再查询数据库并回填。这种模式叫 Cache-Aside，也叫旁路缓存。

流程很直观，难点却不在 GET 和 SET，而在两个独立系统无法通过普通代码组成一个原子事务：数据库可能提交成功而缓存失效失败，读请求也可能在写请求提交后把旧快照重新写回缓存。

本课不追求一句“先更新数据库，再删除缓存”的口诀，而是建立一套可验证的推理方法：先确定真相来源和一致性目标，再画并发时间线、列失败矩阵，最后选择 TTL、重试、Outbox、CDC 或版本保护。

## 先定义缓存契约

为用户详情接口增加缓存前，先写出契约：

| 问题 | 示例答案 |
| --- | --- |
| 真相来源 | PostgreSQL `users` 表 |
| 缓存键 | `cache:user-api:v1:user:{id}` |
| 缓存内容 | 已脱敏的用户详情 JSON |
| TTL | 基础 5 分钟，加随机抖动 |
| 可接受陈旧时间 | 普通资料最多 30 秒 |
| 不可接受陈旧字段 | 封禁状态、权限、安全凭据 |
| Redis 不可用时 | 限流后回源数据库 |
| 数据库不可用但缓存命中时 | 是否允许短期 stale 由业务决定 |
| 更新后如何失效 | 提交后删除，失败事件可靠重试 |

如果“最多允许旧多久”没有答案，就无法判断单靠 TTL 是否足够，也无法定义告警阈值。缓存一致性不是绝对的“有或没有”，而是业务可接受的陈旧窗口和错误方向。

余额显示旧 1 秒与权限撤销晚 1 秒，风险完全不同。高风险字段可以不缓存、使用更短 TTL、独立键，或在关键写后强制从主库读取。

## Cache-Aside 的读路径

典型读取流程：

```text
请求 GET /users/1001
        │
        ▼
Redis GET cache:user-api:v1:user:1001
        │
   ┌────┴────┐
 命中       未命中
   │          │
反序列化   查询主数据库
   │          │
返回       序列化并 SET EX
              │
             返回
```

伪代码：

```js
async function getUser(userId) {
  const key = `cache:user-api:v1:user:${userId}`;

  const cached = await redis.get(key);
  if (cached !== null) {
    return JSON.parse(cached);
  }

  const user = await db.users.findById(userId);
  if (user === null) return null;

  await redis.set(key, JSON.stringify(toPublicUser(user)), {
    EX: ttlWithJitter(300)
  });
  return toPublicUser(user);
}
```

这段代码还缺少超时、错误分类、请求合并和负缓存，但已经体现三个基本原则：

1. 数据库是真相来源。
2. 只有实际被读取的数据进入缓存。
3. 回填失败不改变数据库读取结果，只影响后续性能。

### 命中不是“键存在”这么简单

读取后还要验证：

- value 是否能反序列化。
- schema 版本是否兼容当前代码。
- 必要字段是否存在，类型是否正确。
- value 是否是负缓存标记。
- 数据是否超过应用定义的软过期时间。

损坏缓存通常应视作 miss 并上报指标，而不是让整个接口持续返回 500。删除损坏键属于修复动作，要确保只操作由当前服务拥有的键。

## Cache-Aside 的写路径

常见写法是：

```text
校验请求
   │
开启数据库事务
   │
UPDATE users ...
   │
COMMIT 成功
   │
DEL 对应缓存键
```

伪代码：

```js
async function updateUser(userId, command) {
  const updated = await db.transaction(async (tx) => {
    return tx.users.update(userId, command);
  });

  await invalidateUserCache(userId);
  return updated;
}
```

关键顺序是先让数据库提交，再使缓存失效。缓存删除后，下一个读取请求自然从数据库取得新值并回填。

### 失效必须发生在提交之后

如果事务提交前删除缓存：

| 时刻 | 写请求 | 读请求 |
| --- | --- | --- |
| T1 | 删除缓存 | |
| T2 | 数据库事务尚未提交 | 缓存 miss |
| T3 | | 从数据库读到旧值并回填 |
| T4 | 提交新值 | 缓存仍是旧值 |

即使没有故障，也可能稳定留下旧缓存。因此不能在事务函数内部、COMMIT 之前发出普通 DEL。

ORM 的“保存完成”回调也不一定代表外层事务已经提交。应使用 after-commit 机制，或让应用在事务 API 成功返回后安排失效。

### 为什么通常删除，而不是直接更新缓存

删除有几个优势：

- 下一次读取从真相来源重新构建，避免写接口漏算派生字段。
- 多个写入口只需遵循同一失效协议。
- 删除不存在的键天然幂等。
- 低频数据更新后不一定再次被读取，可以节省缓存内存。

直接更新缓存并非永远错误。若新值已在数据库提交后完整可得、读取非常频繁且构建成本高，可以写穿或主动刷新。但必须解决多个写请求乱序覆盖、派生视图重建和写缓存失败。

## 为什么“先写缓存，再写数据库”危险

若缓存先写新值、数据库随后失败，接口可能从缓存读到一个从未提交的状态。即使数据库最终成功，缓存写和数据库写之间仍存在其他请求观察到中间状态的窗口。

```text
SET cache = new
        │
        ├── 其他请求读到 new
        │
UPDATE database 失败
```

除非缓存本身就是真相来源，并有独立的持久化和异步落库协议，否则不要把缓存成功当成业务写入成功。

## 跨系统原子性缺口

数据库 COMMIT 与 Redis DEL 是两个网络操作，不可能靠 `try/catch` 变成同一原子提交：

```text
COMMIT database  ──成功──> 新值成为真相
DEL cache        ──超时──> 不知道是否执行
```

此时不能回滚已提交的数据库事务。即使代码尝试“补偿写回旧值”，也可能覆盖其他事务的新提交。

需要把结果分成四类：

| 数据库提交 | 缓存失效 | 结果 |
| --- | --- | --- |
| 失败 | 未执行 | 真相未变化，无需失效 |
| 成功 | 成功 | 下一读回源新值 |
| 成功 | 明确失败 | 旧缓存保留，必须重试或等待 TTL |
| 成功 | 结果未知 | DEL 可能已执行；幂等重试通常安全 |

DEL 是幂等操作，因此网络超时后可以有限重试。问题在于应用进程可能在 COMMIT 后、发出 DEL 前崩溃，这需要持久化事件机制，而不是仅靠内存重试。

## TTL 是最后防线，不是同步机制

所有派生缓存都应有有限 TTL。若失效消息永远丢失，TTL 至少让旧值最终退出。

但 TTL 不能保证更新后立即一致：5 分钟 TTL 允许最坏接近 5 分钟的陈旧值。把 TTL 缩到 1 秒会提高数据库负载，并可能在热门键同时过期时制造回源洪峰。

TTL 应由以下因素共同决定：

- 业务允许的最大陈旧窗口。
- 数据更新频率。
- 数据库可承受的 miss 率。
- value 构建成本与大小。
- Redis 容量和淘汰策略。
- 失效消息可靠性。

为 TTL 加随机抖动可以分散大量键的集中失效，例如基础 300 秒加 0～30 秒随机值。抖动改变的是负载分布，不会修复写后读一致性。

## 最容易漏掉的并发竞态：旧读回填

即使严格“先提交数据库，再删除缓存”，仍存在旧读回填窗口：

| 时刻 | 读请求 A | 写请求 B |
| --- | --- | --- |
| T1 | 缓存 miss | |
| T2 | 从数据库读到旧版本 v1，但尚未回填 | |
| T3 | | 数据库提交 v2 |
| T4 | | 删除缓存 |
| T5 | 把先前读到的 v1 SET 进缓存 | |

最终缓存是 v1，直到 TTL、下一次失效或其他修复发生。这个窗口通常较短，但在慢查询、GC 暂停、网络抖动或大 value 序列化时会扩大。

配套模型会确定性复现这个时间线：

```bash
node examples/database/redis/02-cache-aside-consistency.mjs
```

它不连接真实 Redis 或数据库，只把两个系统建成内存状态机，因此不会修改任何外部数据。

## 处理旧读回填的方案

没有一个方案适用于所有一致性要求，应按风险选择。

### 接受短暂陈旧并用 TTL 收敛

适合头像、简介、非关键配置等允许短暂旧值的数据。实现最简单，吞吐也最好。需要实际测量旧值持续时间，而不是口头说“最终会一致”。

### 延迟再次删除

写请求提交并删除后，延迟一段时间再删除同一键，试图清理慢读回填的旧值。这通常叫延迟双删。

它可以降低特定竞态概率，但不是严格证明：

- 延迟长度很难覆盖所有慢读。
- 进程崩溃会丢失内存定时任务。
- 延迟过短无效，过长扩大陈旧窗口。
- 高频写入时，旧的延迟任务可能删除刚回填的新值，增加 miss。

若采用，第二次删除应由可靠延迟队列执行，并将其视为概率优化，不是强一致保证。

### 代际号与条件回填

为实体维护 Redis generation：

```text
读：读取 generation=g1 → 查数据库 → 仅当 generation 仍为 g1 时写缓存
写：数据库提交 → 原子增加 generation → 删除 value 键
```

“比较 generation 并 SET”必须在 Redis 内以 Lua/Function 或合适原子命令完成，否则比较和写入之间又有竞态。

这种方法能阻止已经观察到旧 generation 的慢读在失效后回填，但数据库提交与 generation 增加仍不是原子操作；失效事件延迟时仍存在窗口。它增加 Redis 操作、额外键和集群同 slot 约束，只应用于确有价值的热点或关键数据。

### 数据版本随 value 存储

缓存内容携带数据库版本：

```json
{
  "schemaVersion": 2,
  "entityVersion": 37,
  "cachedAt": "2026-07-14T10:00:00Z",
  "data": {"id": 1001, "name": "Alice"}
}
```

写缓存时只允许更高实体版本覆盖更低版本，可以减少乱序事件倒退。版本必须由真相数据库单调生成，例如行 version 或变更序列；客户端时间戳会受时钟偏差影响，不适合直接当全局顺序。

版本比较同样要在 Redis 内原子完成。只把 version 放进 JSON、仍然普通 SET，并不能阻止旧写覆盖。

### 关键读绕过缓存

用户刚完成修改后的响应可以直接返回数据库提交结果，或让随后少量读取走主库。权限撤销、支付状态确认等关键路径可能完全不读通用缓存。

这是一种业务一致性设计：不强迫所有请求付出强一致成本，只在明确场景选择真相读取。

## 可靠失效：Transactional Outbox

Outbox 模式把业务更新和“需要使缓存失效”的事件写入同一个数据库事务：

```text
BEGIN
UPDATE users SET ... WHERE id = 1001;
INSERT INTO outbox(event_id, type, aggregate_id, payload, created_at)
VALUES (..., 'UserChanged', '1001', ...);
COMMIT
```

后台发布器读取未处理事件，删除缓存并记录处理结果。因为业务行和事件同事务提交，不会出现“数据库已更新但应用在创建失效任务前崩溃而永久丢事件”。

消费者必须幂等：同一事件可能因超时或崩溃重复处理。DEL 天然适合重复执行；若同时刷新缓存，则要使用版本防止旧事件覆盖新值。

Outbox 需要治理：

- 唯一事件 ID。
- 批量领取与并发消费者协议。
- 失败重试、退避和死信。
- 未处理事件延迟告警。
- 已处理记录归档或清理。
- 事件 payload 的 schema 版本。

它提供可靠的最终一致性桥梁，不提供 Redis 与数据库的瞬时原子可见性。

## CDC：从数据库变更日志驱动失效

Change Data Capture 从 MySQL binlog、PostgreSQL WAL 或托管变更流捕获已提交变化，再发布缓存失效事件。

优点：

- 覆盖多个写入口，不要求每段业务代码都记得 DEL。
- 只处理已提交变更。
- 可供搜索索引、审计和其他派生视图复用。

代价：

- 架构和运维复杂度显著增加。
- 变更事件可能重复、延迟或乱序。
- 需要把表级变化映射为缓存键。
- schema 变更、重放位置和灾难恢复必须管理。

CDC 适合已有事件基础设施、写入口很多或缓存重建链复杂的系统。单体小应用不必为了一个用户缓存立即引入整套平台。

## 消息顺序与重复投递

假设用户连续更新为 v2、v3，两条事件到达顺序可能是 v3、v2。

若消费者只 DEL，两次删除都是安全的，只会增加一次 miss。若消费者把事件 payload 主动 SET 到缓存，旧 v2 可能覆盖新 v3。

因此：

- 失效优先于“按事件直接更新”通常更容易保持正确。
- 主动更新必须携带单调版本并做原子比较。
- 消费成功标记要在副作用成功后提交。
- 不把消息“至少一次”误称为业务“恰好一次”。

## 缓存整行、聚合结果还是查询结果

### 实体缓存

```text
cache:user:v1:1001
```

失效映射简单：用户 1001 变化就删除对应键。适合按 ID 查询。

### 聚合缓存

```text
cache:user-detail:v1:1001
```

可能包含用户、角色和组织信息。任何依赖变化都要失效同一个键。依赖图越复杂，遗漏失效的风险越大。

### 查询缓存

```text
cache:user-list:v3:status=active&page=1
```

列表键数量可能随筛选组合爆炸，一名用户变化可能影响多个页。不能靠在生产使用 KEYS 模糊扫描删除。可选方案包括短 TTL、命名空间版本、标签集合、按稳定查询预计算，或干脆不缓存高组合查询。

缓存粒度越大，命中时节省越多，但失效影响范围也越大。应从高频、构建昂贵且依赖清晰的读开始。

## 缓存键必须包含所有结果维度

若接口结果受以下因素影响，键也必须能区分它们：

- 资源 ID。
- 租户 ID。
- 权限或用户身份。
- 语言、货币和时区。
- 查询参数、排序和分页。
- API/schema 版本。

漏掉 tenant ID 可能导致跨租户数据泄漏；漏掉权限维度可能把管理员视图返回给普通用户。缓存键不是简单性能细节，而是授权边界的一部分。

不要把原始访问令牌或敏感个人信息直接放进键。需要区分身份时可使用稳定内部 ID 或不可逆摘要，并控制日志输出。

## 负缓存与不存在状态

数据库查不到记录时，如果完全不缓存，恶意或偶然的大量不存在 ID 会持续打到数据库。可以缓存短期“NOT_FOUND”标记：

```json
{"kind":"not_found","cachedAt":"..."}
```

负缓存 TTL 通常短于正常对象，因为记录可能很快被创建。创建成功后必须使负缓存失效。

需要区分：

- Redis GET 返回 null：缓存 miss。
- 缓存中明确的 NOT_FOUND：数据库已查过不存在。
- 数据库超时：未知，不能当不存在缓存。
- 无权限：不能用共享 NOT_FOUND 随意缓存，否则可能污染其他用户视图。

下一课会把负缓存放进缓存穿透、布隆过滤器和请求合并场景继续分析。

## Read-Through、Write-Through 与 Write-Behind

| 模式 | 读取 miss 谁加载 | 写入路径 | 主要边界 |
| --- | --- | --- | --- |
| Cache-Aside | 应用 | 数据库后失效 | 灵活；一致性由应用负责 |
| Read-Through | 缓存层/加载器 | 另行定义 | 应用简单；加载器成为基础设施 |
| Write-Through | 缓存层同步写后端 | 缓存入口 | 写延迟更高；仍需处理部分失败 |
| Write-Behind | 先写缓存，异步落后端 | 异步批量 | 吞吐高；有数据丢失与冲突风险 |

模式名称不能替代故障设计。Write-Through 也需要说明缓存写成功、数据库写失败时怎么办；Write-Behind 必须定义持久队列、顺序、幂等、积压和恢复。

对以关系数据库为真相来源的普通 CRUD 接口，Cache-Aside 通常是最容易理解和演进的起点。

## Redis 不可用时如何降级

Cache-Aside 的正确性来源是数据库，所以 Redis 故障时理论上可以绕过缓存。但全量回源可能击穿数据库，必须有限制：

- 为数据库回源设置并发上限。
- 对热点键进行单请求合并。
- 使用断路器快速跳过持续失败的 Redis 调用。
- 保留严格的数据库查询超时和连接池上限。
- 对非关键接口返回降级结果或限流。
- 让恢复过程逐步预热，不瞬间放开全部流量。

“缓存可选”不等于“缓存故障没有影响”。容量规划必须评估无缓存时数据库能承受多少流量，以及能持续多久。

### 是否允许 stale-if-error

有些只读接口可以在数据库故障时返回刚过软 TTL、但未过硬 TTL 的旧缓存：

```text
软 TTL：正常应刷新
硬 TTL：超过后绝不能再返回
```

这能提高可用性，却显式牺牲新鲜度。权限、余额、库存确认等路径通常不应使用；页面配置、公共内容可以按业务评估。响应和指标应标识降级状态。

## 多级缓存的一致性

应用进程内 L1 缓存加共享 Redis L2 缓存可以进一步降低延迟：

```text
进程内 L1 -> Redis L2 -> 数据库
```

但数据库更新后删除 Redis，不会自动删除每个应用实例的本地副本。需要短 L1 TTL、广播失效或 Redis client-side caching tracking。

Redis 官方 client-side caching 能追踪客户端读取过的键，并在键修改、过期或淘汰时发送 invalidation。仍要处理失效连接断开和消息/读取响应竞态：失效通道丢失时，应清空本地缓存；使用双连接时，要防止旧 GET 响应在 invalidation 后重新写入 L1。

多级缓存增加的是另一个一致性层，只有在 Redis 网络延迟或访问量确实成为瓶颈时才值得引入。

## 数据序列化与 schema 演进

缓存 value 应包含或由键前缀表达 schema 版本。发布时要考虑旧代码和新代码并存：

- 新代码能否读取旧格式？
- 旧代码会不会覆盖新格式？
- 回滚后能否读取新格式？
- 未知字段是忽略还是报错？

安全方式之一是新版本使用新键前缀，让旧缓存按 TTL 自然消失。不要在高峰发布时同步扫描并改写所有键。

序列化前就应完成脱敏。缓存不应保存数据库实体中的密码摘要、内部令牌和无关敏感字段；否则 Redis ACL 或日志泄漏会扩大影响面。

## 写后读一致性

用户刚修改昵称，随即 GET 仍读到旧缓存，会产生明显体验问题。可用策略包括：

1. 更新接口直接返回已提交的新对象。
2. 提交成功并确认缓存失效后再响应。
3. 在短期会话上下文中让该用户的读取绕过缓存。
4. 缓存中使用单调版本，拒绝低版本。
5. 若失效失败，返回成功但记录异步修复，或按业务把请求视为部分失败。

是否等待 Redis DEL 成功取决于接口契约。数据库已提交后，即使 DEL 失败也不能声称数据库写入被回滚；应返回业务更新成功并异步修复，或返回明确的“已保存但缓存同步待处理”内部状态，避免客户端盲目重试整个写操作。

## 可观测性

只看命中率无法判断缓存健康。至少监控：

### 读路径

- hit、miss、negative hit、损坏 value 数量。
- Redis GET 延迟、错误和超时。
- 回源数据库次数、延迟和并发。
- 回填成功、失败和 value 大小。
- stale-if-error 使用次数。

### 写路径

- 数据库提交后失效成功率与延迟。
- DEL 重试次数和最终失败。
- Outbox 未处理数量、最老事件年龄。
- CDC 处理延迟、重复和乱序丢弃。
- 版本保护拒绝旧回填的次数。

### 容量

- Redis used memory、evictions、expired keys。
- 热键、大键和网络带宽。
- 无缓存降级时数据库连接池与 CPU 余量。

日志中带上请求 ID、实体 ID 的安全摘要、缓存键模式、数据库版本和事件 ID，但不要打印完整敏感 value。

## 测试缓存一致性

普通单线程单元测试只能证明 happy path。并发测试应能控制每一步暂停：

```text
读 miss
读数据库旧值
暂停
写事务提交
缓存失效
恢复旧读并回填
断言最终缓存
```

还应注入：

- COMMIT 后进程退出。
- DEL 执行前超时、执行后响应丢失。
- Outbox 事件重复和乱序。
- Redis 整体不可用。
- 缓存 value 损坏或 schema 不兼容。
- 大量同键并发 miss。

测试目标不只是“最终返回正确”，还要断言数据库调用次数、陈旧窗口、重试上限和连接池不会失控。

## 配套状态模型

配套文件 `examples/database/redis/02-cache-aside-consistency.mjs` 使用内存 FakeDatabase、FakeCache 和 Outbox，不依赖第三方包。它验证：

- miss 后回填与后续 hit。
- 数据库提交后删除缓存。
- 慢读在删除后回填旧值的竞态确实存在。
- generation 条件回填能拒绝已失效的旧读。
- DEL 失败时，持久 Outbox 重试能让缓存最终收敛。

运行：

```bash
node examples/database/redis/02-cache-aside-consistency.mjs
```

模型用于验证事件顺序，不代表生产 Redis 客户端实现。真实 generation 的增加与 value 删除、generation 比较与 SET，都必须分别由 Lua/Function 等服务端逻辑形成所需原子边界；Outbox 也必须与业务更新共享数据库事务。

## 实施清单

### 进入开发前

1. 明确数据库真相来源和缓存拥有者。
2. 为每类数据定义最大可接受陈旧时间。
3. 写出键的所有结果维度与 schema 版本。
4. 评估无缓存时数据库容量与降级策略。

### 读路径

1. GET 设置短超时，异常时按策略回源或降级。
2. 验证缓存格式，损坏 value 视作 miss 并告警。
3. 只缓存成功的确定结果；数据库超时不能负缓存。
4. SET 与 TTL 使用单条命令。
5. 热点 miss 需要请求合并与回源限流。

### 写路径

1. 先提交数据库，再使缓存失效。
2. 不在提交前回调中 DEL。
3. 把失效设计为幂等操作。
4. 对进程崩溃窗口使用 Outbox 或 CDC。
5. 主动刷新缓存时携带单调版本并原子比较。

### 运行阶段

1. 同时观察命中率、回源量和陈旧指标。
2. 监控失效失败、Outbox 积压与 CDC 延迟。
3. 演练 Redis 故障和全量冷缓存恢复。
4. 定期核对缓存拥有者、键前缀和数据敏感性。

## 常见误区

### “先更新数据库，再删除缓存就强一致了”

它是合理默认顺序，但数据库提交和 Redis 删除并不原子，慢读仍可能在删除后回填旧值。需要按业务陈旧容忍度增加 TTL、可靠失效或版本保护。

### “DEL 失败就回滚数据库”

数据库一旦 COMMIT，普通应用代码不能安全撤销为旧状态。应将删除视为幂等异步修复，避免补偿覆盖新的并发更新。

### “TTL 越短越一致”

短 TTL 缩小最坏陈旧窗口，也提高 miss、数据库负载和热点回源风险。它必须与失效可靠性、容量和抖动共同设计。

### “更新缓存比删除缓存更及时”

主动更新可能被乱序写覆盖，也可能遗漏其他派生字段和查询键。只有能证明完整值、顺序与失败处理时才采用。

### “缓存挂了直接回源就行”

缓存吸收的流量可能远超数据库冗余容量。没有回源限流和请求合并，降级会把 Redis 故障扩大成数据库故障。

## 本课小结

- Cache-Aside 以数据库为真相来源，miss 时加载，写提交后使缓存失效。
- 缓存失效必须在数据库 COMMIT 后执行，删除通常比主动更新更容易保持正确。
- 数据库提交与 Redis 删除不是原子操作，进程崩溃和网络超时需要可靠重试。
- TTL 是陈旧值的最后收敛边界，不是实时同步机制。
- 慢读可能在写后删除完成后重新回填旧值，必须按风险选择接受、二次删除、generation、版本或绕过缓存。
- Outbox 与 CDC 能可靠传播已提交变更，但仍是最终一致，需要处理重复、延迟和乱序。
- 查询缓存和聚合缓存依赖多，失效难度高于按 ID 的实体缓存。
- 键必须包含租户、权限、语言、参数和 schema 版本等所有结果维度。
- Redis 故障回源需要并发上限、断路器和数据库容量保护。
- 一致性测试要控制并发时间线并注入部分失败，不能只验证串行 happy path。
