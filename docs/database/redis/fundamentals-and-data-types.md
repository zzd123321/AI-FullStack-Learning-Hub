# Redis 基础与核心数据类型

关系数据库擅长持久保存结构化事实、维护约束并支持复杂查询。Redis 则把键映射到内存中的数据结构，提供低延迟访问和丰富的原子命令，常用于缓存、会话、计数、排行榜、限流和轻量任务协调。

Redis 很快，但“快”不是数据模型。是否适合 Redis，要看数据是否允许丢失或过期、是否需要关系约束、访问模式能否用键直接定位、内存成本是否可控，以及故障恢复后如何重建。

## 本课目标与阅读路线

完成本课后，你应能按访问模式而不是按“哪个命令记得住”选择 Redis 数据结构，说明键、TTL、原子命令与持久化分别解决什么问题，并能判断一份数据应留在关系数据库、放入 Redis，还是需要两者配合。后续缓存一致性、限流和队列设计都会以这些边界为前提。

## Redis 不只是字符串缓存

最简单的 Redis 模型是：

```text
key -> value
```

但 value 不只有字符串，还可以是 Hash、List、Set、Sorted Set、Stream 等结构。每种结构都有自己的命令、复杂度和适用语义。

例如同一份用户信息可以有几种建模方式：

```text
String: user:1001 -> '{"id":1001,"name":"Alice"}'
Hash:   user:1001 -> {name: Alice, status: active}
Set:    user:1001:roles -> {member, editor}
```

选择结构时先写访问动作：是整块读写用户 JSON、单独增加某个字段、判断角色是否存在，还是按分数取排行榜？不要先选择“最熟悉的 Hash”，再勉强适配所有需求。

## Redis 在系统中的三种角色

| 角色 | 数据真相在哪里 | Redis 故障后的期望 |
| --- | --- | --- |
| 派生缓存 | MySQL/PostgreSQL 或其他服务 | 可以从真相来源重建 |
| 临时协调状态 | Redis 与业务协议共同定义 | 允许过期，但需处理重复和超时 |
| 主数据存储 | Redis 本身 | 必须设计持久化、复制、备份与恢复 |

第一种最常见，也最容易建立清晰边界。第三种并非绝对错误，但不能因为开启 AOF 就默认获得关系数据库的约束、事务隔离、任意查询和灾难恢复能力。

设计前必须回答：Redis 不可用时接口是降级、失败还是回源？Redis 数据全部丢失时能否重建？如果不能，恢复点和恢复时间目标是什么？

## 请求是怎样到达 Redis 的

应用通过 TCP 或 TLS 连接发送 Redis Serialization Protocol（RESP）编码的命令，服务端执行后返回响应。正规客户端库会负责协议编码、连接复用和响应解析，不应通过字符串拼接模拟 `redis-cli` 命令。

一次普通调用包含：

1. 从连接池取得连接。
2. 将命令和参数编码为 RESP。
3. 通过网络发送到 Redis。
4. Redis 查找键并执行命令。
5. 响应返回，客户端解码。
6. 连接归还连接池。

即使命令本身只耗费微秒，跨主机网络往返、连接排队、TLS、序列化和响应体传输仍可能占主要延迟。因此后端必须分别监控服务端命令耗时、网络 RTT 和连接池等待。

## “单线程”需要准确理解

Redis 的核心命令执行模型长期以来以事件循环中串行执行命令为重要特征，这让单条命令天然拥有清晰的原子执行边界。现代 Redis 还可能使用其他线程处理网络 I/O、持久化、释放内存等工作，因此简单说“Redis 整个进程只有一个线程”并不准确。

对应用最关键的结论是：一个耗时很长的命令会延迟同一执行路径上的其他请求。平均 O(1) 不代表任何输入规模都安全，O(N) 命令也不代表必然不能用；必须把 N、响应大小和调用频率代入判断。

常见延迟来源包括：

- 对巨大集合执行一次返回全部成员的命令。
- 读取或写入超大 value。
- 运行时间过长的 Lua 脚本或函数。
- 在生产键空间使用 `KEYS *`。
- 持久化 fork、内存压力、交换分区或宿主机抖动。
- 热点键把大量流量集中到单节点或单分片。

## 键空间就是你的数据模型

Redis 没有关系表和外键，键名承担了实体定位、命名空间、版本和租户边界。

推荐使用可读的冒号分段：

```text
prod:user-api:v1:user:1001
prod:user-api:v1:user:1001:roles
prod:order-api:v2:order:9001
```

一个键名通常可以包含：

```text
环境:服务:模式版本:实体:标识:用途
```

不一定每个项目都需要这么多段，但必须形成团队规范。键太短难以排障，键太长则在大量小 value 场景产生明显内存开销。

### 不要把逻辑数据库当命名空间

单机 Redis 的 `SELECT 0`、`SELECT 1` 提供编号逻辑数据库，但它们不提供权限、容量、故障和性能隔离，Redis Cluster 也只支持数据库 0。应用应使用键前缀或独立实例/集群表达真正边界，不要依赖逻辑库编号区分生产租户。

### 键模式应支持演进

缓存序列化格式变化时，可把版本放进前缀：

```text
user-api:v2:user:1001
```

发布新版本后读取新键，旧键让 TTL 自然淘汰，通常比扫描并原地改写所有缓存安全。模式版本不应混入每次部署编号，否则会制造无意义的全量冷缓存。

## String：字节序列与原子计数

String 是最基础的 value 类型，本质上是二进制安全的字节序列。文本、JSON、序列化对象、整数计数器都可以存入 String。

### 缓存完整对象

```redis
SET user-api:v1:user:1001 "{\"id\":1001,\"name\":\"Alice\"}" EX 300
GET user-api:v1:user:1001
```

`EX 300` 在同一条 SET 命令中设置 300 秒过期，避免 SET 成功而后续 EXPIRE 未执行留下永久键。

完整 JSON 适合总是整体读取、字段一起失效的对象。部分字段频繁更新时，每次反序列化并重写整个 JSON 可能浪费网络和 CPU。

### 条件写入

```redis
SET request:idem:4f7a processing EX 60 NX
```

`NX` 表示仅键不存在时设置。返回成功说明当前调用取得标记，返回空值说明已有调用写入。它可以构成幂等或互斥协议的一部分，但可靠锁还必须考虑唯一令牌、超时、持有者崩溃和只能由持有者释放等问题。

### 原子计数

```redis
INCR api:counter:2026-07-14
INCRBY api:counter:2026-07-14 10
```

不要用 `GET`、应用加一、`SET` 实现并发计数，那会丢失更新。INCR/INCRBY 在服务端以单条命令完成读改写。

计数是否可以在过期或故障时丢失，需要由业务决定。财务余额和库存真相通常不应只保存在一个缓存计数器中。

## Hash：一个键下的字段映射

Hash 类似小型字段字典：

```redis
HSET user-api:v1:user-profile:1001 name Alice status active
HGET user-api:v1:user-profile:1001 status
HGETALL user-api:v1:user-profile:1001
```

适合：

- 需要单独读取或原子修改字段。
- 字段数量可控，生命周期一致。
- 字段值仍可表示为字符串。

`HINCRBY` 可以原子增加 Hash 中的整数值。Hash 不是关系表：没有列类型、NOT NULL、外键或跨 Hash 查询优化。把每个业务对象无限扩展为几千字段的大 Hash，也会形成大键。

过期通常首先按整个键设计。较新 Redis 版本提供字段级过期能力，但客户端、服务版本和运维工具支持必须核对；需要广泛兼容时，不要假设每个 Hash 字段都能独立 TTL。

## List：按插入顺序排列的字符串

List 维护元素的插入顺序，支持从两端压入和弹出：

```redis
LPUSH notification:user:1001 event-3
RPUSH notification:user:1001 event-4
LRANGE notification:user:1001 0 19
```

常见用途：

- 有界的最近事件列表。
- 简单先进先出或后进先出结构。
- 使用阻塞弹出实现等待新元素。

列表按位置访问中间元素的成本会随距离增长，不适合随机访问大型集合。`LRANGE key 0 -1` 对大列表会返回全部数据，既阻塞执行又消耗网络和客户端内存。

用 List 实现任务队列时要考虑消费者取出后崩溃。如果直接弹出即从队列消失，任务可能丢失。可靠队列还需要处理中状态、确认、重投、重复消费和死信；复杂场景通常应评估 Redis Streams 或专用消息系统。

## Set：无序且不重复的成员集合

Set 中每个成员唯一，适合成员判断和集合运算：

```redis
SADD user:1001:roles member editor
SISMEMBER user:1001:roles editor
SCARD user:1001:roles
```

典型场景：

- 用户已拥有的标签或角色缓存。
- 当天独立访问者的精确集合。
- 共同关注、交集、并集和差集。

集合成员无业务顺序，不能依赖 SMEMBERS 返回顺序。大集合不要一次 SMEMBERS 全量取回；使用 SSCAN 增量遍历，但要接受遍历期间集合变化带来的有限一致性保证。

若只需要近似去重计数而不需要取回具体成员，可评估 HyperLogLog，以精度换显著更小的空间。

## Sorted Set：唯一成员加分数排序

Sorted Set 中成员唯一，每个成员关联一个浮点 score，并按 score 排序；分数相同再按成员字典序排列。

```redis
ZADD leaderboard:weekly 1200 user:1001 980 user:1002
ZINCRBY leaderboard:weekly 30 user:1002
ZRANGE leaderboard:weekly 0 9 REV WITHSCORES
```

典型场景：

- 排行榜。
- 按时间戳排列的延迟任务。
- 滑动窗口限流中的请求时间点。
- 按优先级取成员。

score 使用浮点表示，超大整数的精确表达范围有限。将毫秒时间戳或业务分数放入 score 前，要验证精度、并列排序和相同成员更新分数的语义。

Sorted Set 不保存成员的完整业务对象。常见做法是成员保存稳定 ID，详情放在 String/Hash 或真相数据库中，但这会形成多键一致性问题，需要明确更新顺序与失败补偿。

## Stream：可消费的追加日志

Stream 是按 ID 追加的记录序列，支持范围读取、阻塞读取和消费组。与 List 相比，它更适合需要消费确认、消费者分工和未确认消息追踪的场景。

```redis
XADD order-events MAXLEN ~ 100000 * type paid orderId 9001
```

`MAXLEN ~` 是近似裁剪，用于控制增长。仅使用 XADD 而不设计裁剪或归档，Stream 会持续占用内存。

消费组不是“恰好一次”魔法。消息可能因超时、消费者崩溃或认领而重复投递，业务处理仍需幂等。跨系统写数据库和确认消息也需要处理部分成功。

## 其他结构的定位

| 结构 | 适合解决的问题 | 重要边界 |
| --- | --- | --- |
| Bitmap | 大量布尔状态、签到位图 | 基于 String，偏移决定空间 |
| Bitfield | 紧凑整数与位级计数 | 明确位宽、符号和溢出策略 |
| HyperLogLog | 大基数近似去重计数 | 不能列出成员，有统计误差 |
| Geospatial | 经纬度附近搜索 | 基于 Sorted Set，非通用 GIS |
| JSON | 服务端操作结构化 JSON | 核对发行版、模块和客户端支持 |
| 概率数据结构 | 布隆过滤、频率估计等 | 误判率、容量和模块支持需设计 |

选择高级结构前，先确认目标 Redis 发行版是否包含相应能力。不能因为官网文档展示某种结构，就假定所有托管服务、开源版本和本地镜像都已启用。

## 单条命令原子，不等于业务流程原子

Redis 执行一条 INCR、HSET 或条件 SET 时，不会被另一个普通命令插入其中。但一次接口通常包含多条命令和数据库访问：

```text
GET cache
查询 PostgreSQL
SET cache
返回响应
```

这个流程不是 Redis 事务，也不是跨数据库事务。多个请求可以交错执行，外部数据库提交与 Redis 写入也可能只成功一个。

优先寻找表达完整意图的单条命令，例如：

- `INCR` 代替 GET + SET。
- `SET ... NX EX` 代替 EXISTS + SET + EXPIRE。
- `HINCRBY` 代替 HGET + HSET。
- `ZPOPMIN` 代替 ZRANGE + ZREM。

如果没有单命令表达，再考虑 MULTI/EXEC、WATCH、Lua 脚本或 Redis Functions，并控制执行时间。

## MULTI/EXEC 不是关系数据库事务

Redis 事务以 MULTI 开始，后续命令先排队，EXEC 时按顺序连续执行：

```redis
MULTI
HSET session:abc user_id 1001 status active
EXPIRE session:abc 1800
EXEC
```

其他客户端的命令不会插入 EXEC 执行的命令序列中。但 Redis 事务没有关系数据库式回滚：若某条已排队命令在执行时因 value 类型不匹配而报错，其他命令仍可能执行。

命令在入队阶段就能识别的语法或参数数量错误，可能导致 EXEC 拒绝整个事务；运行时错误则不会撤销已经执行的命令。因此应用必须提前验证命令与类型，并检查 EXEC 返回数组中的每一个结果。

DISCARD 只丢弃尚未 EXEC 的命令队列，不是撤销已经提交的事务。

## WATCH：乐观并发控制

WATCH 让 EXEC 具有 compare-and-set 语义：若被监视键在 WATCH 后、EXEC 前发生变化，EXEC 不执行事务并返回空结果。

```text
WATCH account:1001
GET account:1001
应用计算新值
MULTI
SET account:1001 新值
EXEC
```

失败后必须重新读取并有限重试。WATCH 状态绑定连接，所以使用连接池时，从 WATCH 到 EXEC/UNWATCH 必须使用同一条专用连接，异常路径要正确释放监视状态。

若已有 INCR、ZINCRBY 等原子命令，应优先使用，不要为了“像数据库事务”而增加 WATCH 往返和冲突重试。

## Lua 脚本与 Functions

服务端脚本可以读取键、计算并写回，使多步逻辑作为一个不可被普通命令插入的执行单元。适合条件更新、令牌桶、比较令牌后释放锁等逻辑。

代价是脚本运行期间会占用命令执行路径。脚本必须：

- 只访问明确的有限键和有限数据量。
- 不根据无界集合做长循环。
- 通过 KEYS 和 ARGV 传入键与参数，不拼接不可信脚本文本。
- 在 Redis Cluster 中考虑所有相关键是否位于同一 hash slot。
- 具备版本管理、测试、超时监控和失败降级。

Lua 也不能把 Redis 与 MySQL/PostgreSQL 合并成同一个原子事务。

## Pipelining：减少 RTT，不保证原子性

流水线允许客户端连续发送多条命令，再批量读取响应，减少逐条请求的网络往返：

```text
发送 GET key:1
发送 GET key:2
发送 GET key:3
一次读取多个响应
```

它与事务的区别：

| 能力 | Pipelining | MULTI/EXEC |
| --- | --- | --- |
| 减少网络往返 | 是 | 客户端实现可能顺带做到 |
| 阻止其他客户端命令插入 | 否 | EXEC 执行期间是 |
| 提供回滚 | 否 | 否 |
| 需要读取每个响应 | 是 | 是 |

批次不能无限大。服务端要暂存响应，客户端也要保存请求与结果；超大 pipeline 会造成内存峰值、尾延迟和单连接独占。应根据响应大小与延迟目标压测批次。

## TTL：让临时数据有生命周期

```redis
EXPIRE user-api:v1:user:1001 300
TTL user-api:v1:user:1001
```

TTL 常见返回：

- 正数：剩余秒数。
- `-1`：键存在，但没有过期时间。
- `-2`：键不存在。

过期时间是键级生命周期，不是定时任务的精确调度器。Redis 通过访问时检查和主动过期机制删除到期键，物理删除时刻可能略晚于理论截止时间；应用读取时不会把逻辑已过期键当作有效结果。

### 常见 TTL 陷阱

1. 先 SET 再 EXPIRE，中间崩溃会留下永久键；优先使用 SET 的 EX/PX 选项或 MULTI。
2. 某些覆盖键的命令会替换 value 并影响原 TTL，要逐命令核对。
3. 所有热点键同一时刻过期会造成回源洪峰，应加入合理随机抖动。
4. 续期会把临时会话变成长寿键，需要绝对有效期上限。
5. TTL 不等于业务删除或合规擦除，副本、持久化和备份也要纳入生命周期。

## 过期与淘汰不是一回事

过期由键自己的 TTL 决定；淘汰是在内存达到 `maxmemory` 后，Redis 根据策略选择键释放空间。

常见策略族包括：

- `noeviction`：达到上限后拒绝会增加内存的写命令。
- `allkeys-lru` / `allkeys-lfu`：在全部键中近似按最近或频率选择。
- `volatile-lru` / `volatile-lfu`：只在设置了过期的键中选择。
- `allkeys-random` / `volatile-random`：随机选择。
- `volatile-ttl`：倾向淘汰较早到期的键。

策略名称不是容量方案。需要设置内存上限、为持久化与复制缓冲及碎片保留余量，并监控 used memory、RSS、碎片、evicted keys 和拒绝写入。

若同一实例混放“绝不能被淘汰的主数据”和“可以淘汰的缓存”，全局淘汰策略很难同时满足两者，通常应拆分实例或服务等级。

## RDB、AOF、复制和备份

Redis 提供多种持久化选择：

| 机制 | 核心作用 | 主要权衡 |
| --- | --- | --- |
| RDB | 定期生成数据集快照 | 文件紧凑、恢复快；可能丢失快照后的写入 |
| AOF | 记录写命令并在启动时重放 | 可缩小丢失窗口；文件与重写有额外开销 |
| RDB + AOF | 组合两类能力 | 资源和运维复杂度更高 |
| 无持久化 | 纯缓存，可从来源重建 | 重启后数据消失 |

持久化不等于高可用：单机磁盘文件不能在主机故障时自动接管。复制和自动故障转移提升可用性，也不等于备份，因为误写和逻辑删除可能复制到所有节点。

备份必须存放在独立故障域，并定期演练恢复。若 Redis 只是可重建缓存，关闭持久化可能合理；若承担会话或队列状态，就要明确允许丢失多少以及重启后的行为。

## 大键、热键与危险命令

### 大键

大键可能是一个巨大 String，也可能是包含百万成员的集合。风险包括网络传输、单命令耗时、删除释放内存、复制和持久化压力。

不要仅监控键数量，还要采样 value 大小和集合基数。设计上可分页、分桶或拆分，但拆分会增加多键操作与一致性成本。

### 热键

热键是访问量极度集中的键。即使 value 很小，也可能打满单分片 CPU 或网络。可考虑本地缓存、请求合并、只读副本、业务分桶或预计算，但计数等写热点不能在不知道合并语义时随意分散。

### KEYS 与 SCAN

`KEYS pattern` 会遍历并一次返回匹配键，在大键空间可能阻塞 Redis，不应出现在常规生产请求路径。

SCAN 增量遍历，每次返回游标和一批键，对生产更友好，但不是一致性快照：遍历期间变化可能导致元素重复或看不到某些短暂状态。调用方必须循环到游标回到 0，并能去重。

类似地，大 Hash、Set、Sorted Set 分别优先使用 HSCAN、SSCAN、ZSCAN，而不是一次返回全部内容。

## 连接池、超时与故障处理

后端接入 Redis 时至少设置：

- 连接建立超时。
- 单次命令或 socket 读写超时。
- 连接池最大连接数和获取超时。
- 有上限的重试，并区分读、幂等写和非幂等写。
- 熔断、降级和指标。

超时不代表命令一定没有执行。客户端可能在 Redis 执行成功后、响应返回前超时；直接重试非幂等写会重复增加计数或重复入队。因此需要幂等键、唯一请求 ID，或读取当前状态再决定。

不要为每个 HTTP 请求新建 TCP 连接，也不要无限扩大连接池。过多连接增加服务端与客户端资源，掩盖慢命令和下游过载。连接池等待时间必须单独记录，否则“Redis 超时”可能其实是应用一直没取得连接。

## Redis Cluster 与多键操作

Redis Cluster 将键分配到多个 hash slot。涉及多个键的某些操作、事务或脚本要求相关键位于同一 slot，可用 hash tag 控制：

```text
cart:{user:1001}:items
cart:{user:1001}:summary
```

花括号内相同部分让两键映射到同一 slot。它方便原子多键操作，也可能把大量负载集中到一个分片。键设计阶段就要考虑集群路由，不能等数据量增长后才发现核心事务跨 slot。

客户端需要理解 MOVED/ASK 重定向、拓扑刷新和故障转移。把普通单机客户端直接指向集群节点，可能只在低流量测试中偶然工作。

## 安全边界

Redis 应只对可信网络和应用开放，不能把端口直接暴露给浏览器或公网。生产环境至少考虑：

1. 网络防火墙、私有网络和安全组。
2. TLS 加密传输。
3. Redis ACL 的命名用户、命令类别与键前缀最小权限。
4. 禁止应用账号执行管理、配置、清空和调试类危险命令。
5. 凭据通过密钥系统注入并轮换，不写入仓库和日志。
6. 不同环境、敏感等级和故障域使用独立实例或集群。

protected mode 是最后一道默认保护，不是完整生产安全方案。ACL 也不能弥补公开暴露的网络端口。

键和值可能包含用户 ID、会话和个人数据。排障日志不要输出完整令牌、验证码、密码重置链接或敏感缓存内容。

## 与 MySQL/PostgreSQL 的职责划分

| 需求 | 关系数据库 | Redis |
| --- | --- | --- |
| 外键与复杂约束 | 擅长 | 不提供同等关系约束 |
| 多条件临时查询 | SQL 与索引支持 | 通常需预先设计键或结构 |
| 多行事务与隔离 | 成熟 | MULTI/EXEC 语义更简单 |
| 低延迟键访问 | 可以但受存储与查询影响 | 核心优势 |
| 自动过期 | 可实现但非核心模型 | 原生 TTL |
| 排行榜与集合运算 | 可用 SQL | Sorted Set/Set 直接表达 |
| 数据容量成本 | 磁盘为主、缓存辅助 | 内存成本更高 |

最常见的组合是关系数据库作为真相来源，Redis 保存可失效、可重建的读模型。下一课会专门分析这种缓存模式下的穿透、击穿、雪崩、双写一致性和失效策略。

## 安全示例脚本

配套文件使用 redis-cli 的普通标准输入模式执行，并逐条显示响应：

```bash
redis-cli < examples/database/redis/01-fundamentals.redis
```

不要在这里添加 `--pipe`。redis-cli 的 pipe mode 面向按 RESP 格式生成的批量数据，不是把普通逐行命令文件静默变快的开关。

脚本只访问 `learning15:` 前缀，不使用 KEYS、FLUSH、DEL、配置修改或持久化管理命令。每一组写入与 EXPIRE 放在同一个 MULTI/EXEC 中，并设置 10 分钟 TTL；即使重复运行，也只影响这组学习键。文件以 ECHO 输出分段标题，避免依赖 redis-cli 对注释行的解析差异。

执行前应确认连接的是专用本地学习实例。生产 Redis 需要认证与 TLS 时，应通过安全的客户端配置或环境注入凭据，不要把密码写进命令历史或示例文件。

## 常见误区

### “Redis 在内存中，所以所有操作都是 O(1)”

不同命令复杂度不同，响应体大小也会直接影响延迟。对百万成员集合执行全量读取，不会因为数据在内存中就变得安全。

### “MULTI/EXEC 和数据库事务一样”

Redis 保证 EXEC 中命令连续执行，但没有关系数据库式回滚，隔离与错误语义也不同。跨 Redis 与数据库的操作更不是同一个事务。

### “设置 TTL 就不会内存不足”

写入速度可能远高于过期速度，value 也可能过大。必须设置容量上限、淘汰策略并监控实际内存。

### “缓存丢了就回源，不需要高可用”

全量冷缓存会瞬间把压力转移到数据库。能重建不等于可以在任意速度下重建，还需要限流、请求合并、预热和数据库容量评估。

### “一个 Redis 实例可以混放所有业务”

不同业务会争抢内存、CPU、连接和故障预算，全局淘汰与持久化策略也难以兼容。重要业务应按容量、敏感性和可用性要求隔离。

## 本课小结

- Redis 将键映射到内存数据结构，适合可按键定位的低延迟访问，不替代关系约束和复杂 SQL。
- 键前缀、版本、TTL 和租户边界共同构成 Redis 数据模型。
- String 适合对象与计数，Hash 适合字段映射，List 表达顺序，Set 表达唯一集合，Sorted Set 表达按分数排序。
- Stream 支持追加日志和消费组，但业务仍要处理重复消费与无界增长。
- 单条命令通常具有原子边界；MULTI/EXEC 没有关系数据库式回滚，流水线也不提供原子性。
- TTL 决定键生命周期，maxmemory 淘汰处理容量压力，二者不能互相替代。
- RDB、AOF、高可用与备份解决不同问题，必须按允许的数据丢失范围设计。
- 大键、热键、全量遍历和长脚本会拖慢其他请求，应按真实规模评估命令。
- 连接池、超时、幂等重试、ACL、TLS 和故障降级是后端接入 Redis 的必要部分。
