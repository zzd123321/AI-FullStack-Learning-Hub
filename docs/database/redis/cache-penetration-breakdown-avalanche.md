# 缓存穿透、击穿、雪崩与热点治理

缓存的价值不是让一次查询变快，而是吸收本来会进入数据库的大量重复读取。因此，缓存系统真正危险的状态不只是“Redis 慢了”，而是它突然失去吸收流量的能力，让并发成倍落到数据库、连接池和下游接口。

中文工程语境常把这类问题称为缓存穿透、缓存击穿、缓存雪崩和热 key。术语并非严格标准，不同团队对“击穿”和“雪崩”的边界也可能不同。比记名称更重要的是回答四个问题：什么请求在 miss、同时影响多少 key、为什么请求会集中、压力最终落在哪里。

本课沿着请求放大链路推理，再讨论每种方案的正确性、容量代价和失效边界。数据库仍是真相来源；Redis 中的缓存副本允许重建，但重建速度必须受到保护。

## 四类问题先分清

| 问题 | 典型触发条件 | 影响范围 | 数据库侧现象 | Redis 侧现象 |
| --- | --- | --- | --- | --- |
| 缓存穿透 | 大量查询数据库本来就不存在的数据 | 可由许多不同 key 组成 | 重复执行无结果查询，CPU、I/O 或连接被消耗 | miss 很高，写不进普通正值缓存 |
| 缓存击穿 / stampede | 一个或少数热点 key 过期、被淘汰或尚未加载 | 单个热点或小热点集 | 同一 SQL 在短时间并发执行很多次 | 某个 key 的 miss 和回填突然集中 |
| 缓存雪崩 | 大批 key 同时失效、Redis 大面积不可用或持续淘汰 | 大片工作集乃至全部流量 | 多种查询同时上升，连接池迅速耗尽 | 总命中率骤降、超时或 eviction 激增 |
| 热 key | 流量高度倾斜到少数 key，即使一直命中 | 单 key、单分片或单节点 | 不一定升高 | 单核、网卡、分片或客户端连接成为瓶颈 |

它们可以同时发生。例如一个促销商品本来就是热 key；它过期后发生击穿；重建流量挤满数据库连接池，使其他查询超时；应用重试又把局部问题扩散成雪崩。

### 用数量级理解放大

假设详情接口峰值为每秒 10,000 个请求，缓存命中率 95%。忽略其他开销时，数据库只承担约 500 次查询：

```text
数据库查询速率 ≈ 接口请求速率 × (1 - 命中率)
              ≈ 10,000 × 5%
              ≈ 500 次/秒
```

如果 Redis 整体失效，数据库入口可能瞬间从 500 次/秒变为 10,000 次/秒。即使数据库平时只使用一半容量，也不代表它拥有 20 倍的瞬时余量。

再看一个占 2,000 QPS 的热点商品。它正常命中时几乎不查询数据库；TTL 到期的几十毫秒内，若所有请求都独立回源，就可能并发产生几十甚至上百次相同 SQL。这就是击穿的本质：一个缓存 miss 被并发复制。

因此，命中率是平均指标，不能单独描述风险。还要观察 miss 是否集中在同一个 key、单位时间新增多少唯一 miss、数据库能承受多少并发回源。

## 缓存穿透：不存在也需要一个确定结果

典型读路径只缓存查到的对象：

```text
GET cache:user:999999 → miss
SELECT ... FROM users WHERE id = 999999 → 不存在
返回 404，但不写缓存
```

同一个不存在的 ID 再次访问时，完整过程会重演。攻击者如果不断更换随机 ID，还会绕过只针对单 key 的合并，使请求持续穿过 Redis 到达数据库。

穿透不一定是攻击。前端旧链接、爬虫、数据同步延迟、租户参数错误和客户端重试都可能制造大量不存在查询。治理时应先确认业务含义，而不是把所有 404 当恶意流量。

### 第一层：在缓存之前拒绝无效输入

越早拒绝，成本越低。接口层应验证：

- ID 的格式、长度和合法范围。
- 枚举值、分页上限和查询参数组合。
- 租户、权限和资源路径是否匹配。
- 客户端是否超过速率或并发配额。

例如数据库主键是正整数，就不应让 `-1`、超长数字或任意字符串进入 Redis 和 SQL。但“ID 在合法范围”不能证明记录存在，格式校验只能挡住明显无效请求。

还要先完成必要的认证与授权。否则，用一个全局负缓存告诉用户“资源不存在”，可能掩盖不同调用者本应得到的 `403` 或泄漏资源存在性。缓存键和结果必须包含所有影响响应语义的维度。

### 第二层：负缓存

数据库明确返回“不存在”时，可以缓存一个显式哨兵：

```text
key:   cache:user-api:v1:user:999999
value: {"kind":"not_found"}
TTL:   30 秒
```

后续请求命中哨兵后直接返回 404，不再访问数据库。这里必须区分三类结果：

```text
确定存在       → 缓存业务对象
确定不存在     → 可短期缓存 not_found
查询失败/超时  → 不得伪装成不存在
```

如果把数据库超时负缓存成 404，基础设施故障会被转换成虚假的业务事实，而且在 TTL 内持续返回错误答案。

负缓存 TTL 通常短于正常对象 TTL，因为“不存在”可能很快变成“存在”。创建资源成功后，还应主动删除对应负缓存；否则新记录在负缓存过期前不可见。若创建和删负缓存之间无法原子提交，则仍要接受短暂最终一致，或使用可靠事件修复。

负缓存也有容量风险。随机 ID 攻击能制造海量哨兵，挤出真正的热点数据。因此要同时考虑：

- 对来源限流，而不是无限接收唯一 miss。
- 规范化 key，避免同一语义产生多种字符串形式。
- 使用较短 TTL，并监控负缓存占用。
- 为缓存实例设置合适的内存上限与淘汰策略。
- 必要时把负缓存与核心热点隔离到不同前缀、实例或容量池。

### 第三层：布隆过滤器

当合法 ID 空间很大、随机不存在查询很多，而为每个不存在值保存哨兵又太贵时，可以在数据库之前放一个布隆过滤器。

布隆过滤器用多个哈希函数把元素映射到位数组：

```text
请求 id
  │
  ▼
布隆过滤器判断
  ├─ 一定不存在 → 直接返回，不查数据库
  └─ 可能存在   → 查缓存或数据库确认
```

它有两个关键概率性质：

- 返回“不存在”时，理论上可以确定元素未被加入。
- 返回“可能存在”时，仍可能是假阳性，必须继续查真相来源。

“没有假阴性”有前提：过滤器没有损坏，且所有应存在元素都已及时加入。如果数据库已经创建记录而过滤器更新延迟，过滤器就会错误拦截真实记录——这不是算法产生的假阴性，而是业务同步不完整。

安全的维护方案需要明确事件顺序。例如创建记录时，可把“向过滤器添加 ID”纳入可靠事件流程；若业务不允许新数据短暂不可见，就不能在同步尚未完成时用过滤器的否定结果直接返回。删除通常不从普通布隆过滤器移除位，因为多个元素共享位；已删除元素偶尔通过过滤器只会多一次数据库查询，不会返回错误对象。

Redis 的 `BF.RESERVE` 可以按预计容量和目标假阳性率创建过滤器，`BF.ADD`/`BF.MADD` 添加元素，`BF.EXISTS` 查询。应先确认部署版本和发行形态支持这些概率数据类型，再在测试环境按预计容量验证内存与误判率。容量估计过小可能导致扩展子过滤器增多，查询成本和实际误判行为也会变化。

布隆过滤器不是权限系统，也不是数据库副本；它只负责廉价排除大量“不可能存在”的值。对规模不大、创建频繁或同步链路不可靠的业务，负缓存往往更简单。

### 穿透治理的决策顺序

```text
输入明显非法？
  ├─ 是 → 参数校验直接拒绝
  └─ 否
      │
      ▼
调用频率异常？
  ├─ 是 → 按调用者/租户/接口限流
  └─ 否
      │
      ▼
同一不存在值经常重复？
  ├─ 是 → 短 TTL 负缓存
  └─ 否
      │
      ▼
不存在值空间巨大且可维护存在集合？
  ├─ 是 → 评估布隆过滤器
  └─ 否 → 数据库保护、查询优化和持续观测
```

这些手段可以叠加，但每加一层，都要定义它返回错误时的影响和恢复办法。

## 缓存击穿：让同一个 miss 只回源一次

击穿的对象通常真实存在，因此负缓存不能解决。问题是多个请求几乎同时看到 miss，并各自开始加载：

```text
请求 A ─ GET miss ─ SELECT ─ SET
请求 B ─ GET miss ─ SELECT ─ SET
请求 C ─ GET miss ─ SELECT ─ SET
请求 D ─ GET miss ─ SELECT ─ SET
```

正确方向是把同 key 的并发加载合并为一次，其他请求共享结果或受到明确限制：

```text
请求 A ─ miss ─ 成为加载者 ─ SELECT ─ SET ─ 返回
请求 B ─ miss ─ 等待同一结果 ──────────── 返回
请求 C ─ miss ─ 等待同一结果 ──────────── 返回
```

### 进程内 Singleflight

应用可以维护 `Map<cacheKey, Promise>`。第一个 miss 创建加载 Promise；同一进程内后续请求复用这个 Promise；完成或失败后从 Map 删除。

它的优点是快、不依赖额外 Redis 命令，适合先做基础保护。边界也很明确：部署了 20 个应用实例，最坏仍可能有 20 次数据库加载；进程重启后协调状态消失。

实现时应注意：

- 只按同一个规范化缓存 key 合并，不要把所有 miss 串行化。
- 等待者必须有请求级超时，不能无限挂起。
- 加载失败可以共享给当前等待者，但不能把失败 Promise 永久留在 Map。
- 为等待者数量设上限；热点超过容量时应拒绝或返回可接受的旧值。
- 数据库 loader 自身仍需连接池和并发上限。

### 跨实例互斥重建

多个应用实例需要更强协调时，可用一个短租约选出重建者。Redis 原子命令可以写成：

```redis
SET lock:cache:user:1001 random-owner-token NX PX 3000
```

`NX` 表示仅在 key 不存在时写入，`PX` 为租约设置毫秒级过期，避免持有者崩溃后锁永久残留。获得租约后要再次读取缓存，因为在竞争期间另一个实例可能已经完成回填，这叫 double-check。

完整流程是：

1. 缓存 miss。
2. 尝试获得该缓存 key 对应的重建租约。
3. 获得者再次检查缓存；仍 miss 才查询数据库并回填。
4. 释放时原子比较 owner token，只允许持有者删除。
5. 未获得者短暂退避后重读、返回 stale，或按过载策略失败。

不能直接 `DEL lockKey` 释放。请求执行超过租约后，旧租约可能已过期并被新请求获得；旧持有者此时 DEL 会误删新持有者的租约。比较 token 与删除必须在服务端形成一个原子操作，通常使用 Lua 脚本或等价的原子能力。

租约也不能凭空创造数据库容量。若加载本身超过租约、Redis 不可用、持有者卡死，等待策略仍要处理超时。分布式锁的完整安全模型会在下一课展开；此处只把它视为“缓存重建协调”，不能把它当成业务数据库事务锁。

### 逻辑过期：用可控陈旧换稳定

对读多写少且允许短时旧值的热点，可把 value 设计为：

```json
{
  "data": { "id": 1001, "name": "Alice" },
  "softExpireAt": "2026-07-15T12:00:00Z",
  "version": 7
}
```

Redis key 保留较长的硬 TTL；业务时间 `softExpireAt` 到达后，不把它立即当作 miss：

1. 普通请求仍可返回 stale value。
2. 只有一个请求或后台任务获得刷新资格。
3. 刷新成功后替换 value 和逻辑过期时间。
4. 刷新失败时保留旧值到硬 TTL 上限，并告警。

这样，热点过期不会把所有用户都变成数据库等待者。代价是明确的陈旧窗口，所以权限撤销、余额、库存扣减结果等高风险数据不能未经业务确认就返回 stale。

必须同时保留硬边界。若永不物理过期，失去访问的 key 会永久占内存；若后台刷新持续失败，用户也可能无限看到旧数据。硬 TTL、最大 stale 时间和失败降级必须分别定义。

### 提前刷新与预加载

对可预测热点，可以在软过期前由后台任务刷新，或在活动开始前预热。提前刷新把数据库工作移出用户请求路径，但仍需：

- 防止多个任务同时刷新同一 key。
- 控制刷新并发，避免预热本身压垮数据库。
- 校验预热数据版本，避免旧批次覆盖新值。
- 活动结束后恢复正常 TTL 和容量策略。

随机提前刷新也能避免所有节点在同一阈值同时行动。无论主动刷新多完善，都要保留 miss 时的受限回源，因为预热可能漏项或失败。

## 缓存雪崩：保护整个依赖链

雪崩不是“很多 miss”这一个现象，而是大量 miss、超时或重试在同一时间相关联，超过下游系统的恢复能力。常见触发源包括：

1. 批量写入的 key 使用完全相同的 TTL，同一秒集中到期。
2. Redis 主节点故障、网络分区、故障转移或客户端拓扑未及时更新。
3. `maxmemory` 附近持续淘汰，刚回填的工作集又被挤出。
4. 发布时变更 key 版本，全部请求进入新而空的命名空间。
5. 误删一批缓存，或批量数据同时失效。
6. Redis 变慢后应用无上限重试，形成重试风暴。

### TTL 抖动只解决“同时到期”

给基础 TTL 增加随机值，可以把到期时间摊开：

```js
function ttlWithJitter(baseSeconds, spreadSeconds) {
  return baseSeconds + Math.floor(Math.random() * (spreadSeconds + 1));
}

// 基础 5 分钟，再随机增加 0～60 秒
const ttl = ttlWithJitter(300, 60);
```

这里的随机量应按业务陈旧上限和批量规模选择，而不是复制一个固定百分比。若响应最多允许陈旧 5 分钟，就不能在 5 分钟基础上任意再增加 5 分钟；可改为在允许窗口内使用对称或仅减小的抖动。

TTL 抖动不能解决：Redis 整体不可用、内存持续淘汰、统一切换 key 版本、批量主动删除。它也不适用于必须在准确时刻失效的安全令牌或业务状态。这些数据的过期是业务语义，不应为了缓存均匀随意随机化。

### 分阶段预热和恢复

新版本命名空间或新 Redis 集群上线时，应限制预热速率并分批加载热点：

```text
第 1 阶段：最高频、回源最贵的核心 key
第 2 阶段：近期活跃工作集
第 3 阶段：由真实请求按受限 Cache-Aside 自然填充
```

故障恢复后也不能立刻放开全部流量。Redis 刚恢复时缓存仍然很冷，数据库也可能正处理积压。应采用逐步放量、回源并发上限和实时负载反馈，让系统沿可控斜率恢复。

### 过载保护比“尽量成功”更重要

缓存失效时，应用需要在数据库之前设置第二道容量闸门：

- **短超时**：Redis 和数据库调用都使用基于延迟目标的超时。
- **有限重试**：只对可安全重试的操作，使用指数退避和随机抖动；总重试预算受限。
- **回源并发上限**：限制每实例、每接口或每租户同时访问数据库的数量。
- **断路器**：数据库或 Redis 持续失败时快速失败，避免每个请求都等待超时。
- **负载削减**：优先拒绝低优先级请求、昂贵筛选和非核心字段。
- **stale-if-error**：业务允许时，在有限窗口返回旧缓存，而不是进入已过载数据库。
- **请求截止时间传播**：上游已超时后，下游不再继续执行无用查询。

限流会让一部分请求明确失败，但能让系统保留处理核心流量和恢复的能力。无保护地全部回源，看似更努力，实际可能让所有请求一起超时。

### 淘汰雪崩与到期雪崩不同

TTL 到期是按设计删除；eviction 是 Redis 达到 `maxmemory` 后按策略为新写入腾空间。若工作集大于可用内存，会出现循环：

```text
key 被淘汰 → 请求回源并回填 → 回填挤出其他 key
          → 其他请求 miss → 更多回填和淘汰
```

此时增加 TTL 没有帮助，因为 key 还没到期就被淘汰。要查看 `INFO stats` 中的 `evicted_keys`、内存水位、value 大小分布和工作集变化，并重新评估：

- 是否缓存了低价值或过大的对象。
- 实例总内存与 `maxmemory` 之间，是否为复制/AOF 缓冲、内存碎片和其他非数据开销预留了余量。
- 淘汰策略是否符合访问模式。
- 是否需要拆分实例、扩容或缩短低价值数据 TTL。

`noeviction` 也不是“没有故障”：达到上限后，创建新缓存 value 的写命令会报错，读仍可能成功。应用必须把回填失败作为可观察状态处理，不能假设 SET 总会成功。

## 热 key：命中也可能过载

热 key 的问题是流量倾斜。一个 key 若承担全站 30% 读取，即使 100% 命中，也可能集中消耗一个 Redis 分片的单核执行时间、网络带宽或客户端连接。若 value 很大，网络和序列化成本会更明显。

### 先证明“热”在哪里

至少从三层观察：

| 层级 | 关注指标 |
| --- | --- |
| 应用 | 按逻辑资源聚合的 QPS、并发、value 字节数、命中和回源 |
| 客户端 | 连接池等待、超时、重试、每节点流量 |
| Redis | ops/sec、CPU、网络、命令延迟、keyspace hits/misses、分片倾斜 |

不要把敏感 ID、完整 key 或高基数标签无控制地写进监控系统。可用哈希、采样和受限 Top-N 指标识别热点。某些诊断命令或扫描工具会增加生产负载，应先核对版本、淘汰策略和托管服务能力，在受控窗口使用。

### 热点读取的治理选择

1. **进程内 L1 缓存**：把极热且允许短时陈旧的数据复制到每个应用实例。它减少 Redis QPS，但带来多副本失效问题；必须设置很短 TTL、版本或失效通知。
2. **读副本分担**：客户端把允许陈旧的读取分散到副本。复制存在延迟，刚写后的读取和强一致场景不能盲目走副本。
3. **压缩或拆小 value**：接口只取需要字段，避免每次传输巨型 JSON。拆分后要权衡多命令、原子性和一致性。
4. **请求合并与批处理**：同实例内相同读取共享结果；多个不同 key 可在语义允许时批量获取，减少往返。
5. **业务结果预计算**：把昂贵聚合提前计算，避免每次请求临时展开大集合。

### key 分片不是通用答案

可交换、可合并的写热点，例如近似计数，可以把一个计数器拆成多个桶：

```text
counter:video:42:0
counter:video:42:1
...
counter:video:42:15
```

写入随机或按调用者散列到桶，读取时求和。代价是读取放大、结果可能非瞬时一致、重置和去重更复杂。

普通用户对象不能简单复制成 16 个 key 再随机读，因为写入时必须可靠更新或失效全部副本。只有操作满足可交换、可结合，或业务能承担多副本一致性成本时，key 分片才合适。

在 Redis Cluster 中，给一个 key 改成多个后缀可能把负载分散到不同 hash slot；若使用相同 hash tag（如 `{video:42}`），这些 key 仍会落在同一 slot。是否使用 hash tag 要根据多键原子需求和分散目标决定。

## 组合成分层防线

缓存治理不是在负缓存、锁和限流中三选一。它们分别作用于不同位置：

```text
客户端请求
   │
   ├─ 参数校验、认证、按来源限流
   │
   ▼
应用 L1 / Redis
   ├─ 正值缓存、负缓存、布隆过滤器
   ├─ TTL 抖动、逻辑过期、预热
   │
   ▼ miss
同 key 请求合并 / 重建租约
   │
   ▼
全局回源并发上限、截止时间、断路器
   │
   ▼
关系数据库：连接池、查询超时、索引和容量保护
```

一套可落地的默认策略可以是：

1. 参数和权限校验后构造规范化 cache key。
2. 正常对象使用满足陈旧契约的 TTL，并为批量写入增加抖动。
3. 明确的 not found 使用更短负缓存，查询错误不负缓存。
4. 同实例按 key singleflight；跨实例极热点再评估重建租约。
5. 数据库回源有独立并发上限，不能按应用线程数无限放大。
6. Redis 故障时按业务优先级选择 stale、受限回源或快速失败。
7. 监控按 miss 原因和 key 热度拆分，而不只看平均命中率。

## 用失败矩阵检查方案

| 事件 | 可能结果 | 必须定义的处理 |
| --- | --- | --- |
| 数据库明确 not found | 穿透重复发生 | 短负缓存；创建后失效 |
| 数据库查询超时 | 结果未知 | 不负缓存；有限重试或失败 |
| 热点 key 到期 | 同键并发回源 | singleflight、租约、stale 或提前刷新 |
| 重建者崩溃 | 等待者无法得到新值 | 有界租约、等待超时、重新竞争 |
| 重建超过租约 | 可能出现多个重建者 | 合理租期、续租边界、版本保护与 DB 限流 |
| Redis GET 超时 | 不知道 key 是否存在 | 受限回源或降级，不能无限重试 |
| Redis SET 回填失败 | 本次响应可成功，后续继续 miss | 记录失败并限制回源；不回滚只读结果 |
| 大批 key 同时到期 | 数据库突发过载 | TTL 分散、分阶段预热、回源闸门 |
| Redis 达到内存上限 | 淘汰循环或写入报错 | 容量、value、TTL、淘汰策略联合治理 |
| Redis 故障恢复 | 冷缓存再次冲击数据库 | 渐进放量、热点优先预热 |

## 可观测性：命中率只是起点

建议至少记录以下指标，并按接口、租户或资源类型做受控聚合：

- cache hit、positive miss、negative hit。
- miss 原因：到期、主动失效、淘汰、Redis 错误、解析失败。
- 数据库 fallback QPS、并发数、耗时和拒绝数。
- singleflight 实际加载次数、被合并请求数、等待时间和超时数。
- 重建租约获取成功率、持有时间和过期次数。
- 返回 stale 的数量、陈旧年龄与刷新失败。
- TTL 分布、同一时间窗口预计到期 key 数。
- Redis `keyspace_hits`、`keyspace_misses`、`evicted_keys`、CPU、内存、网络和命令延迟。
- 数据库连接池使用率、排队时间、查询超时和慢查询。
- 负缓存条目量、命中率和唯一不存在 ID 增长速度。

`hit / (hit + miss)` 下降不一定代表故障：版本发布后的正常冷启动也会下降；命中率不变也不代表安全，一个超大热 key 可能仍让单节点过载。告警应结合回源量、延迟、淘汰和下游饱和度。

高基数 key 不宜直接作为指标标签，否则监控系统本身可能被打爆。常见做法是固定维度指标加采样日志，并单独维护受限的热点 Top-N。

## 怎样验证，而不是等线上证明

缓存故障测试需要制造可控并发和时间：

1. 让 100 个请求在同一 key miss 后同时暂停，断言数据库 loader 只执行一次。
2. 让 loader 超时或失败，断言等待者会退出，in-flight 状态会清理。
3. 连续读取同一个不存在 ID，断言数据库只在负缓存首次或过期后访问。
4. 模拟数据库超时，断言不会写入 not_found。
5. 生成一批相同基础 TTL，断言加入抖动后到期分布被摊开。
6. 让 Redis GET/SET 超时，断言回源并发不超过预算。
7. 模拟 eviction 持续上升，观察是否形成回填—淘汰循环。
8. 清空专用测试缓存后渐进放量，测量数据库恢复余量。

测试环境应使用专用实例和学习前缀，不对共享或生产 Redis 做全量清除。故障演练要有停止条件，例如数据库连接池超过 70% 或 P99 超过目标即停止加压。

## 配套状态模型

`examples/database/redis/03-cache-failure-patterns.mjs` 不依赖 Redis 和第三方包，用可控 Promise 与假时钟验证三个关键机制：

- 100 个并发 miss 在朴素实现中会启动 100 次加载，而 singleflight 只启动一次。
- not found 可以短期负缓存；数据库错误不会被伪装成 not found。
- 相同 TTL 会让 1,000 个 key 同秒到期，加入确定性随机抖动后会分散到多个时间桶。

运行：

```bash
node examples/database/redis/03-cache-failure-patterns.mjs
```

模型只验证应用状态转换，不是生产 Redis 客户端。真实系统还必须验证多实例、网络超时、连接池、部署版本和故障转移。

## 常见误区

### “所有 miss 都加分布式锁”

冷 key 很多时，每个 miss 再增加锁命令会放大 Redis 负担。先做数据库回源上限和进程内合并，只为经过测量的跨实例热点增加租约协调。

### “布隆过滤器说存在，就可以直接返回成功”

布隆过滤器有假阳性，“可能存在”只能允许继续查询，不能替代数据库事实。过滤器同步不完整还会错误拦截刚创建的数据。

### “给 TTL 加随机数就不会雪崩”

抖动只分散按 TTL 同时到期，不能解决 Redis 故障、统一删缓存、命名空间切换和内存淘汰。完整方案必须有过载保护和恢复节奏。

### “Redis 挂了就全部查数据库”

缓存存在通常正因为数据库无法承受全部流量。回源必须受并发预算控制，必要时返回 stale、降级结果或明确的过载错误。

### “热点 key 拆成多个副本就行”

读取虽然分散，写入和失效却变成多副本一致性问题。只有数据可合并或已有可靠版本与广播机制时才拆分。

### “互斥重建能保证缓存永远新鲜”

互斥只减少重复加载，不解决数据库与缓存的提交原子性、旧读回填和陈旧窗口。它必须与上一课的一致性策略组合。

## 本课小结

- 穿透是不存在数据持续越过缓存；击穿是热点 miss 被并发复制；雪崩是大片缓存能力同时丧失；热 key 是流量倾斜本身。
- 参数校验、来源限流、短负缓存和布隆过滤器分别拦截不同类型的无效查询。
- 只有数据库明确返回 not found 才能负缓存；超时和错误属于未知结果。
- singleflight 合并单实例同 key 加载；跨实例租约需要过期、owner token、原子释放和有界等待。
- 逻辑过期允许有限 stale，换取热点刷新稳定性；高风险数据必须先定义能否陈旧。
- TTL 抖动只分散同时到期，雪崩治理还需要回源并发上限、断路器、分阶段预热和渐进恢复。
- eviction 造成的 miss 与 TTL 到期机制不同，应从工作集、内存、value 和淘汰策略处理。
- 热 key 即使命中也会压垮单分片；L1、副本读、拆小 value 和可合并分片各有一致性代价。
- 平均命中率不足以判断安全，必须观察 miss 原因、同键并发、回源量、淘汰和数据库饱和度。
