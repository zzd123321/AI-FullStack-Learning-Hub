# 锁、MVCC 与并发等待

::: tip 第一次学习只抓住四件事
- **必须理解**：锁协调互相冲突的操作，MVCC 让许多读写通过版本并存减少阻塞；两者不是互相替代。
- **必须会判断**：锁等待是暂时排队，死锁是循环依赖；应用必须限制事务时长并处理死锁重试。
- **必须完成**：观察一次锁等待，确认阻塞事务、等待语句和事务持有时间。
- **可以后看**：版本链、undo、vacuum、间隙锁等产品内部细节。
:::

上一课说明了隔离级别决定事务可以观察到什么现象。本课继续回答三个后端开发中更直接的问题：普通查询为什么通常不阻塞写入，两个写请求为什么仍会互相等待，以及接口偶发超时如何定位到真正的阻塞事务。

数据库并不是在“只用锁”和“只用 MVCC”之间二选一。MySQL InnoDB 与 PostgreSQL 都同时使用多版本并发控制和锁：MVCC 主要解决读写并发与可见性，锁负责保护正在变化的资源、协调互斥写入并实现显式并发协议。

## 先建立一张全景图

一次数据库访问可以从三个维度理解：

| 维度 | 要回答的问题 | 常见机制 |
| --- | --- | --- |
| 可见性 | 当前语句能看到哪个已提交版本 | MVCC 快照、隔离级别 |
| 互斥 | 谁可以修改同一资源 | 行锁、表锁、索引记录锁 |
| 等待与失败 | 冲突时是等、跳过还是报错 | 超时、NOWAIT、SKIP LOCKED、死锁检测 |

因此，“SELECT 不加锁”并不严谨。普通 SELECT 通常不取得会阻止其他事务修改该行的锁，但执行查询仍可能短暂使用元数据锁、内部锁或轻量同步机制。学习并发控制时，更准确的区分是“快照读”与“锁定读”。

## MVCC：让读取旧版本而不是等待新版本

MVCC 是 Multi-Version Concurrency Control，即多版本并发控制。它的核心思想是：数据更新时不立即让旧版本从所有并发事务的视野中消失，而是保留足够的信息，让不同快照判断哪个版本对自己可见。

假设余额最初是 100：

| 时刻 | 事务 A | 事务 B |
| --- | --- | --- |
| T1 | 开始并建立快照 | |
| T2 | | 将余额改为 80 并提交 |
| T3 | 依据隔离级别仍可能读到 100 | |

事务 A 读到 100，不代表数据库把 B 的提交“撤销”了，而是 A 的快照仍选择旧版本。什么时候建立快照、同一事务是否复用快照，由数据库和隔离级别共同决定。

MVCC 带来一个重要性质：普通读取通常不需要等待正在修改数据的事务；写入也不需要等待早先快照完成读取。这提升了并发能力，但不意味着没有代价。数据库必须保存旧版本信息，并在任何活跃快照都不再需要后清理它们。

## MySQL InnoDB 如何保存版本

InnoDB 的聚簇索引记录包含事务相关的隐藏信息。更新记录时，旧值所需的信息写入 undo log，并通过版本链让一致性读能够找到对当前 Read View 可见的版本。

可以把它抽象成：

```text
当前记录版本 <- undo 信息 <- 更旧版本 <- 更旧版本
```

这不是说每个旧版本都以完整行副本永久存放。实现会记录足够的撤销信息来重建旧版本。后台 purge 在确认旧版本不再被活跃事务需要后，才能清理相关历史。

需要区分两类日志：

- undo log 支持事务回滚和 MVCC 旧版本读取。
- redo log 记录页修改所需的重做信息，主要服务崩溃恢复和持久性。

把 redo 解释为“旧版本链”或把 undo 解释为“数据库备份”都不准确。

## PostgreSQL 如何保存版本

PostgreSQL 更新一行时，通常在表的堆存储中产生新的行版本（tuple），旧版本保留到 VACUUM 可以回收为止。行版本带有事务可见性相关信息，快照依据创建和删除该版本的事务状态判断是否可见。

这与 InnoDB 的“当前记录加 undo 版本链”实现不同，但目标一致：让读事务依据快照访问适合自己的版本。PostgreSQL 的 WAL 主要用于崩溃恢复和复制，不等同于 MVCC 旧行版本本身。

普通 VACUUM 会回收死元组占用的空间供同一张表以后复用、维护可见性信息并更新部分统计信息。autovacuum 不是可有可无的清理工具，它也是防止事务 ID 回卷问题的重要机制。

## 快照读、当前读与锁定读

普通查询通常是快照读：

```sql
SELECT id, stock
FROM inventory
WHERE product_id = 42;
```

它根据快照选择可见版本。若另一个事务刚把库存从 10 改为 9 但尚未提交，查询不会把未提交的 9 当成已提交数据；具体读到哪个已提交版本取决于隔离级别。

准备根据读取结果立即更新时，应用可能需要锁定读：

```sql
SELECT id, stock
FROM inventory
WHERE product_id = 42
FOR UPDATE;
```

锁定读要找到当前可锁定版本并持有相应锁，不能仅返回任意旧快照版本。其他事务若也要取得冲突锁，通常等待到当前事务 COMMIT 或 ROLLBACK。

不要把所有“先查后改”都机械改为 `FOR UPDATE`。库存扣减更适合把条件和更新合为一个原子语句：

```sql
UPDATE inventory
SET stock = stock - 1
WHERE product_id = 42
  AND stock >= 1;
```

应用检查影响行数即可判断扣减是否成功，事务持锁时间也往往更短。

## 共享锁与排他锁

可先用两种基础模式建立直觉：

- 共享锁（S）：允许其他事务也读取并取得兼容的共享锁，但阻止冲突写入。
- 排他锁（X）：用于修改资源，与其他事务在同一资源上的共享或排他锁冲突。

简化兼容矩阵如下：

| 已持有 / 新请求 | S | X |
| --- | --- | --- |
| S | 兼容 | 冲突 |
| X | 冲突 | 冲突 |

真实数据库还有多种表级、行级和内部锁模式，不能只靠这张表解释所有现象。不过它足以说明：两个普通读可以并发，两个修改同一行的事务必须协调顺序。

锁通常属于事务，而不是某一行应用代码。连接执行 COMMIT、ROLLBACK 或因错误回滚后，事务锁才释放。将连接借给请求后忘记结束事务，会把锁的生命周期从几毫秒扩大到不可预测的时间。

## 意向锁不是“准备稍后再锁”

InnoDB 在取得行级共享或排他锁前，会在表级登记意向共享锁（IS）或意向排他锁（IX）。它们的作用是快速表明“这张表的某些行已被某事务以某种方式锁定”，帮助表级锁判断兼容性。

例如某事务要修改一行，通常持有表上的 IX 和目标索引记录上的 X。另一个事务修改不同记录时，两个 IX 可以兼容；但请求冲突的全表锁时，数据库无需扫描每一条行锁。

意向锁通常不是两个不同记录更新互相阻塞的原因。诊断时应继续查看具体记录、范围或表锁，而不是看到 IX 就断言表已被完全排他锁住。

## InnoDB 锁的是索引记录

InnoDB 的行级锁建立在索引记录上。SQL 看起来是在锁“表中的行”，实际锁范围与查询采用的索引访问路径密切相关。

### Record lock

记录锁锁定某个索引记录。若使用唯一索引的完整等值条件定位唯一记录，InnoDB 通常只需锁定该索引记录，而不锁相邻间隙。

```sql
SELECT id, email
FROM users
WHERE id = 1001
FOR UPDATE;
```

### Gap lock

间隙锁锁定索引记录之间、第一条记录之前或最后一条记录之后的范围。它主要限制其他事务向该间隙插入新索引记录，而不是锁住一个已经存在的具体值。

### Next-key lock

next-key lock 可以理解为“索引记录锁 + 该记录之前的间隙锁”。InnoDB 在 REPEATABLE READ 下进行范围搜索或扫描时使用它来避免锁定读出现幻影插入。

```sql
SELECT id
FROM orders
WHERE created_at >= '2026-07-01'
  AND created_at <  '2026-08-01'
FOR UPDATE;
```

若 `created_at` 有索引，锁范围与扫描的索引区间有关；若缺少合适索引，数据库可能扫描并锁住大量索引记录，导致看似无关的请求互相等待。

### Insert intention lock

多个事务准备在同一索引间隙的不同位置插入时，会使用插入意向锁协调。只要位置不冲突，这种机制允许它们并发，而不是让整个间隙内所有插入完全串行。

因此，解释 InnoDB 阻塞必须同时回答：

1. 实际隔离级别是什么？
2. SQL 使用了哪个索引？
3. 访问类型是唯一点查还是范围扫描？
4. 锁定的是记录、间隙还是 next-key 范围？

## PostgreSQL 的行级锁模式

PostgreSQL 提供四种常见锁定读强度：

| 语法 | 典型用途 | 相对强度 |
| --- | --- | --- |
| `FOR KEY SHARE` | 保护行的键不被删除或改成影响外键的值 | 最弱 |
| `FOR SHARE` | 共享保护整行，阻止冲突更新 | 较弱 |
| `FOR NO KEY UPDATE` | 修改非键字段时的排他协调 | 较强 |
| `FOR UPDATE` | 可能修改、删除行或改变键 | 最强 |

这种细分能让外键检查与不改变键的更新获得更多并发。`UPDATE` 取得的具体行锁强度取决于是否修改可能参与外键引用的键列。

PostgreSQL 还有多种表级锁模式。普通 SELECT 会取得 `ACCESS SHARE`，普通 INSERT/UPDATE/DELETE 通常取得 `ROW EXCLUSIVE`；它们名称里带 ROW 不代表这是行锁。多数表级锁不会阻止普通查询，`ACCESS EXCLUSIVE` 才会与所有表级模式冲突。

DDL、VACUUM 和索引维护涉及的锁模式不同。生产变更前必须查对应命令的官方说明，不能凭命令名称猜测是否阻塞。

## MVCC 没有消除写写冲突

若 A、B 同时更新同一行，数据库不能简单地让两者各写一个版本并都算成功，否则后提交者可能无声覆盖先提交者，约束和业务不变量也可能被破坏。

常见过程是：

1. A 更新行并持有写锁。
2. B 尝试更新同一行，等待 A。
3. A 提交或回滚。
4. B 继续执行，并依据数据库与隔离级别重新判断目标版本或报告冲突。

所以“PostgreSQL 读不阻塞写、写不阻塞读”描述的是 MVCC 的主要优势，不等于“写不阻塞写”。

## 等待、立即失败与跳过

锁不可得时，应用有三种基本策略。

### 等待

默认行为通常是等待到锁释放或达到超时。它适合短暂冲突，但必须设置合理的事务、语句和请求超时，使等待不会无限吞噬连接池。

超时之间要留出层次：数据库锁等待或语句超时应早于 HTTP 网关的总超时，应用才有机会回滚、记录上下文并返回明确错误。具体数值取决于接口延迟目标，不能复制一组全局固定值。

### NOWAIT

```sql
SELECT id, status
FROM jobs
WHERE id = 9001
FOR UPDATE NOWAIT;
```

锁不可得时立即报错。它适合调用方宁愿快速失败、转入重试或提示“正在处理”，不愿占用数据库连接等待的场景。

### SKIP LOCKED

```sql
SELECT id, payload
FROM jobs
WHERE status = 'pending'
ORDER BY id
FOR UPDATE SKIP LOCKED
LIMIT 1;
```

多个消费者可以跳过其他消费者已锁住的任务，各自领取不同记录。这是队列式访问的工具，不是普通列表接口的性能开关：结果会故意忽略被锁定行，因此不是完整一致的业务集合。

一个可靠的数据库任务队列还需要：

- 在同一短事务中领取任务并改变状态或设置租约。
- 处理消费者崩溃后的租约过期与重新领取。
- 以任务 ID 或业务幂等键避免重复副作用。
- 记录尝试次数、最后错误并设置失败上限。
- 在提交后执行耗时外部调用，避免长时间持锁。

`SKIP LOCKED` 解决的是并发领取，不提供消息恰好执行一次的保证。

## 死锁：形成环形等待

锁等待本身不一定是故障；死锁是等待关系形成闭环，任何一个事务都无法自行继续。

| 时刻 | 事务 A | 事务 B |
| --- | --- | --- |
| T1 | 锁住账户 101 | 锁住账户 102 |
| T2 | 等待账户 102 | 等待账户 101 |

数据库会检测死锁并选择一个事务作为受害者回滚，让另一个继续。受害者不一定是“写错代码”的那个请求，也不能假定每次都回滚较晚开始的事务。

减少死锁的主要方法：

- 所有代码路径以相同顺序锁资源，例如始终先锁较小账户 ID。
- 缩短事务，不在事务内调用远程接口、等待用户输入或执行大计算。
- 建立与 WHERE、JOIN、ORDER BY 匹配的索引，减少扫描与锁范围。
- 将大批量写入拆成可控批次，但同时保证业务原子边界。
- 避免不同接口以相反顺序更新同一组表。

即使设计良好，死锁仍是正常的并发错误路径。应用应回滚并重试整个事务，而不是只重发最后一条 SQL，因为事务之前的读取结论和写入都已失效。

重试要满足以下边界：

1. 只匹配数据库明确的死锁或可重试序列化错误。
2. 每次重新开启事务并重新执行全部业务判断。
3. 使用有限次数和指数退避加随机抖动。
4. 外部副作用需要幂等键，避免事务提交状态不明时重复扣款或发信。
5. 超过上限后记录指标并返回可观测的错误。

## 乐观锁与悲观锁

悲观锁假设冲突值得提前互斥，典型做法是 `SELECT ... FOR UPDATE`。它适合冲突概率高、必须基于当前状态完成多步更新的短事务。

乐观锁允许并发读取，在写入时验证版本：

```sql
UPDATE documents
SET content = ?,
    version = version + 1
WHERE id = ?
  AND version = ?;
```

影响行数为 0 表示记录不存在或版本已变化，应用重新读取、合并或提示冲突。它适合读取多、冲突较少、用户编辑时间较长的场景。

二者都不是万能方案。乐观锁在高冲突下会大量失败重试；悲观锁在事务过长时会制造等待链。条件更新、唯一约束和数据库原子表达式往往比手写“先读再锁”更简洁。

## 长事务为什么危险

长事务不只意味着“锁持有更久”。即使它只读，也可能长期保留旧快照，阻止旧版本被安全回收。

对 InnoDB 而言，长事务可能让 undo 历史积累、延迟 purge、增加存储和读取旧版本的成本。对 PostgreSQL 而言，旧快照可能让 VACUUM 暂时无法移除仍可能可见的死元组，造成表和索引膨胀风险；长期缺乏必要的 vacuum 还会带来事务 ID 回卷风险。

常见来源包括：

- Web 请求开启事务后调用慢速第三方接口。
- 连接池连接以 `idle in transaction` 状态等待。
- 管理脚本开启事务后忘记提交或回滚。
- 批处理一次更新过多记录。
- 游标或导出任务长期保留同一快照。

后端代码应遵循“先准备、后开事务、尽快提交”：参数校验、无需一致性的远程读取和大计算尽量放在事务外；事务内只执行必须原子完成的数据库操作。

## 从接口超时定位锁等待

看到数据库接口变慢时，不要立即终止任意会话。安全诊断顺序是：

1. 找到正在等待的会话和等待开始时间。
2. 找到直接阻塞者，而不只看最慢 SQL。
3. 沿等待关系继续向上找到根阻塞事务。
4. 查看根事务开始时间、当前或最近 SQL、应用名、用户与客户端地址。
5. 将数据库连接标识关联到请求 ID、任务 ID和服务实例。
6. 判断它是在正常处理、空闲未提交，还是已经失去调用方。
7. 优先让应用正常提交或回滚；终止会话属于有影响操作，需要明确授权和回滚风险评估。

等待链中最末端的 SQL 往往只是受害者。只取消被阻塞查询会暂时减轻连接压力，却不会消除根阻塞者。

## MySQL 的锁诊断入口

MySQL 8.4 中常用信息包括：

- `performance_schema.data_locks`：当前持有或等待的数据锁。
- `performance_schema.data_lock_waits`：请求锁与阻塞锁之间的关系。
- `information_schema.innodb_trx`：当前 InnoDB 事务及状态。
- `SHOW ENGINE INNODB STATUS`：包含最近一次检测到的 InnoDB 死锁信息。

这些视图是快速变化的瞬时快照，分开查询时状态可能已经改变。读取全局事务和语句还可能需要额外权限，并可能暴露其他业务会话信息，生产环境应通过受控诊断账号和审计流程访问。

诊断时重点关注事务何时开始、等待哪把锁、阻塞事务是否仍在执行，以及 SQL 的访问索引。仅看到 `LOCK_MODE = X` 还不足以解释范围，必须结合索引名、锁数据和执行计划。

## PostgreSQL 的锁诊断入口

PostgreSQL 常用信息包括：

- `pg_stat_activity`：后端状态、事务开始时间、等待事件和当前查询。
- `pg_locks`：可授予和正在等待的锁。
- `pg_blocking_pids(pid)`：返回阻塞指定后端进程的 PID。
- 数据库日志：按配置记录锁等待和死锁上下文。

`pg_locks` 的 `granted = false` 表示该锁请求仍在等待。它应与 `pg_stat_activity` 按 PID 关联，而不是单独阅读。`wait_event_type = 'Lock'` 能提示锁等待，但不同版本的字段和锁类型需要查对应版本文档。

`pg_blocking_pids` 比手工推导所有锁兼容关系更直接，不过并行查询、预备事务和软阻塞关系会让结果需要结合现场解释。

## 只读检查脚本

配套脚本只检查当前连接，不创建表、不修改业务行：

```bash
mysql app_learning < examples/database/13-mysql-locks-and-mvcc-inspection.sql
psql --dbname=app_learning --file=examples/database/13-postgresql-locks-and-mvcc-inspection.sql
```

MySQL 当前连接若没有取得 InnoDB 数据锁，`data_locks` 查询返回空集是预期结果；若账号不能读取 Performance Schema，相应查询会报告权限问题。PostgreSQL 查询通常至少能看到当前后端与虚拟事务等锁，具体集合随版本和语句变化。

脚本没有自动制造两会话阻塞，因为可重复的锁等待演示需要共享测试表、两个长期会话和严格清理。把这种演示塞进单文件自动执行，容易留下未提交事务或让执行器永久等待。

## MySQL 与 PostgreSQL 对照

| 主题 | MySQL InnoDB | PostgreSQL |
| --- | --- | --- |
| 旧版本主要位置 | undo 信息重建旧版本 | 表堆中的旧 tuple |
| 后台旧版本治理 | purge | VACUUM / autovacuum |
| 范围并发控制 | RR 下常见 gap/next-key locks | MVCC、谓词锁或序列化冲突机制取决于级别 |
| 锁定读 | `FOR SHARE`、`FOR UPDATE` | 四种行锁强度 |
| 当前锁视图 | `data_locks`、`data_lock_waits` | `pg_locks` |
| 活跃事务入口 | `innodb_trx` | `pg_stat_activity` |

对照表用于建立导航，不表示两套实现一一对应。例如 PostgreSQL SERIALIZABLE 中的 predicate lock 用于检测读写依赖，并不等同于 InnoDB 阻止插入的 gap lock。

## 后端工程检查清单

### 设计阶段

1. 写清楚业务不变量和可能并发的接口。
2. 优先使用唯一约束、条件更新和原子表达式。
3. 需要先读后改时，明确采用乐观锁还是锁定读。
4. 为实际查询形状建立索引，并用执行计划验证。
5. 统一跨行、跨表操作的锁定顺序。

### 编码阶段

1. 使用数据库驱动提供的事务 API，任何异常路径都回滚。
2. 不在事务中等待 HTTP、消息队列或用户操作。
3. 设置有层次的锁、语句和请求超时。
4. 将死锁和序列化失败作为有限重试路径。
5. 用幂等键保护可能重复发生的外部副作用。
6. 连接归还池前恢复自动提交和会话级设置。

### 运行阶段

1. 监控活跃事务时长、等待数量、死锁率和回滚率。
2. 监控连接池等待，区分数据库锁等待与无可用连接。
3. MySQL 关注 undo/purge 压力，PostgreSQL 关注 dead tuples 与 autovacuum。
4. 将数据库连接 ID/PID 与应用请求 ID 关联。
5. 保留慢请求的 SQL 模板、参数范围和执行计划，而不是记录敏感参数原文。
6. 终止事务前评估回滚耗时、业务状态和级联影响。

## 常见误区

### “用了 MVCC 就不需要锁”

MVCC 主要让快照读与写并发。写写冲突、锁定读、约束检查和结构变更仍需要锁或等价协调机制。

### “行锁只锁 WHERE 命中的行”

InnoDB 锁范围受索引扫描路径影响；范围查询可能锁记录和间隙，缺索引时可能扫描远超最终结果的记录。PostgreSQL 也会因行锁、表锁或约束关系出现比业务对象更复杂的等待。

### “死锁就是数据库故障”

死锁检测和选择受害者是数据库恢复进展的正常机制。频繁死锁通常说明访问顺序、事务长度或索引需要优化，但应用仍必须正确处理偶发死锁。

### “只要把超时调大就能解决”

调大超时只会让更多请求和连接堆积。先找根阻塞事务，再优化事务边界、索引、访问顺序和重试策略。

### “SKIP LOCKED 可以让所有列表查询更快”

它主动返回不完整集合，只适用于允许分工领取的队列式语义。报表、余额、权限和普通列表不能用它掩盖阻塞。

## 本课小结

- MVCC 用快照和旧版本提升读写并发，锁负责互斥写入和显式协调。
- 快照读返回可见版本，锁定读取得当前可锁定版本并持锁到事务结束。
- InnoDB 的记录锁、间隙锁和 next-key lock 与索引访问路径紧密相关。
- PostgreSQL 提供四种行级锁定读强度，并以 VACUUM 清理不再需要的旧 tuple。
- 写写冲突仍会等待；NOWAIT 快速失败，SKIP LOCKED 只适合队列式领取。
- 死锁必须回滚并有限重试整个事务，统一锁顺序可降低发生率。
- 长事务既延长锁等待，也阻碍旧版本回收；事务中不应进行慢速外部调用。
- 排障要沿等待关系找到根阻塞者，终止会话前必须评估业务和回滚影响。
