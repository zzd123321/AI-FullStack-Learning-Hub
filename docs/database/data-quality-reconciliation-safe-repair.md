# 数据质量、跨系统对账与安全修复

数据库返回 `COMMIT`，只能说明这次事务满足了当前数据库能够检查的规则；它不自动证明金额符合业务含义、订单与支付跨系统一致、CDC 没漏事件，也不证明迁移后的新库与旧库相同。

后端接口常见的 `200 OK` 也是同一回事：它是一次请求的技术结果，不是长期数据正确性的证明。本课从“什么才算正确”开始，建立“预防—检测—定位—审批—修复—复验”的闭环。

## 先定义数据质量，而不是先写扫描 SQL

“没有 NULL”不是完整的数据质量。至少要区分以下维度：

| 维度 | 要回答的问题 | 示例 |
| --- | --- | --- |
| 完整性 | 应有的数据是否存在 | 已支付订单必须有支付成功事实 |
| 唯一性 | 同一业务事实是否被重复记录 | 同一支付渠道流水号只能入账一次 |
| 有效性 | 值是否满足格式和范围 | 金额非负，币种来自允许集合 |
| 一致性 | 多个字段或系统是否表达同一事实 | 订单实付 = 支付成功 - 已退款 |
| 时序性 | 状态变化是否遵守时间与状态机 | `PAID` 不能早于支付成功时间 |
| 新鲜度 | 数据是否在约定延迟内到达 | 报表落后主库不超过 10 分钟 |
| 可追溯性 | 能否解释值从何而来、何时改变 | 保存 source event、批次与规则版本 |

其中“新鲜度差”不一定是“内容错误”。一个在允许延迟内的只读副本与主库暂时不同是预期状态；把它误报成数据损坏，会制造无效修复。

## 正确性的核心是业务不变量

不变量是无论经过多少请求、重试和并发，系统都应持续满足的命题。例如：

```text
订单应付金额 = 商品小计 + 运费 - 优惠
订单已付净额 = 成功支付总额 - 成功退款总额
可用库存 >= 0
会计分录同一币种借方合计 = 贷方合计
租户 A 的行不能被租户 B 读取或关联
```

定义不变量时要写清五件事：

1. **范围**：按订单、账户、租户还是自然日判断。
2. **权威来源**：冲突时以哪个原始事实为准，而不是以哪个表查询方便为准。
3. **时间语义**：业务时区、截止水位、迟到事件和退款冲正如何处理。
4. **精度**：金额用最小货币单位整数或精确小数，不能用浮点近似相等。
5. **例外**：测试账户、人工审核态和历史规则版本是否排除。

不变量应该可判定。诸如“订单数据基本正常”无法直接检测、告警或验收。

## 权威来源不是“最后写入者”

跨系统差异发生时，先建立数据所有权：

- 订单服务拥有订单生命周期，支付服务拥有渠道交易事实。
- 缓存、搜索索引、数仓宽表通常是派生副本，不应反向覆盖主记录。
- 财务流水宜采用不可变事实与冲正分录，不应直接把历史余额改成“看起来正确”。
- 人工录入也必须有依据、审批人和审计记录，不能天然获得最高优先级。

`updated_at` 更新更晚不代表更可信：它可能来自重放、回填、时钟漂移或错误脚本。权威性来自领域所有权和证据链。

## 三层防线：尽量让错误无法写入

### 第一层：数据库约束

优先使用数据库能原子维护的规则：

- `PRIMARY KEY`、`UNIQUE` 防止标识与幂等键重复。
- `NOT NULL` 防止必填事实缺失。
- `CHECK` 限制单行字段间关系。
- `FOREIGN KEY` 维护同一数据库内的引用完整性。
- 合适的数据类型避免用任意字符串表达金额、时间和状态。

约束比只写在某个接口中的校验更可靠，因为后台任务、管理脚本和未来服务也会经过它。但约束并非万能：跨行聚合、跨库关系和外部支付事实通常不能由一个普通 `CHECK` 表达。

SQL 的三值逻辑也容易被忽视。`CHECK (amount_cents >= 0)` 对 `NULL` 的结果是 unknown，MySQL 和 PostgreSQL 都允许该行通过；必填时还要配合 `NOT NULL`。

### 第二层：事务内业务校验

需要读取当前状态再决定写入时，把读取、条件写和必要的幂等记录放进同一事务。不要采用“先查询库存够不够，再无条件扣减”的接口流程；更稳妥的是让更新条件直接包含旧状态或版本：

```sql
UPDATE inventory
SET available = available - :quantity,
    version = version + 1
WHERE sku_id = :sku_id
  AND available >= :quantity
  AND version = :expected_version;
```

受影响行数为 0 表示前置条件已经不成立，调用方应重新读取或返回冲突，而不是继续伪造成功。

### 第三层：异步对账

异步对账发现约束表达不了、跨系统传播造成或历史上已经存在的问题。它是最后一道检测网，不应成为放弃约束和事务设计的理由。

## 对账最难的部分：让两边处在可比较状态

如果系统仍在持续写入，依次读取 A 和 B：

```text
t1 读取订单：PAID
t2 退款提交
t3 读取支付汇总：净额已经扣除退款
```

即使两边都正确，也会得到混合时刻的结果。反复重跑可能“自愈”，但这只是观测竞态。

### 同一数据库：使用一致快照

MySQL InnoDB 在 `REPEATABLE READ` 下可用只读一致快照，让同一事务内的普通 `SELECT` 看到同一 read view。PostgreSQL 的 `REPEATABLE READ` 也能固定事务快照；多个并行 worker 需要完全相同的视图时，可由一个事务导出 snapshot，其他事务在规定顺序中导入。

长快照会阻碍旧版本清理、增加 undo/WAL 保留和复制压力。大表对账不能无限维持快照，应限制时长、分块并监控存储膨胀。

### 跨系统：使用共同业务水位

两个独立数据库无法共享 MVCC 快照。应选择双方都能解释的截止点，例如：

- CDC 日志位置、LSN 或 binlog position。
- 单调事件序号或 outbox ID。
- 已封账的业务日期。
- 明确包含规则的 `event_time <= cutoff`，同时另行处理迟到事件。

仅使用机器当前时间通常不够：时钟可能漂移，事件会迟到，两个系统消费进度也不同。对账批次必须记录 `source_watermark`、`target_watermark`、规则版本和数据范围，结果才可复现。

## 从总量到逐行：分层缩小差异

对账可以像诊断慢接口一样逐层下钻：

1. **批次级**：截止水位、记录数、金额总和、最大/最小业务键。
2. **分桶级**：按稳定业务键 hash 或范围分块，比较每块 count 与 digest。
3. **记录级**：只对不同分块提取主键与规范化字段。
4. **字段级**：分类为缺失、多余、值不同或状态时序冲突。

只比较 `COUNT(*)` 会漏掉“一边少一行、同时多另一行”；只比较总金额也会漏掉相互抵消的错误。摘要是定位工具，不是密码学意义上的绝对证明；关键账务仍需保存差异明细并验证业务不变量。

### 分块必须稳定

`LIMIT 10000 OFFSET ...` 在持续变化的数据集上会跳行或重复，而且越往后越慢。优先使用不可变、唯一且有索引的游标：

```sql
WHERE business_id > :last_seen_id
  AND business_id <= :batch_upper_bound
ORDER BY business_id
LIMIT :chunk_size
```

若按 hash 分桶，源端和目标端必须使用完全相同的算法、字符编码和版本。不要依赖不同数据库实现可能不同的内置 hash。

## 规范化：相同事实必须产生相同字节

对账摘要之前先定义 canonical form：

- 字段顺序固定，字段名明确，不能简单拼接无分隔字符串。
- 明确区分 `NULL`、空字符串、缺失字段与数值 0。
- 金额统一为最小货币单位或固定 scale 的 decimal。
- 时间转为 UTC，并规定微秒/毫秒精度与截断方式。
- Unicode、大小写、尾随空格是否等价要按业务定义。
- JSON 对象键排序；数组若有业务顺序则不能擅自排序。
- 排除纯技术字段前要确认它们确实不影响业务语义。

规范化规则本身需要版本号。规则升级造成的 digest 改变，不应被误判为生产数据突然损坏。

## CDC 对账要识别重复、乱序和缺口

CDC/消息系统常见的至少一次投递意味着重复是正常输入，消费端需要以 event ID 或业务幂等键去重。还要分别监控：

- **重复**：同一事件再次出现，结果不应重复累加。
- **乱序**：版本 12 先于版本 11 到达，旧事件不能覆盖新状态。
- **缺口**：序号从 10 跳到 12，不能假设 11 永远不存在。
- **毒消息**：持续失败的事件不能静默跳过，需隔离并告警。
- **截断/重建**：重做全量快照时要与增量水位无缝衔接。

消费者显示“lag = 0”只说明已读到当前末尾，不证明此前没有静默跳过或错误转换。因此需要端到端业务对账。

## 差异分类决定处置方式

对账产物至少包含：批次、业务键、差异类型、源/目标版本、字段差异、证据位置与首次发现时间。

| 差异 | 可能原因 | 首选动作 |
| --- | --- | --- |
| 目标缺失 | CDC 漏事件、过滤错误 | 确认水位后幂等补发/重放 |
| 目标多余 | 删除事件遗漏、错误导入 | 先证明目标是派生副本，再隔离或标记失效 |
| 内容不同 | 转换规则、精度、乱序覆盖 | 修正转换/版本判断，再按权威事实重建 |
| 暂时不同 | 合法复制延迟、迟到事件 | 等待宽限期后复验，不立即修复 |
| 权威源自身矛盾 | 上游不变量已破坏 | 暂停自动修复，升级为业务事故 |

差异数量突然变为 0 也可能是扫描任务没读到数据、权限变化或过滤条件错误。监控必须同时观察扫描覆盖率、耗时、水位推进和失败数。

## 安全修复不是无条件 UPDATE

危险脚本常把扫描结果导出，审批数小时后再按主键覆盖；这期间用户可能已经完成合法更新。安全流程应具备以下门禁：

1. **不可变 manifest**：记录批次、键、原值摘要、期望版本、建议值、原因和规则版本。
2. **dry run**：输出预计影响量、租户分布和金额影响，不写数据。
3. **条件写**：`WHERE id = ? AND version = ?` 或比较旧值摘要；条件不再满足就拒绝。
4. **幂等**：以 repair ID 去重，重试不会重复记账或重复发消息。
5. **小批 canary**：低速、限量、可暂停，观察锁、复制延迟和业务指标。
6. **双人审批与审计**：高风险账务修复不能由同一个人生成并执行。
7. **复验**：在新水位重新对账，而不是以受影响行数宣告成功。

对于账本、余额和库存，优先追加带关联 ID 的冲正/补偿事实，让历史可解释；直接改写旧流水会破坏审计链。对于搜索索引或缓存这类派生数据，通常从权威源重建比逐字段修补更安全。

修复期间若根因仍在持续制造坏数据，应先止血或隔离写入口，否则修复速度可能永远追不上新增差异。

## 对账任务自身也要像生产服务一样设计

- 使用只读、最小权限账号；修复账号与扫描账号分离。
- 设置 statement/lock timeout、并发与 I/O 预算，避开业务高峰。
- 按租户或稳定键限速，避免一个大客户独占任务。
- checkpoint 只在整块结果持久化后推进，失败可安全续跑。
- 结果表和对象存储需有保留期、脱敏、访问控制与审计。
- 对账 SQL 也要检查执行计划，避免大表全扫拖垮主库。
- 监控扫描覆盖率、watermark lag、差异率、误报率、修复拒绝率和复验通过率。

正确性指标不能被一个百分比掩盖。例如 99.99% 对十亿行仍是十万条差异；支付金额差一分钱和推荐标签差一条也不是同一严重度。SLO 应按不变量和业务影响分级。

## MySQL 与 PostgreSQL 的约束盘点

本课附件只读查询系统目录：

- MySQL `information_schema.TABLE_CONSTRAINTS` 可查看约束类型和 `CHECK` 是否 `ENFORCED`。
- PostgreSQL `pg_constraint.convalidated = false` 表示约束尚未验证历史行；外键或 `CHECK` 使用 `NOT VALID` 时可以先约束后续新写，再安排 `VALIDATE CONSTRAINT` 完成存量校验。

“约束对象存在”不等于“所有历史数据已经通过验证”，也不等于业务规则已完整表达。盘点结果必须回到不变量清单逐项映射。

## 上线检查清单

- 每个关键领域都有可判定的不变量、owner、严重度与权威来源。
- 可由数据库维护的规则已经使用正确类型与约束表达。
- 对账记录双方水位、范围、规则版本和运行代码版本。
- 快照或业务截止点能排除持续写入造成的假差异。
- canonical form 明确 NULL、金额、时间、Unicode 与 JSON 语义。
- 扫描使用稳定游标/分桶，有限速、超时、checkpoint 和覆盖率监控。
- 差异经过宽限期和分类，不把正常复制延迟当作损坏。
- 修复 manifest 不可变，支持 dry run、条件写、幂等和审计。
- 账务采用冲正或补偿事实，不无痕覆盖历史。
- 修复后在新水位复验，并验证根因已经停止产生新差异。

## 常见误区

### “两边行数相等，所以迁移成功”

缺失和多余可能互相抵消，还可能存在字段转换错误。至少分块比较摘要并下钻记录差异。

### “每分钟重跑，最终一致就不需要水位”

没有共同截止点，无法区分传播延迟和真实错误，结果也不可复现。

### “发现源和目标不同，就让目标等于源”

必须先确认权威来源、规则版本和目标是否发生新写；源端自身也可能已经错误。

### “加 CHECK 就解决了所有历史问题”

约束表达能力有限；未验证约束不保证历史行，且 `CHECK` 对 NULL 的语义经常被误解。

### “修复 UPDATE 影响了预期行数，所以问题结束”

它可能覆盖并发更新、触发新副作用或被持续根因再次破坏。条件写、审计和新水位复验缺一不可。

### “lag 为零证明 CDC 完整”

它不证明没有跳过、错误转换、重复应用或历史缺口，仍需端到端不变量对账。

## 本课小结

- 数据正确性来自明确的不变量，而不是 `COMMIT`、HTTP 200 或行数相等。
- 约束阻止局部错误，事务维护即时规则，异步对账覆盖跨行和跨系统关系。
- 持续写入时必须使用一致快照或共同业务水位，避免制造假差异。
- 对账按批次、分块、记录、字段逐层下钻，并对比较字段做有版本的规范化。
- CDC 的重复、乱序、缺口和转换错误需要分别检测，lag 为零不是完整性证明。
- 安全修复依赖不可变 manifest、dry run、版本条件、幂等、canary、审计与复验。
- 账本类事实优先追加冲正，派生副本优先从权威源重建。
- 对账系统自身也要受资源预算、权限、可观测性和数据保护约束。
