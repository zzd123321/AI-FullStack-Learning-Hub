# 从业务需求到表结构：实体、关系、范式与约束

会写 SQL 不等于会设计数据库。查询是在既有结构上取数据，设计则要决定哪些事实值得长期保存、它们如何关联、哪些状态必须由数据库拒绝，以及需求变化时结构能否安全演进。

本课以用户、资料、地址和角色为主线，从一组熟悉的后端接口推导关系模型。重点不是背诵范式定义，而是建立一条可重复的设计路径：业务语言 → 数据事实 → 约束 → 查询形状 → 物理实现。

## 从接口字段退后一步

假设前端联调时见过下面的响应：

```json
{
  "id": 1001,
  "email": "alice@example.com",
  "displayName": "Alice",
  "status": "active",
  "profile": {
    "bio": "Backend learner",
    "avatarUrl": null
  },
  "addresses": [
    {
      "id": 301,
      "label": "home",
      "city": "Shanghai",
      "detail": "...",
      "isDefault": true
    }
  ],
  "roles": ["member", "editor"]
}
```

不能把响应 JSON 原样等同于一张表。API 为一次交互组织数据，关系模型为长期一致性组织事实。一个响应可能来自五张表，也可能包含计算字段、权限裁剪和其他服务的数据。

先把需求改写成业务事实：

- 用户以稳定 ID 标识，邮箱在约定作用域内唯一。
- 每个用户至多有一份扩展资料。
- 一个用户可以保存多个地址，每个地址只属于一个用户。
- 用户可以拥有多个角色，一个角色也可以分配给多个用户。
- 用户只能处于允许的状态。
- 角色分配不能引用不存在的用户或角色。

这些句子比字段清单更重要，因为它们直接导出主键、唯一约束、外键、检查约束和关系表。

## 设计的五层视角

| 层次 | 关注点 | 产物 |
| --- | --- | --- |
| 概念模型 | 业务中有哪些对象与关系 | 用户、地址、角色及关系图 |
| 逻辑模型 | 事实如何表示且不自相矛盾 | 表、列、键、范式、约束 |
| 物理模型 | 特定数据库如何高效存储访问 | 类型、索引、分区、命名空间 |
| 接口模型 | 客户端需要怎样的读写契约 | DTO、分页结构、错误码 |
| 运维模型 | 数据如何增长、归档、迁移与审计 | 生命周期、迁移方案、监控指标 |

初学时常直接从 JSON 跳到 `CREATE TABLE`，结果是为当前页面优化了结构，却没有定义数据含义。正确顺序是先保证逻辑模型能表达事实，再依据查询形状补充物理设计。

## 识别实体、属性与值对象

### 实体

实体拥有独立身份和生命周期。例如用户即使修改邮箱和昵称，仍是同一用户；因此用户需要不会随这些属性变化的稳定标识。

判断一个概念是否应成为实体，可以问：

1. 是否需要被独立引用？
2. 属性变化后是否仍视为同一个对象？
3. 是否有独立创建、更新、停用或归档流程？
4. 是否会被多个其他对象关联？

### 属性

属性描述实体，例如用户邮箱、显示名和状态。每个属性要明确：

- 数据含义和单位。
- 是否必填，缺失与空字符串是否不同。
- 值域与格式由谁验证。
- 是否唯一，唯一性的业务作用域是什么。
- 是否可变，修改后是否需要保留历史。
- 是否敏感，能否进入日志、索引和分析系统。

### 值对象

金额、时间范围、地址片段等概念可能没有独立身份，而是由若干值共同表达。是否拆表取决于复用、约束、查询和生命周期，不是“对象嵌套就一定拆表”。

例如订单收货地址通常需要保存下单时的快照。若只引用用户当前地址，用户后来修改地址会改变历史订单含义。此时把地址字段复制到订单快照是有业务语义的冗余。

## 主键：稳定标识一行

主键必须唯一且非 NULL。一张关系表通常应明确主键，以便外键引用、更新定位、复制同步和故障排查。

### 自然键

自然键来自业务本身，例如国家代码或标准编号。它的优点是有业务含义，缺点是业务规则可能变化、值可能较长，或所谓“永不变化”后来被证明并不可靠。

邮箱通常不适合作用户表主键：用户可能更换邮箱，大小写与国际化规则复杂，还可能因隐私要求需要替换。

### 代理键

代理键只用于标识，例如数据库生成的 BIGINT 或应用生成的 UUID。它将实体身份与可变业务属性分离。

选择时要考虑：

| 方案 | 优点 | 代价 |
| --- | --- | --- |
| 自增 BIGINT | 紧凑、索引局部性好、容易调试 | 可预测；跨节点生成需额外设计 |
| 随机 UUID | 可离线生成、跨系统合并方便 | 更宽；随机写入可能降低索引局部性 |
| 有序分布式 ID | 大致有序、可跨节点生成 | 依赖生成器、时钟和位分配规则 |

没有一种主键适合所有系统。对外暴露不可猜测 ID 的安全需求，也不能只靠换成 UUID：授权检查仍然不可省略。

### 候选键与唯一约束

能够唯一标识实体的属性组合都是候选键，选作 PRIMARY KEY 的只是其中一个。其余业务唯一性仍应通过 UNIQUE 约束表达。

```sql
CONSTRAINT uq_users_email UNIQUE (email)
```

应用先查“邮箱是否存在”只能改善提示，不能取代唯一约束。两个请求可能同时查到不存在，只有数据库唯一约束能在最终写入点裁决竞争。

## 关系如何映射为表

### 一对一：用户与扩展资料

如果每名用户至多一份资料，可以让资料表的主键同时成为外键：

```sql
CREATE TABLE user_profiles (
  user_id BIGINT PRIMARY KEY,
  bio VARCHAR(500),
  avatar_url VARCHAR(500),
  CONSTRAINT fk_profiles_user
    FOREIGN KEY (user_id) REFERENCES users (user_id)
);
```

`PRIMARY KEY (user_id)` 保证一个用户最多一行资料，外键保证资料必须属于真实用户。是否拆为一对一表要有理由，例如字段很少读取、权限不同、字段很宽或生命周期不同；不要为了“看起来整洁”把每组三列都拆表。

### 一对多：用户与地址

在“多”的一侧保存外键：

```text
users 1 ─────< user_addresses N
```

地址表的 `user_id` 指向用户。同一个用户可以出现多次，不同地址行各自只有一个所有者。

### 多对多：用户与角色

多对多需要关联表：

```text
users N >────< N roles
          user_roles
```

`user_roles` 至少包含 `user_id` 与 `role_id`，二者组成复合主键，防止同一角色重复分配：

```sql
PRIMARY KEY (user_id, role_id)
```

关联表不是只能放两个 ID。若业务需要，还可以保存 `granted_at`、`granted_by`、作用域或到期时间。此时“角色分配”本身已成为有属性的业务事实。

### 自关联：组织树与回复链

员工的经理、分类的父节点可以用同表外键表达：

```sql
manager_id BIGINT NULL REFERENCES employees(employee_id)
```

邻接表适合读取父节点和直接子节点。深层树的整棵子树查询可能需要递归 CTE、物化路径或闭包表，不能仅凭“树结构”选择复杂模型。

## 外键维护引用完整性

外键保证子表引用的父键存在。它解决的是数据关系合法性，不自动解决接口权限、业务流程或查询性能。

常见删除动作：

| 动作 | 含义 | 适用判断 |
| --- | --- | --- |
| `RESTRICT` / `NO ACTION` | 有子记录时拒绝删除 | 子记录必须先明确迁移或处理 |
| `CASCADE` | 删除父记录时级联删除子记录 | 子记录完全依附父记录、无独立保留价值 |
| `SET NULL` | 父记录删除后解除关系 | 关系可选且 NULL 有清晰含义 |

不要为了“省代码”对所有外键使用 CASCADE。删除用户后，资料和临时地址也许可级联；已支付订单、审计记录和财务流水通常必须保留或匿名化。

MySQL InnoDB 中 `NO ACTION` 与立即执行的 `RESTRICT` 等效。PostgreSQL 的 `NO ACTION` 与 `RESTRICT` 在可延迟约束等细节上并非完全相同，建模时要阅读目标版本文档。

外键列还需要按查询形状考虑索引。数据库为了约束检查可能建立或要求某些索引，不代表所有反向查询已经优化。例如频繁按 `role_id` 查用户时，复合主键 `(user_id, role_id)` 不能很好服务该前导列查询，通常还要索引 `(role_id, user_id)`。

## 约束应尽量靠近数据

后端校验负责友好错误、权限与复杂流程；数据库约束负责无论数据来自哪个服务、脚本或管理工具，都不能提交基础非法状态。

| 规则 | 首选数据库机制 |
| --- | --- |
| 用户 ID 必须存在 | FOREIGN KEY |
| 邮箱在作用域内唯一 | UNIQUE |
| 状态只能取有限值 | CHECK 或受控字典表 |
| 必填字段不能缺失 | NOT NULL |
| 数量不能为负 | CHECK |
| 两列组合不重复 | 复合 UNIQUE / PRIMARY KEY |

### CHECK 与 NULL

SQL 的 CHECK 在表达式结果为 FALSE 时拒绝；结果为 TRUE 或 UNKNOWN 时通常通过。因此：

```sql
price DECIMAL(12, 2) CHECK (price >= 0)
```

并不会自动禁止 NULL。若价格必填，还要写 `NOT NULL`。

CHECK 一般用于当前行内稳定、确定性的条件。跨行聚合规则例如“每名用户只能有一个默认地址”，通常应使用唯一索引、重新建模或事务协议，而不是试图在 CHECK 内查询其他行。

### 状态列还是状态表

稳定且很小的集合，如 `active`、`disabled`，可以使用 CHECK。若状态有显示名、排序、权限、启停配置或需要运营动态维护，更适合独立字典表和外键。

数据库原生 ENUM 约束直观，但新增、重命名状态可能增加迁移成本。是否采用要结合状态变化频率和数据库实现，而不是把所有字符串都改成 ENUM。

## 函数依赖：范式背后的语言

若一组属性 X 的值确定另一组属性 Y，记作 `X → Y`。

例如在用户表中：

```text
user_id → email, display_name, status
email   → user_id
```

前提是 `user_id` 和 `email` 都有唯一性保证。函数依赖描述业务事实，不是看几行样例数据猜出来的规律。

再看订单明细：

```text
(order_id, product_id) → quantity, deal_price
product_id             → current_product_name
order_id               → ordered_at, customer_id
```

这些依赖帮助判断哪些属性属于订单、商品或某一条订单明细。

## 第一范式：一个单元表达一个关系值

第一范式（1NF）要求关系中的属性在当前模型下是不可再分的单值，不能把重复组塞进同一行。

反例：

| user_id | role_ids |
| --- | --- |
| 1001 | `2,5,9` |

逗号字符串无法可靠建立外键、唯一性和单项索引，删除角色 5 还要解析文本。正确做法是每个角色分配一行 `user_roles`。

“原子”取决于业务操作。完整地址可作为展示文本，也可拆为国家、省市和详细地址；如果业务要按城市筛选或校验国家代码，就应保存结构化字段。

JSON 不必然违反 1NF，但它会把部分结构和约束移出普通列。JSON 适合模式变化快、很少参与关系约束的附加信息；核心标识、金额、状态、关联键和高频筛选字段通常应建为明确列。

## 第二范式：消除对复合键的部分依赖

第二范式（2NF）在 1NF 基础上，要求非键属性依赖整个候选键，而不是复合键的一部分。它主要在主键或候选键由多列组成时显现。

错误的订单明细：

| order_id | product_id | ordered_at | product_name | quantity | deal_price |
| --- | --- | --- | --- | --- | --- |

若主键是 `(order_id, product_id)`：

- `ordered_at` 只依赖 `order_id`。
- 当前 `product_name` 只依赖 `product_id`。
- `quantity` 与成交价才依赖这条订单明细。

应拆成：

- `orders(order_id, ordered_at, customer_id, ...)`
- `products(product_id, current_name, ...)`
- `order_items(order_id, product_id, quantity, deal_price, ...)`

注意成交时商品名若是法律凭证或历史展示的一部分，可以作为订单快照保存。那不是“不懂范式”，而是它依赖订单明细这一历史事实，不再代表商品当前名称。

## 第三范式：消除非键之间的传递依赖

第三范式（3NF）在 2NF 基础上，避免非键属性通过另一个非键属性间接依赖候选键。

例如：

| employee_id | department_id | department_name |
| --- | --- | --- |

若 `employee_id → department_id`，且 `department_id → department_name`，那么部门名对员工 ID 是传递依赖。把部门拆为 `departments`，员工只保留 `department_id`，可以避免部门改名时更新大量员工行。

3NF 直接减少三类异常：

- 更新异常：同一事实复制多份，只改了一部分。
- 插入异常：没有员工时无法先保存部门。
- 删除异常：删除最后一名员工意外丢失部门信息。

## BCNF 与“范式越高越好”的误解

BCNF 要求每个非平凡函数依赖的决定因素都是超键，比 3NF 更严格。复杂排课、人员角色等模型可能满足 3NF 却不满足 BCNF。

多数业务表先做到 3NF、明确候选键和约束，已经能消除大量问题。范式是检查冗余来源的工具，不是分表数量竞赛。继续分解必须验证是否无损连接、依赖能否保持，以及主要查询是否仍然可接受。

## 何时可以反范式

反范式是有意识地复制可推导数据，以降低读取成本或保留历史，而不是随意把所有字段放进一张宽表。

合理场景包括：

- 订单保存成交时的商品名称、单价和税率快照。
- 统计表保存按天聚合的指标。
- 读模型保存搜索或展示需要的冗余字段。
- 分布式边界内保存来自其他服务的必要快照。

采用前必须回答：

1. 哪一份是真相来源？
2. 冗余何时、由谁更新？
3. 允许多长时间不一致？
4. 更新失败如何补偿和重建？
5. 如何监控两份数据漂移？

如果回答不了，就还没有形成可维护的反范式设计。

## NULL、默认值与三种“没有”

设计可空列时要区分：

- 未知：目前不知道出生日期。
- 不适用：企业账号没有个人生日。
- 尚未发生：订单还没有支付时间。

NULL 可以表达缺失，但无法自动区分所有语义。有时需要状态列、类型列或独立子表共同表达。

不要用 `''`、`0`、`1970-01-01` 伪装缺失值，它们会污染统计和约束。默认值也应是业务上真实、稳定的初始事实，而不是为了少传一个字段。例如订单状态默认 `pending` 合理，用户生日默认今天显然错误。

## 时间、金额和可变事实

### 时间

明确时间列表示瞬时时刻还是不带时区的本地日历时间。订单创建时间是瞬时事件；商店每天 09:00 营业是本地规则，需要同时知道时区。

应用通常以 UTC 传输和存储瞬时时刻，在展示边界转换时区。PostgreSQL `timestamp with time zone` 保存的是规范化瞬时值；MySQL TIMESTAMP 与 DATETIME 的时区转换语义不同，应按官方文档和驱动设置验证。

### 金额

金额使用 DECIMAL/NUMERIC 或最小货币单位整数，不使用二进制浮点数。还要保存币种，因为 `100.00` 没有币种就不是完整金额事实。

### 可变业务值

商品现价、用户现地址和当前税率不应回写改变历史订单。区分“指向当前实体的引用”和“事件发生时的快照”，是数据库设计中最重要的语义边界之一。

## 软删除不是免费的布尔字段

`deleted_at` 可以保留记录用于恢复、审计或异步清理，但会改变几乎所有查询和约束：

- 每个正常查询都要排除已删除行。
- 外键仍然认为软删除的父行存在。
- 唯一邮箱能否被新账号复用需要明确。
- 索引可能要包含或过滤删除状态。
- 数据最终何时物理清理仍需生命周期规则。

MySQL 和 PostgreSQL 对“仅活跃行唯一”的实现不同。PostgreSQL 可使用带条件的唯一索引；MySQL 常借助可空生成列或重新设计唯一键。不要仅在应用中先查再插入。

软删除也不能替代审计日志。审计需要记录谁在何时以什么请求改变了哪些重要状态，并控制访问与保留期限。

## 多租户必须把作用域写进约束

如果邮箱只需在租户内唯一，应表达为：

```sql
UNIQUE (tenant_id, email)
```

任何按业务编号引用的数据也要防止跨租户串联。常见方案是让候选键包含 `tenant_id`，子表以复合外键引用同一租户下的父记录。

仅在每条 SQL 的 WHERE 中“记得加 tenant_id”很脆弱。除了数据库约束，还应在仓储层、权限层和测试中建立租户边界；PostgreSQL 场景也可评估行级安全策略，但它不是普通外键的替代品。

## 命名与表结构可读性

命名应稳定、一致、能从错误信息中定位业务：

- 表名统一采用单数或复数，不在同一库混用。
- 主键统一为 `id`，或使用 `user_id` 这类显式名称，两种都可，但团队应一致。
- 外键列名称体现目标，例如 `user_id`，不要叫 `user`。
- 时间语义用 `created_at`、`paid_at`，布尔语义用 `is_default` 等清晰名称。
- 显式命名重要约束，如 `fk_user_roles_user`、`uq_users_email`。
- 避免保留字、难懂缩写和把类型写进名称。

数据库注释应解释业务语义、单位和不明显的约束原因，而不是重复“用户 ID 是用户的 ID”。

## 表结构不等于 API 契约

不要直接 `SELECT *` 后原样返回。数据库模型为一致性服务，API 模型为用例服务：

- API 可以把多表聚合成嵌套响应。
- 敏感列如密码摘要、内部风控状态永远不应因“表里有”而返回。
- 数据库 snake_case 可以映射为接口 camelCase。
- API 的派生字段不一定需要落库。
- 写接口 DTO 不应允许客户端提交所有数据库列。

联调时若字段变化，应先判断是接口展示变化还是业务事实变化。只有后者通常需要数据库迁移。

## 从需求到 DDL 的完整流程

1. 收集写入命令、查询场景、报表、生命周期和合规需求。
2. 用一句话定义每张表“一行代表什么事实”。
3. 识别实体、值对象、关系和事件快照。
4. 列出候选键、函数依赖和业务不变量。
5. 映射 1:1、1:N、N:M，并规范化到合适程度。
6. 选择精确类型，明确 NULL、默认值和单位。
7. 用主键、唯一、外键、NOT NULL、CHECK 表达基础规则。
8. 写出关键读写 SQL，再按查询形状设计索引。
9. 评估并发更新、删除动作、历史保留和租户边界。
10. 设计向前兼容迁移、回滚策略和上线监控。

设计评审时应同时查看表结构和关键 SQL。只有 ER 图而没有访问路径，会漏掉索引和并发问题；只有当前 SQL 而没有业务不变量，会得到难以演进的数据堆积。

## 配套建模脚本

两份脚本建立相同的用户领域模型并查询系统目录验证约束：

```bash
mysql app_learning < examples/database/14-mysql-database-design.sql
psql --dbname=app_learning --file=examples/database/14-postgresql-database-design.sql
```

它们只应在专用学习数据库执行。脚本使用 `IF NOT EXISTS`，不会删除、截断或覆盖既有业务表；对象分别使用 `learning14_` 前缀和 `learning14` schema，与前面示例隔离。为遵守非破坏性原则，脚本不会自动清理创建的学习对象。

重复执行会保留已存在结构。`IF NOT EXISTS` 不会验证旧表是否与文件中的定义完全一致；若曾手工改过学习表，应换一个空白学习数据库，而不是关闭外键检查强行继续。

MySQL 和 PostgreSQL 脚本分开维护，因为自动编号、布尔值、时间类型、schema 和系统目录不同。生产迁移还必须使用版本化迁移工具，不能把教学文件直接当作上线脚本。

## 设计评审清单

### 语义与键

1. 能否用一句话说明每张表的一行代表什么？
2. 主键是否稳定，候选键是否用唯一约束保留？
3. 每个字段的单位、NULL 含义和可变性是否明确？
4. 历史快照与当前实体引用是否区分？

### 关系与约束

1. 1:1、1:N、N:M 是否映射正确？
2. 外键删除动作是否符合数据生命周期？
3. 跨租户或跨作用域引用是否被约束阻止？
4. 基础不变量是否只存在于应用代码？
5. CHECK 与 NULL 的三值逻辑是否符合预期？

### 范式与性能

1. 是否存在逗号列表、重复列组或无约束 JSON？
2. 非键列是否依赖完整候选键，而不是键的一部分？
3. 是否存在可拆出的传递依赖？
4. 每一处冗余是否有真相来源、同步和重建策略？
5. 索引是否来自关键查询，而不是给每列盲目加索引？

### 演进与运维

1. 预计的数据量、增长率和保留期限是什么？
2. 软删除、归档、匿名化和审计是否各有明确目标？
3. 结构变化能否以兼容的多阶段迁移发布？
4. 是否能从约束名和日志快速定位失败原因？
5. 生产变更是否考虑锁、回填、复制延迟和回滚？

## 常见误区

### “前端返回什么，表里就存什么”

响应结构针对页面，表结构针对事实与一致性。直接对应会导致重复字段、嵌套列表难以约束，并把接口版本变化变成数据库重构。

### “有 ORM 就不需要设计关系”

ORM 负责映射和生成 SQL，不会替你定义业务候选键、删除语义、事务边界和历史快照。错误模型只会被 ORM 更快地写入数据库。

### “所有规则都放在应用里更灵活”

应用可能有多个实例、批处理和管理脚本，并发请求也会绕过先查后写。基础完整性必须由数据库作最终防线。

### “外键会影响性能，所以生产库不要外键”

外键确有写入检查、索引和变更成本，但移除后必须用另一套可证明、可监控的机制保证引用完整性。应基于规模、架构边界和实测决策，不能用口号换取孤儿数据。

### “达到 3NF 就完成设计”

范式只处理特定冗余与依赖问题。类型、约束、查询索引、并发、租户隔离、历史和迁移同样属于设计。

## 本课小结

- API 响应是用例视图，关系表保存稳定业务事实，二者不应机械一一对应。
- 主键标识实体，其余候选键仍要用唯一约束表达。
- 一对一可共享主键，一对多在多方保存外键，多对多使用关联表。
- 外键维护引用完整性，删除动作必须按子数据生命周期选择。
- 1NF 消除重复组，2NF 消除部分依赖，3NF 消除非键传递依赖。
- 反范式必须有真相来源、同步时机、一致性目标和重建策略。
- NULL、时间、金额、软删除、租户作用域和历史快照都需要明确语义。
- 数据库约束与应用校验相互补充，唯一性竞争必须由数据库最终裁决。
- 设计完成后还要用真实读写 SQL 验证索引、并发与迁移边界。
