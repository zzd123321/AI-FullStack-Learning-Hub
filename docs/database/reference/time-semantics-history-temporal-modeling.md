# 时间语义、历史版本与时态数据建模

用户在 7 月 16 日提交订单，运营人员在 7 月 18 日补录一条“从 7 月 15 日起生效”的折扣规则。随后客服问：**按当时有效的规则，订单价格是多少？按系统当时已经知道的信息，又会算出多少？**

一个 `created_at` 回答不了这两个问题。后端系统中的“时间”至少可能表示瞬时时刻、本地日历规则、业务有效区间、系统获知时间、处理时间或数据库提交顺序。字段都叫 `timestamp`，不代表语义相同。

本课从接口联调常见的日期错位开始，逐步建立可查询、可约束、可修订的历史模型。

## 先问“这个时间描述什么”

看到任何时间字段，先完成一句话：

```text
paid_at 表示支付平台确认付款发生的全球唯一时间点。
business_date 表示按门店所属时区归属的营业日。
valid_from 表示这版价格规则开始对业务事实生效的时间点。
recorded_at 表示本系统第一次持久化这版信息的时间点。
```

如果说不清，类型、默认值、索引和 API 格式都无从正确选择。

常见语义如下：

| 语义 | 示例 | 建模要点 |
| --- | --- | --- |
| 瞬时时刻 instant | 支付确认、登录发生 | UTC 可比较时间点，输入输出带偏移量 |
| 本地日期 | 生日、营业日 | `DATE`，没有时区转换 |
| 本地时间规则 | 门店每天 09:00 开门 | 当地日期/时间 + IANA 时区规则 |
| 持续时长 | 超时 30 秒 | duration，不是墙上时钟结束点 |
| 业务有效时间 | 合同从何时有效 | `valid_from` / `valid_to` |
| 系统记录时间 | 系统何时得知/更正 | `recorded_from` / `recorded_to` |
| 处理时间 | 消费者何时处理事件 | 可观测延迟，不等于事件发生时间 |

## 时间点、偏移量和时区不是一回事

瞬时时刻是在时间线上唯一的位置，例如 `2026-07-16T01:00:00Z`。`+08:00` 是某个时刻相对 UTC 的固定偏移量，而 `Asia/Shanghai` 是一套会随历史和政策变化的时区规则。

固定偏移量无法表达“未来每年当地 09:00”的规则。许多地区有夏令时：当地某个钟面时间可能不存在，也可能出现两次。即便某地区当前不使用夏令时，未来规则也可能变化。

因此：

- 已发生的支付时间保存为 instant，API 使用带 `Z` 或偏移量的 RFC 3339/ISO 8601 字符串。
- 未来本地计划保存当地日期/时间和 IANA zone ID，例如 `America/New_York`。
- 若审计要求保留用户原始表达，可额外保存输入偏移量或原始文本，但不能用它替代规范化时间点。
- 展示时区是用户界面选择，不应反向改变已保存的事实时间。

## MySQL 与 PostgreSQL 时间类型的关键差异

### MySQL

- `TIMESTAMP` 在连接会话时区与 UTC 之间转换，适合范围允许时表达 instant。
- `DATETIME` 保存给定的日历字段，不执行同样的会话时区转换；它既可能被用于本地日历时间，也常被团队约定存 UTC，但这种约定必须由应用和测试保证。
- `DATE` 表示日历日期，不应经过时区换算。
- `TIMESTAMP(fsp)` 与 `DATETIME(fsp)` 的 `fsp` 为 0 到 6；MySQL 默认省略小数秒时精度为 0。
- 两类时间的可表示范围不同，选型前要核对产品寿命和官方版本范围。

### PostgreSQL

- `timestamp with time zone`（`timestamptz`）表示 instant，输入按显式或会话时区解释，内部规范化，输出按当前会话时区展示；它不会保存原始 zone ID。
- `timestamp without time zone` 只保存日历字段；即使字面量中看似带偏移量，若被确定为无时区类型，偏移信息也不会形成 instant 语义。
- `date`、`time` 与 `interval` 分别表达日期、一天内时间和时间间隔。
- 时间戳精度可到微秒；应用语言和驱动是否保持同等精度必须实测。

“数据库里看到的文本不一样”不一定说明 instant 不同，因为会话时区会影响格式化输出。比较前应查看数据库类型、会话时区和带偏移量的序列化结果。

## 连接池让会话时区成为共享状态

连接池会复用 session。如果请求 A 改了 `time_zone`/`TimeZone` 却没有恢复，请求 B 可能以错误规则解析无偏移量输入或输出时间。

稳妥策略是：

1. 启动时把数据库、连接池和应用的 instant 会话基线设为 UTC。
2. 每次借出连接时验证或重置关键会话变量。
3. 参数绑定使用明确的驱动时间类型，不拼接依赖本地格式的字符串。
4. 业务时区作为显式参数参与边界计算，而不是临时修改共享 session。
5. 在日志和健康检查中记录数据库实际时区、应用时区和驱动版本。

MySQL 使用命名时区执行 `CONVERT_TZ` 还依赖时区表正确加载；不可只因开发机查询成功就假设所有环境一致。

## “现在”也有多个定义

在 PostgreSQL 中，`CURRENT_TIMESTAMP`/`transaction_timestamp()` 表示当前事务开始时间；长事务中多次调用会得到同一基准。`statement_timestamp()` 表示当前语句开始时间，`clock_timestamp()` 才会随实际时钟推进。

这不是缺陷。事务时间能让同一事务写入的多行拥有一致批次时间，但它不适合测量一段长事务已经运行多久。

MySQL 的 `CURRENT_TIMESTAMP()` 是 `NOW()` 的同义词，按会话时区返回；当前时间函数在查询开始时求值，`SYSDATE()` 具有不同的执行时刻及复制相关语义。业务代码不应随意互换。

还要区分：

- 应用收到请求的时间。
- 数据库事务/语句开始时间。
- 数据库提交成功的顺序或日志位置。
- 外部事件实际发生时间。

自增 ID、UUID 和墙上时钟都不天然等于提交顺序。服务器时钟也可能漂移或回拨。需要严格因果顺序时，应使用事务、版本号、日志水位或数据库提供的顺序机制，而不是只比较毫秒时间戳。

## 精度不匹配会破坏游标与幂等

浏览器可能提供毫秒，数据库保存微秒，某些语言对象又只保留毫秒。如果写入 `12:00:00.123456`，接口返回 `12:00:00.123`，再拿返回值做乐观锁条件，可能匹配不到原行。

风险还包括：

- 同一精度内多行时间相等，单独按时间分页产生重复或遗漏。
- 把时间戳当幂等键时发生碰撞。
- 数据迁移截断精度后改变边界归属。
- JSON 序列化有时带 `Z`、有时没有，客户端解释不同。

应确定端到端精度契约，并对列表使用 `(occurred_at, id)` 这类稳定复合游标。幂等使用独立稳定键，不使用“当前时间”。

## 时间范围使用半开区间

区间统一为 `[start, end)`：包含开始、不包含结束。

```sql
WHERE occurred_at >= :start_instant
  AND occurred_at <  :end_instant
```

半开区间的优点：

- 相邻区间 `[A, B)` 与 `[B, C)` 没有重叠或缝隙。
- 不需要猜数据库最大精度，也不写 `23:59:59.999999`。
- 月份、账期和版本切换边界可以准确拼接。
- B-Tree 更容易利用原始列范围，不需对每行调用 `DATE()`。

自然日查询要先在业务时区中构造本地 `[00:00, next day 00:00)`，再按该时区规则把两个边界各自转换成 instant。不能简单地给开始时间加固定 24 小时：遇到夏令时切换，一天可能不是 24 小时。

## 有效时间：业务世界何时认为它成立

价格历史可以设计为：

```text
product_price_history
  product_id
  price_cents
  valid_from       inclusive
  valid_to         exclusive, NULL 表示当前无已知终点
  recorded_at
  change_reason
```

查询某时刻 `T` 有效的版本：

```sql
SELECT product_id, price_cents, valid_from, valid_to
FROM product_price_history
WHERE product_id = :product_id
  AND valid_from <= :as_of
  AND (valid_to > :as_of OR valid_to IS NULL);
```

真正困难的是保证同一商品的有效区间不重叠。如果两行同时覆盖 `T`，接口该返回哪一行就变得不确定。

### 应用先查再插不能防并发重叠

两个事务都执行“当前没有重叠”，然后分别插入互相重叠的区间，在普通隔离级别下都可能通过。正确边界应由数据库约束、串行化事务或对稳定父行加锁来保证。

PostgreSQL 可使用 range type 与 exclusion constraint 表达“同一实体的时间范围不得重叠”；新版本还提供与时间周期相关的 `WITHOUT OVERLAPS` 能力。具体语法和外键覆盖语义需按部署版本验证。

MySQL 没有通用的 range type/exclusion constraint 等价物。常见方案是在事务中锁住 `products` 的稳定父行，再检查重叠并写入；所有写入口必须遵守同一锁协议。仅触发器中查询同表也有产品限制与并发边界，不能未经验证就宣称安全。

## 系统时间：本系统何时知道这条信息

有效时间回答“业务上何时成立”，系统时间回答“数据库从何时到何时认为这条版本是当前记录”。

假设 7 月 18 日才获知价格其实从 7 月 15 日生效：

```text
valid_from    = 2026-07-15T00:00:00Z
recorded_from = 2026-07-18T09:20:00Z
```

若只保存有效时间，今天能重算正确价格，却无法回答 7 月 16 日系统为何按旧价出账。保存两条时间轴后，可以分别查询：

- “按现在掌握的全部信息，7 月 16 日有效的价格是什么？”
- “站在 7 月 16 日当时，系统已知且有效的价格是什么？”

同时保存有效时间和系统时间称为双时态建模。它不是每张表都需要，而适用于合同、定价、权限、地址归属、监管报表等需要解释历史认知的领域。

## 双时态查询的两个 as-of 条件

一种便于理解的逻辑模型：

```text
price_versions(
  product_id,
  price_cents,
  valid_from,
  valid_to,
  recorded_from,
  recorded_to,
  correction_id,
  reason
)
```

站在系统时间 `S`，查询业务时间 `V` 的版本：

```sql
SELECT price_cents
FROM price_versions
WHERE product_id = :product_id
  AND valid_from <= :business_as_of
  AND (valid_to > :business_as_of OR valid_to IS NULL)
  AND recorded_from <= :system_as_of
  AND (recorded_to > :system_as_of OR recorded_to IS NULL);
```

每条轴都采用半开区间。旧系统版本被更正时，不直接覆盖或删除：把旧记录的 `recorded_to` 关闭，再插入新的系统版本。二者必须在同一事务中完成。

### 历史表不是审计日志的完全替代

历史版本描述状态变化，审计日志还可能需要操作者、请求 ID、审批单、来源 IP、变更理由和失败尝试。二者可以关联，但目的不同：

- 历史表用于可靠地恢复和查询业务状态。
- 审计日志用于解释谁通过什么渠道做了什么。
- CDC 记录行级变化流，但业务意图和长期保留承诺未必足够。
- 备份用于灾难恢复，不适合作为在线 as-of 查询接口。

## 当前表加历史表，还是单表版本链

### 当前表 + 历史表

当前查询简单且快速，历史单独增长。更新当前行时在同一事务把旧版本写入历史。难点是所有写入口、批处理和管理脚本都必须触发相同历史逻辑，跨两表查询也更复杂。

### 单表版本链

当前版本只是 `valid_to IS NULL` 或 `recorded_to IS NULL` 的一行，历史与当前统一查询。模型一致，但热点写入、唯一性、索引和普通查询都更复杂；每次查询必须明确时间切片。

### 追加事件 + 投影

事件保存业务变化，当前状态和历史查询由投影生成，表达力最强。代价是事件 schema 演进、重放、快照、重复顺序治理；不能因为“事件不可变”就忽略删除、更正和隐私合规。

选择取决于 as-of 查询频率、修订规则、保留期、写量、数据库能力和团队运维经验，而不是哪种模式更“先进”。

## 索引要匹配具体时间切片

单一 B-Tree 很难同时优化实体、有效时间、系统时间和开放区间。先列出真实查询：

```text
按 product_id 读取当前有效版本
按 product_id 查询某个 business_as_of
按 recorded_from 扫描最近修订
按租户和月份归档已关闭版本
```

常见起点包括 `(product_id, valid_from)`、当前行的受限/等价索引以及按 `recorded_from` 的扫描索引。PostgreSQL range 查询可评估 GiST；MySQL 父行锁方案仍需适合重叠检查的 `(product_id, valid_from, valid_to)` 访问路径。

索引无法替代无重叠约束，也不能自动解决双时间轴的基数估计。必须用生产分布执行 `EXPLAIN ANALYZE` 或对应执行计划工具，特别关注开放区间比例、热门实体版本数和时间相关性。

历史表通常只增不减，但索引、vacuum、页分裂、分区数量和归档仍会消耗资源。不要因为普通接口只读当前行，就让历史无限增长到无法恢复。

## 删除、更正与隐私请求

“历史不可变”是审计设计原则，不是凌驾于法规和隐私要求之上的口号。需要删除个人数据时，可采用经审批的物理删除、字段脱敏、密钥销毁或将身份信息与业务事实解耦；方案应由法律、合规和安全共同确定。

业务更正则通常追加补偿版本，保留原值、修订值、原因和操作者。财务金额尤其不应无痕覆盖。删除历史前还要评估备份、CDC、搜索索引、缓存和下游仓库的传播范围。

## API 契约如何表达时间

推荐清晰区分：

```json
{
  "occurredAt": "2026-07-16T01:30:00.123Z",
  "businessDate": "2026-07-16",
  "scheduleLocalTime": "09:00:00",
  "scheduleTimeZone": "Asia/Shanghai",
  "validFrom": "2026-07-15T00:00:00Z",
  "validTo": null
}
```

接口还应明确：

- 缺少偏移量的时间点输入是拒绝，还是按哪个显式 zone 解释。
- 小数秒最大精度、超出时舍入还是拒绝。
- `validTo: null` 表示开放区间，还是未知值。
- 区间边界是否为 `[start, end)`。
- `asOf` 默认使用请求时间、事务时间还是最新已知版本。
- 不存在历史版本返回 404、空结果还是领域错误。

不要把本地日期 `2026-07-16` 自动序列化成 UTC 午夜；它不是 instant。也不要让 JavaScript `Date`、Java `LocalDateTime` 等名字相似的语言类型掩盖业务语义。

## 测试必须穿过日历边界

只测试普通工作日下午远远不够。至少覆盖：

- UTC 日期与用户本地日期不同的时刻。
- 夏令时跳过的本地时间，以及重复出现的本地时间。
- 闰日、月末、年末和不同时长月份。
- 时间戳精度截断、并列时间和复合游标。
- 相邻 `[A, B)`、`[B, C)` 边界恰好相接。
- 一个区间完全包含另一个、部分相交、零长度和开放终点。
- 两个连接并发插入重叠版本，数据库必须拒绝其中一个或事务重试。
- 迟到更正同时影响有效时间与记录时间。
- 连接池复用后 session 时区仍为基线。

测试固定 clock 和显式 zone，不依赖运行机器的“本地时区”。时区数据库会更新，跨 runtime/数据库计算未来日历计划时还要验证 tzdata 版本差异。

配套的 `examples/database/33-temporal-history-model.mjs` 用纯内存模型演示半开区间、重叠拒绝、迟到更正和双 as-of 查询。它用于解释不变量，不代替真实数据库并发测试。

## 从错误时间模型迁移

把无时区列改成 instant 前，必须先回答旧数据当初按什么时区写入。如果历史记录混合了多个未知时区，无法通过一次 `ALTER COLUMN` 凭空恢复事实。

安全流程通常是：

1. 盘点列类型、默认值、会话配置、驱动映射和消费者。
2. 按业务来源把旧值分类，明确可证明、可推断和未知三类。
3. 新增语义清晰的新列，同时写入旧列与新列。
4. 分批回填，并记录使用的 zone、规则版本和置信度。
5. 对账 UTC 边界、自然日分桶和样本业务凭证。
6. 切换读取，观察后再停止旧写入并退役旧列。

未知值应进入人工处理或显式不确定状态，不能静默套用当前服务器时区。历史时区规则也应使用事件发生当时适用的 tzdata，而不是今天的固定偏移量。

## 可观测性

时间错误常表现为“少一天”“重复一小时”或“偶发找不到”，需要结构化证据：

- 输入原文、解析结果、显式偏移量和 zone ID；日志必须脱敏。
- 应用、数据库 session 和操作系统时区基线。
- 事件时间、接收时间、处理时间和提交/水位之间的延迟分布。
- 无有效版本、多有效版本、区间重叠和开放版本数量。
- 迟到更正数量、最大迟到时长和受影响账期。
- 数据库与应用 tzdata 版本，以及升级后的差异测试。

不要用 `created_at` 同时冒充事件发生、请求接收和数据库插入时间。只有分别记录，才能判断延迟发生在哪一段。

## 常见误区

### “统一存 UTC 就解决所有时间问题”

UTC 适合 instant，但生日、营业日和未来当地排班不是 instant。未来日历规则还必须保存 zone ID。

### “时间戳越大，事务一定越晚提交”

事务开始时间、应用时钟和提交顺序不同，长事务甚至可能较早取时、较晚提交。

### “先查没有重叠，再插入就安全”

并发事务可能同时通过检查。无重叠必须由数据库约束或经证明的锁/隔离协议保证。

### “`BETWEEN` 更直观”

`BETWEEN` 两端都包含，容易让相邻时间窗重复计入边界记录。时间窗口优先显式写半开条件。

### “保留 updated_at 就有完整历史”

它只描述当前行最后更新时间，旧值和系统当时认知已经丢失。

## 上线检查清单

- 每个时间字段是否能用一句话说明语义？
- instant、本地日期、本地规则、duration 是否使用匹配的类型？
- API 是否要求时间点携带 `Z`/偏移量，并规定精度？
- 数据库与连接池 session 时区是否有可验证基线？
- 自然日边界是否按业务 zone 分别转换，而非固定加 24 小时？
- 所有时间窗是否统一为 `[start, end)`？
- 有效时间与系统记录时间是否按审计需求分开？
- 同一实体的有效区间重叠是否能在并发下被数据库拒绝？
- 当前、历史和 as-of 查询是否各有计划与索引证据？
- 迟到、更正、删除和开放区间是否定义业务规则？
- 夏令时、月末、精度和并发边界是否进入真实引擎测试？
- 历史增长、迟到延迟、重叠异常和 tzdata 是否可观测？

## 本课小结

- 时间类型必须由业务语义选择，字段名或统一 UTC 口号不能代替建模。
- instant、固定偏移量、IANA 时区规则和本地日历时间彼此不同。
- MySQL `TIMESTAMP`/`DATETIME` 与 PostgreSQL `timestamptz`/`timestamp` 的转换语义不同。
- “现在”可能是事务开始、语句开始或实际时钟；时间戳也不天然代表提交顺序。
- 时间窗口采用 `[start, end)`，自然日边界应按业务 zone 分别转换。
- 有效时间描述业务何时成立，系统时间描述数据库何时知道；双时态模型可回答历史认知问题。
- 历史版本需要无重叠约束、并发安全写入、明确 as-of 查询和匹配索引。
- 迟到更正通常追加并关闭旧版本，而不是无痕覆盖事实。
- 时间模型必须经过 DST、精度、并发、迁移和 tzdata 变化验证。
