# 分区表、数据生命周期与归档

订单、审计日志、消息投递记录等表会持续增长。刚开始时，一张普通表配合合适索引往往已经足够；当数据达到数亿行后，真正困难的可能不是某一次查询，而是下面这些问题：

- 在线接口只查最近 30 天，却要维护多年历史数据。
- 每月清理过期数据会产生大量日志、锁竞争、复制延迟和存储碎片。
- 新索引、统计信息和备份窗口随表增长而变长。
- 历史数据访问很少，却与热点数据竞争缓存和 I/O。

分区把一张逻辑表按规则拆成多个物理分区。应用仍可查询父表，数据库根据谓词决定访问哪些分区。它最重要的价值通常是**缩小需要访问的数据集合，并让整段数据的生命周期操作变得可管理**，而不是让任意 SQL 自动变快。

本课从接口查询形状和保留策略反推分区方案，再讲 MySQL 与 PostgreSQL 的实现差异、分区裁剪、约束、索引、锁和完整归档闭环。

## 先分清分区、分片、复制与归档

| 机制 | 数据放在哪里 | 主要解决什么 | 应用是否感知 |
| --- | --- | --- | --- |
| 表分区 | 通常仍在同一数据库实例 | 裁剪、维护和生命周期管理 | 多数查询可不感知 |
| 分库分表/分片 | 多个数据库实例或集群 | 单实例容量、吞吐或故障域边界 | 路由和事务通常要感知 |
| 复制 | 多个节点保存相同或近似相同数据 | 高可用、读扩展、灾备 | 一致性和路由需要感知 |
| 归档 | 历史数据移到独立存储或归档库 | 降低在线成本并满足保留要求 | 历史查询路径需要感知 |

分区不是分片。分区表中的分区通常共享同一实例的 CPU、内存、磁盘、事务日志和故障域；实例容量不足时，多建分区不会凭空增加资源。

归档也不是备份。归档关注“长期保留和低频读取”，备份关注“在故障、误操作或损坏后恢复系统”。只有一份归档文件且从未验证恢复，不构成可靠备份。

## 什么时候值得分区

分区最适合同时出现以下特征的表：

1. 数据持续追加，并有清晰的时间或业务生命周期。
2. 高频查询通常携带分区键范围，例如 `occurred_at >= ? AND occurred_at < ?`。
3. 大批数据会按相同边界过期、归档或迁移。
4. 单表维护成本已经通过执行计划、锁、日志量或维护窗口得到证实。

常见候选有事件流水、审计日志、时序指标、消息 outbox、订单历史和账单明细。

以下情况不应急着分区：

- 表规模不大，瓶颈其实是缺索引、返回过多列或 N+1 查询。
- 主要查询没有稳定的分区条件，每次仍需访问所有分区。
- 数据会频繁修改分区键，导致跨分区移动。
- 唯一性、外键或 ORM 能力与目标数据库的分区限制冲突。
- 团队还没有自动创建未来分区、监控边界和恢复归档的运维能力。

先用普通索引解决查询路径，再在生命周期或维护成本确实需要时引入分区，通常更稳妥。

## 从接口契约推导分区键

假设审计接口是：

```http
GET /tenants/42/audit-events?from=2026-01-01T00:00:00Z&to=2026-02-01T00:00:00Z
```

对应查询：

```sql
SELECT id, tenant_id, occurred_at, event_type
FROM audit_events
WHERE tenant_id = :tenant_id
  AND occurred_at >= :from_time
  AND occurred_at < :to_time
ORDER BY occurred_at DESC, id DESC
LIMIT :limit;
```

如果数据按 `occurred_at` 每月分区，时间条件可以排除无关月份；分区内部再使用 `(tenant_id, occurred_at, id)` 一类索引完成租户过滤和排序。这里有两层不同决策：

```text
时间谓词 → 选择分区
租户 + 时间 + id → 在选中分区内使用索引
```

分区不是索引的替代品。只有时间分区而没有适配查询的分区内索引，访问当月数千万行仍然可能很慢。

### 好的分区键要同时回答四个问题

- **查询**：常见接口能否自然携带这个条件？
- **生命周期**：过期与归档是否恰好沿这个边界发生？
- **分布**：数据是否会极端倾斜，使某个分区过热或过大？
- **约束**：主键、唯一键和外键能否满足数据库限制？

时间字段经常合适，但不是永远正确。按租户删除全部数据是主要需求、且租户规模均衡时，租户维度可能重要；单个巨型租户又会让“一个租户一个分区”形成热点和分区爆炸。

## RANGE、LIST 与 HASH

### RANGE：按连续范围切分

时间序列最常用：

```text
[2026-01-01, 2026-02-01) → 2026 年 1 月
[2026-02-01, 2026-03-01) → 2026 年 2 月
```

下界包含、上界不包含的半开区间能无重叠地拼接，也避免“月底 23:59:59.999999”这类精度陷阱。查询也应坚持 `>= start AND < end`。

### LIST：按离散值切分

例如区域、业务类型或状态集合。它适合值域稳定且数量有限的维度，不适合每天新增大量枚举值的字段。

### HASH：按哈希余数分散

HASH 可把行较均匀地分散到固定数量分区，适合缓解单一物理结构热点。但它不能天然表达“删除 2024 年全部数据”，范围查询也可能访问所有分区，因此生命周期收益较弱。

## 选择日、月还是年分区

粒度不是越细越好。可以从三个量估算：

```text
单分区数据量 ≈ 每日写入量 × 分区天数
在线分区数 ≈ 在线保留天数 ÷ 分区天数
单次查询 fan-out ≈ 查询时间跨度 ÷ 分区天数
```

例如每天 500 万行、在线保留 180 天：

- 日分区约 500 万行，约 180 个在线分区；查 30 天会涉及约 30 个分区。
- 月分区约 1.5 亿行，约 6～7 个在线分区；查 30 天通常涉及 1～2 个分区。
- 年分区约 18.25 亿行，维护和归档粒度可能过粗。

还要考虑索引大小、单次归档单位、常见查询跨度、建分区自动化和数据库元数据开销。PostgreSQL 官方明确提醒：分区过多会增加规划时间和会话内存；只有典型查询能裁剪到少量分区时，大量分区才可能合理。

不存在通用的“每月分区最佳实践”。应先用真实增长量和查询跨度算出候选，再压测执行计划、规划时间和维护窗口。

## 分区裁剪为什么有效

分区裁剪的核心不是扫描以后再过滤，而是数据库根据分区边界证明某些分区不可能产生结果，从执行计划或执行过程排除它们。

```sql
WHERE occurred_at >= TIMESTAMP '2026-02-01 00:00:00'
  AND occurred_at <  TIMESTAMP '2026-03-01 00:00:00'
```

若表按月 RANGE 分区，数据库可以只访问 2 月分区。反之：

```sql
WHERE tenant_id = 42
```

不包含时间边界，时间分区无法被排除；即便每个分区都有租户索引，仍可能产生跨所有分区的 fan-out。

### 让谓词与分区边界对齐

更容易裁剪的写法通常是直接比较分区列和常量/参数：

```sql
WHERE occurred_at >= :start_time
  AND occurred_at < :end_time
```

把列包在函数中可能让推导变困难：

```sql
WHERE DATE(occurred_at) = :day
```

具体能否裁剪取决于数据库、版本、分区表达式和函数支持，不能仅凭语法猜测。保留原始范围谓词通常也更有利于普通 B-Tree 索引。

MySQL 可用 `EXPLAIN`/`EXPLAIN PARTITIONS` 查看 `partitions`；PostgreSQL 可用 `EXPLAIN (ANALYZE, BUFFERS)` 观察实际执行的分区，以及 `Subplans Removed`、`never executed` 等线索。绑定参数、运行时值和 prepared statement 还可能让裁剪发生在执行期，所以要用接近生产的参数验证。

### 裁剪不等于扫描一定少

即使只选中一个月分区，查询仍可能：

- 在该分区做全表扫描。
- 因低选择性读取大量行。
- 因 `ORDER BY` 不匹配而排序。
- 返回过多列或行，使网络和序列化成为瓶颈。

必须继续检查分区内部的访问方法、估算与实际行数、排序、缓冲区读取和接口返回量。

## 主键、唯一性与外键边界

### MySQL 8.4

MySQL 要求：分区表达式用到的每一列，都必须包含在表的每个唯一键中，包括主键。若按 `occurred_at` 分区，只有 `PRIMARY KEY (id)` 不满足要求，常见形式会变成：

```sql
PRIMARY KEY (id, occurred_at)
```

这改变了数据模型：数据库保证的是 `(id, occurred_at)` 组合唯一，而不是 `id` 单列全局唯一。若业务要求 `id` 单列全局唯一，需要由 ID 生成方案、额外映射表或其他架构边界保证，并认真评估并发失败模式。

另一个重要限制是：MySQL 8.4 中，分区的 InnoDB 表不能拥有外键，也不能被外键引用。这可能直接否决对强依赖外键的核心交易表进行原生分区。

### PostgreSQL 18

PostgreSQL 分区父表上的 `UNIQUE` 或 `PRIMARY KEY` 约束必须包含全部分区键列；用于这些约束的分区键不能包含表达式或函数。原因相同：各分区的索引只能在本分区检查唯一性，只有分区键参与唯一键时，两个相同键值才会被路由到可判定冲突的位置。

设计前要写清楚：

- 业务真正要求单列全局唯一，还是“租户内/时间范围内唯一”？
- 外部 API 是否只暴露 `id`，数据库却需要复合键？
- 子表引用、幂等键和 upsert 冲突目标是否仍正确？
- ORM 生成的主键、迁移和关联查询是否支持复合键？

不要在建表失败后才临时把时间列塞进主键；那是业务约束变化，不只是 DDL 调整。

## 默认分区、迟到数据与未来分区

按时间分区时，需要同时处理三种边界：

1. **未来数据**：下个月分区是否提前创建？
2. **迟到数据**：补录的旧事件是否仍有目标分区？
3. **异常时间**：时钟错误或解析错误是否把数据写到意外年份？

PostgreSQL 中，没有任何分区覆盖且没有 DEFAULT 分区时，插入会报错。这种 fail-closed 行为能尽早暴露运维遗漏，但未来分区创建必须可靠。

DEFAULT 分区可提高写入可用性，却容易变成垃圾桶：漏建分区、异常时间和迟到数据都悄悄堆积。以后附加新分区时，PostgreSQL 还可能扫描并以强锁检查 DEFAULT 分区，除非已有能证明排除目标范围的约束。

合理做法是：

- 至少提前创建若干个未来分区，并监控“可写时间地平线”。
- 对 default/兜底分区的任何新增行告警。
- 明确迟到数据窗口；过期窗口之外走专门补录或归档流程。
- 统一使用 UTC 存储和切分边界，在展示层转换时区。
- 拒绝明显超出合理范围的时间，而不是永久污染兜底分区。

## 分区键应尽量不可变

PostgreSQL 更新分区键后可能把一行从一个分区移动到另一个分区；这不是简单的原地修改，并发下还可能遇到序列化失败。其他实现也可能有重写或限制。

事件发生时间通常应视为事实字段，写入后不再改变。若业务需要修正展示日期，可以区分：

- `occurred_at`：不可变的路由/事实时间。
- `business_date`：可修正的业务归属日期。
- `recorded_at`：数据库接收时间。

不要随意用会频繁变化的 `updated_at` 做时间分区键，否则普通更新会变成跨分区移动。

## 索引与统计信息仍是每个分区的工作

从逻辑上看，可以在父表定义索引；物理上，各分区仍有自己的索引结构。新增分区时必须确认相应索引和约束已创建或附加。

PostgreSQL 分区父表上的索引是“虚拟”的：数据位于子分区索引中。对分区父表直接创建索引当前不能使用 `CONCURRENTLY`；大型在线系统常先在父表 `ONLY` 创建无效索引，再逐分区并发创建并 `ATTACH PARTITION`，最后使父索引有效。这个流程需要版本化迁移、失败恢复和完整性检查，不能手工临场操作。

分区也有独立数据分布。刚创建或批量导入的分区若统计信息缺失，优化器可能严重误估。发布流程应覆盖 `ANALYZE`、统计信息新鲜度和典型查询计划。

## 分区生命周期不是一条 DROP 语句

推荐把每个时间分区视为具有状态的资产：

```text
ACTIVE（仍接收写入）
  ↓ 时间窗口结束且迟到写入截止
SEALED（只读或逻辑冻结）
  ↓ 导出完成
EXPORTED（已有归档对象）
  ↓ 数量、校验和、元数据、读取和恢复验证
VERIFIED（归档已验证）
  ↓ 从在线查询路径移除
DETACHED（已脱离在线父表）
  ↓ 保留期、法务冻结与审批允许
PURGE_ELIGIBLE（具备清理资格）
```

状态必须持久记录，任务重试要幂等。没有通过 `VERIFIED` 的分区不得 detach/drop；`PURGE_ELIGIBLE` 也只是有资格，不代表后台任务可以无审批立即删除。

### 1. 提前创建并验证未来分区

创建任务应重复执行无害，并验证：

- 边界连续、无重叠、无空洞。
- 表空间、权限、索引、约束和存储参数正确。
- 下一个可写边界距离现在仍有充足余量。
- 在 primary 和故障切换候选上，元数据迁移均已完成。

### 2. 封存而不是立即归档

月份结束不代表数据不再到达。应等待业务定义的迟到窗口，并确认没有补偿任务、CDC 或离线作业继续写入。封存点应以业务水位而不是单纯墙上时钟决定。

### 3. 导出并生成清单

归档对象除了数据，还应记录：

- 源实例/库/表/分区和精确边界。
- schema 版本、导出工具版本和时间。
- 行数、字节数、对象列表、内容校验和。
- 加密方式、密钥标识、访问策略和保留期限。
- 导出水位或一致性快照信息。

只比较压缩文件大小不足以证明完整性。行数也不能发现内容被替换，因此至少组合行数、确定性校验摘要和抽样业务不变量。

### 4. 验证“能恢复”，不只验证“能上传”

归档成功的定义应包括：对象可读取、校验和匹配、schema 可解释，并能在隔离环境恢复后运行代表性查询。恢复演练还要验证权限、加密密钥、时区、字符集和应用兼容性。

### 5. 先从在线路径移除，再观察

PostgreSQL `DETACH PARTITION` 能保留独立表，适合先移出父表后观察；`DETACH PARTITION ... CONCURRENTLY` 可降低父表锁级别，但有事务和 default partition 等限制。`DROP` 分区需要父表上的更强锁。

MySQL 的 partition management 可快速处理整段分区，但具体 `DROP`、`REORGANIZE`、`EXCHANGE` 的复制、锁和空间回收行为必须在相同版本与数据规模上演练。

无论哪种数据库，维护操作都应设置锁等待上限、观察事务和复制延迟，并具备取消路径。不要让一个计划内归档任务无限等待锁，最终拖垮线上连接池。

### 6. 最终清理需要独立门禁

清理前再次确认：

- 归档状态仍为 VERIFIED，校验对象仍可读。
- 在线观察期已结束，无查询回退需求。
- 法务冻结、审计和监管保留要求允许。
- 备份/时间点恢复窗口不被破坏。
- 操作对象由精确分区标识和边界双重校验。
- 两人审批或自动化策略授权已存在。

课程示例不提供可直接复制执行的删除分区命令，避免把不可逆操作变成普通脚本步骤。

## PostgreSQL 附加分区的锁与验证成本

把已有表 `ATTACH PARTITION` 到父表时，PostgreSQL 需要证明其中每一行都满足新边界。若没有匹配的 `CHECK` 约束，数据库可能扫描该表并持有较强锁。

若存在 DEFAULT 分区，附加新分区还要证明 DEFAULT 中没有属于新边界的行；缺少排除约束时可能扫描并锁住 DEFAULT 分区。这也是“先把所有异常行塞 default，以后再整理”会积累运维债务的原因。

安全迁移通常先：

1. 在独立表上建立与目标边界等价的 `CHECK` 约束。
2. 验证数据确实满足边界。
3. 为 DEFAULT 分区建立排除新范围的约束或先清理目标行。
4. 设置短锁等待，观察线上事务后执行 attach。
5. 验证父表查询、裁剪、索引和行数。

具体锁级别和 `CONCURRENTLY` 限制会随操作与版本变化，应以目标版本官方文档和预演结果为准。

## 将已有大表迁移为分区表

对繁忙大表直接做一次巨型重写，可能占满磁盘、产生海量 WAL/binlog、拖慢副本并长时间持锁。更稳妥的迁移按阶段进行：

```text
定义目标 schema 与分区边界
  ↓
创建未来分区、索引和约束
  ↓
按主键/时间小批回填历史数据
  ↓
CDC 或受控双写追平增量
  ↓
比较行数、摘要、业务不变量和查询结果
  ↓
短窗口切换读写路径
  ↓
观察并保留回退能力
```

双写不是简单地在业务代码多执行一条 INSERT：一边成功另一边失败时必须可重试、可对账，幂等键和事务边界要明确。能用数据库原生逻辑复制或成熟 CDC 时，也要监控延迟、DDL 兼容和失败重放。

回填应有速率上限并按稳定键分页，避免长事务和大 OFFSET；每批记录水位，失败后从已确认水位继续。验证不能只有总行数，还应分时间段/租户比较、检查 NULL 和唯一性，并抽样执行接口查询。

## 在线与归档查询如何协作

在线库只保留 180 天时，接口必须明确历史查询契约：

- 最近数据走在线数据库，提供低延迟、强索引查询。
- 历史数据走归档库、数据仓库或对象存储查询引擎。
- 跨边界查询由服务拆分为两段，再按稳定排序键合并。
- 历史路径可能是异步任务，而不是让 HTTP 请求等待数分钟。

分页尤其需要注意：在线数据持续变化，归档数据相对静态；一个普通 `OFFSET` 无法稳定跨两个后端。应使用包含来源、时间和唯一键的游标，并定义边界去重规则。

归档检索权限至少与在线数据同等严格。把数据移到对象存储不意味着隐私、租户隔离和删除请求失效。

## 可观测性：既看查询，也看时间地平线

### 查询路径

- 每类接口实际访问的分区数和扫描行数。
- 规划时间与执行时间。
- 分区内索引命中、排序、临时数据和缓冲区读取。
- 无分区谓词查询和跨大量分区查询的频率。

### 生命周期

- 最远未来分区边界距离当前时间。
- default/兜底分区新增行数和最早/最晚时间。
- 每个分区状态、大小、行数与迟到写入水位。
- 导出、校验、恢复演练的成功时间和重试次数。
- 已验证但未 detach、已 detach 但未到清理期的积压。

### 操作风险

- DDL 锁等待、长事务和被阻塞会话。
- 维护期间 WAL/binlog 生成量、复制延迟和磁盘余量。
- attach/detach/index rollout 的耗时与失败阶段。
- 归档对象校验失败、密钥不可用和恢复耗时。

“未来分区只剩 3 天”通常比“插入已经失败”更值得告警；可观测性的作用是把硬故障变成可提前处理的容量事件。

## 上线检查清单

### 设计

- 分区键由真实查询和保留边界共同推导。
- 分区粒度根据增长量、查询跨度和维护窗口计算并压测。
- 主键、唯一键、外键和 upsert 语义经过数据库版本验证。
- 分区键尽量不可变，时间边界统一使用 UTC 半开区间。
- 不把分区误当成跨实例扩容或故障隔离。

### 查询

- 高频接口显式携带可裁剪的分区范围。
- 分区内部仍有匹配筛选、排序和分页的索引。
- 使用真实绑定参数验证计划期/执行期裁剪。
- 监控跨分区 fan-out、规划时间和实际扫描量。
- 默认分区不是无监控的异常数据垃圾桶。

### 生命周期

- 自动提前创建未来分区并监控可写时间地平线。
- 迟到数据窗口、封存水位和异常时间策略明确。
- 导出清单包含 schema、边界、行数、摘要、加密和保留信息。
- 归档完成必须通过读取与隔离恢复验证。
- detach 与最终清理由不同门禁控制，法务冻结优先。

### 操作

- attach/detach/index 维护已在相同版本和数据规模预演。
- DDL 设置锁等待上限，有监控、取消与重试路径。
- 维护期间预留日志、磁盘和复制追赶容量。
- 大表迁移小批回填、可续跑、可对账并保留回退路径。
- 所有不可逆动作按精确对象、边界和审批执行。

## 常见误区

### “分区越多，查询越快”

更多分区会增加元数据、规划、索引和运维成本。查询不能裁剪时，还可能访问更多物理对象。

### “有分区就不用索引”

裁剪只减少候选分区；分区内部仍需合适索引。一个月分区本身仍可能有上亿行。

### “按时间分区后，按 ID 查询也会快”

如果请求只有 `id` 而不知道时间，数据库可能搜索所有时间分区。可让调用链携带时间/分区提示，或维护受控的 ID 到分区映射，但两者都增加契约成本。

### “DEFAULT 分区保证系统永不写失败”

它把漏建分区从显式故障变成静默堆积，还会提高以后 attach 的扫描和锁成本。必须告警并限制异常范围。

### “归档文件上传成功就能删在线数据”

上传成功不能证明内容完整、schema 可解释或密钥可用。必须校验并演练恢复，最终清理还要满足保留和审批条件。

### “detach/drop 分区一定没有锁和复制影响”

元数据操作仍需锁，也可能受长事务、default 分区验证、复制和版本行为影响。快速不代表无风险。

## 本课小结

- 分区主要解决裁剪与生命周期管理，不提供跨实例容量或故障隔离。
- 分区键应从高频查询、归档边界、数据分布和约束共同推导。
- RANGE 时间分区应使用 UTC 半开区间；粒度要平衡单分区大小、查询 fan-out 和分区数量。
- 分区裁剪要求查询谓词能与分区边界对齐，并且仍需分区内索引和真实执行计划验证。
- MySQL 与 PostgreSQL 的唯一约束通常都要求包含分区键；MySQL 分区 InnoDB 还有重要外键限制。
- 未来分区、迟到数据和 default 分区必须形成显式策略，分区键应尽量不可变。
- 归档要经过封存、导出、清单、内容校验、恢复验证、detach、观察和独立清理门禁。
- attach、detach 和索引维护仍可能加锁；大量现有数据应小批迁移、持续对账并保留回退路径。
- 在线与归档查询应有明确接口契约，不能让历史检索无边界地拖慢在线数据库。
