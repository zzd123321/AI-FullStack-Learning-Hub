# 分库分表、路由键与全局一致性

分区仍然主要共享一个数据库实例的计算、存储和故障域。当单实例写入吞吐、数据容量、维护窗口或故障影响范围已经无法满足目标时，才需要考虑把不同数据放到不同数据库节点，也就是分片。

分片不是“把一张大表平均切开”这么简单。原本由数据库隐式完成的定位、唯一性、事务、JOIN、排序、备份和故障恢复，会有一部分上移到路由层与业务系统。一个查询能否只访问一个分片，往往比单个分片内的 SQL 优化更重要。

本课从接口访问模式推导路由键，建立单分片优先原则，再解释热点、全局 ID、跨分片查询和写入、2PC 与 Saga、扩容迁移、拓扑版本和可观测性。

## 先证明单库确实到了边界

引入分片前应有可复现证据，而不是仅凭总行数：

- primary 的 CPU、I/O、日志写入或锁竞争持续达到容量上限。
- 写吞吐无法通过 SQL、索引、批处理和数据模型优化满足。
- 数据与索引增长使备份、恢复、校验、DDL 或故障切换超过时间目标。
- 单实例存储或 IOPS 无法在可接受成本内扩展。
- 一个租户或业务域的故障不应影响所有客户，需要缩小故障域。

在此之前通常先检查：

1. 是否存在低效查询、缺失索引、N+1 和无界返回。
2. 连接池、超时和过载保护是否正确。
3. 读流量是否能安全分担到副本或缓存。
4. 历史数据是否可通过分区和归档退出在线路径。
5. 纵向扩容是否更简单且仍满足未来容量周期。

分片带来的长期开发与运维成本很高。如果优化后的单库仍有足够余量，保持单库通常是更可靠的架构选择。

## 分片系统多了哪些组件

```text
HTTP 请求
  ↓ 提取 tenant_id / account_id
路由层：校验路由键 + 读取拓扑版本
  ↓
逻辑槽位：slot 0...N
  ↓ 拓扑映射
物理 shard：数据库 primary + replicas
  ↓
本地事务、索引和约束
```

关键概念：

- **路由键/shard key**：决定数据归属的业务字段。
- **逻辑槽位**：稳定的中间路由单元，避免业务键直接绑定物理节点。
- **物理分片**：实际数据库实例或高可用集群。
- **拓扑目录**：逻辑槽位到物理分片的版本化映射。
- **路由器**：应用库、代理或独立服务，依据目录选择节点。

路由器不只是负载均衡器。它必须理解写入角色、拓扑 epoch、迁移状态、超时、一致性 token 和失败语义。

## 垂直拆分与水平分片

### 垂直拆分：按业务边界分库

例如用户、订单、库存分别属于不同数据库。它减少模块耦合和故障范围，但原来的数据库 JOIN 和事务会跨服务边界。

好的垂直边界通常来自业务所有权，而不是把“列多的表”机械拆开。一个服务应拥有自己的数据写入规则，其他服务通过 API、事件或只读投影使用，而不是共享表写权限。

### 水平分片：同类行分布到多个节点

例如按 `tenant_id` 把订单分到 16 个逻辑槽，再把槽位分配到 4 个物理分片。每个分片拥有相同 schema，但存放不同租户的数据。

垂直拆分解决业务所有权，水平分片解决同类数据的容量与吞吐。两者可以组合，但复杂度也会叠加。

## 从接口访问模式选择路由键

假设接口是：

```http
GET /tenants/42/orders/1001
POST /tenants/42/orders
GET /tenants/42/orders?after=...
```

所有请求天然携带 `tenant_id`，订单的大部分写入和查询也限定在一个租户内，那么 `tenant_id` 是候选路由键：

```text
tenant_id → hash → logical slot → physical shard
```

好的路由键通常满足：

- 高频读写请求都能在执行 SQL 前获得它。
- 需要原子更新的数据尽可能具有相同路由键。
- 值的数量足够多且分布相对均衡。
- 写入后基本不变，避免跨分片搬行。
- 隐私删除、备份恢复和租户迁移边界与它一致。

### 路由键必须进入接口契约

只有 `GET /orders/1001` 而订单 ID 不编码或映射到分片时，路由器不知道查询哪个节点，只能：

- 广播到所有分片。
- 查一个全局 `order_id → shard` 目录。
- 修改接口，让上游同时提供 `tenant_id`。

第一种随分片数增长；第二种引入新的强依赖和一致性问题；第三种通常最直接，但会改变调用链。分片设计必须审查 API、消息事件、后台任务和管理工具是否都能携带路由上下文。

不要让客户端自由决定物理 shard 名称。客户端提供业务路由键，受控路由层根据当前拓扑解析目标，才能在迁移和故障切换时保持控制权。

## RANGE、HASH 与目录路由

### RANGE：范围映射

```text
tenant 1...9999      → shard-a
tenant 10000...19999 → shard-b
```

范围路由容易理解和做范围查询，但业务 ID 往往随时间增长，新数据会集中到最后一个分片；大租户还会导致范围极不均衡。

### HASH：固定哈希槽

```text
slot = hash(tenant_id) mod 1024
slot → shard
```

哈希通常分布更均匀，点查路由稳定，但租户范围查询失去局部性。不要使用语言运行时可能变化或进程随机化的默认 `hash()`；算法、字符编码、规范化和种子必须版本化，并用跨语言测试向量锁定。

业务键不应直接 `mod 当前物理节点数`。从 4 个节点扩到 5 个节点会改变大量键的目标。加入稳定逻辑槽后，扩容只需移动选定槽位：

```text
业务键 → 1024 个稳定逻辑槽 → 当前物理分片
```

### 目录路由

目录直接记录某租户或对象属于哪个 shard，适合大租户独占、数据主权、区域放置和精确迁移。代价是目录必须高可用、可缓存且有版本；创建租户时还要原子地分配归属。

实践中可组合：普通租户走哈希槽，超大租户由目录覆盖到独立分片。覆盖规则必须有唯一优先级，避免不同路由器得出不同结果。

## 均匀键值不代表负载均匀

即使每个分片租户数量相同，负载也可能差异巨大：

- 一个巨型租户贡献 40% 写入。
- 少数热点商品或直播间承受瞬时流量。
- 某租户执行重报表，主要消耗读取和临时空间。
- 行数相近，但大 JSON/附件元数据使存储差异明显。

容量规划至少同时观察：

```text
每路由键：QPS、写入字节、数据量、索引量、CPU 时间、锁等待
每槽位：上述指标的聚合与峰值
每物理分片：primary/replica 资源、连接、日志与恢复能力
```

路由键选择要用带权分布压测，而不是只统计 distinct count。超大租户可进入独立分片、分级限流或进一步按业务子键拆分，但二级分片会使该租户内查询和事务更复杂。

## 数据模型要让关联数据同地放置

如果订单按 `tenant_id` 分片，订单明细、支付尝试、幂等记录和 outbox 通常也应携带相同 `tenant_id` 并同地放置：

```sql
orders(tenant_id, order_id, ...)
order_items(tenant_id, order_id, ...)
payment_attempts(tenant_id, order_id, ...)
outbox(tenant_id, event_id, aggregate_id, ...)
```

这样创建订单、写明细和记录待发布事件仍可在一个本地数据库事务中完成。逻辑主外键也包含路由键，防止误把不同租户的相同 `order_id` 关联起来。

若两个实体经常需要原子写入，却选择了不同路由键，跨分片事务将成为日常路径，这通常说明聚合边界或分片键需要重新审视。

## 单分片查询应是默认路径

路由器可把查询分为：

| 类型 | 目标 | 风险 |
| --- | --- | --- |
| point | 一个确定分片 | 最可预测 |
| bounded fan-out | 少量明确分片 | 需要并行度与 deadline |
| scatter/gather | 所有分片 | 成本随分片数增长 |
| offline | 仓库/搜索/分析系统 | 结果可能延迟，但隔离在线负载 |

面向租户的在线接口应主要是 point query。跨所有租户的运营报表、全文搜索和全局排行更适合 CDC 构建的搜索索引或数据仓库，而不是每次请求广播生产数据库。

### scatter/gather 的尾延迟

广播查询的完成时间接近最慢分片：

```text
总延迟 ≈ 路由开销 + max(各分片延迟) + 合并开销
```

分片越多，遇到一个慢节点、连接池排队或超时的概率越高。请求必须有：

- 全局 deadline，而不是每个分片各自完整超时。
- 并发上限和每分片预算。
- 部分失败语义：整体失败、标记不完整或异步重试。
- fan-out 数量、最慢分片和合并行数指标。
- 取消传播，客户端放弃后停止剩余查询。

### 全局排序与分页

不能从每个分片取 20 行后直接拼接。全局 Top 20 至少要让每个候选分片按相同稳定键返回局部 Top K，再做 k-way merge：

```text
ORDER BY created_at DESC, order_id DESC
```

下一页游标必须保存足够的全局边界，或保存每个分片的局部游标。若只保存一个 OFFSET，新增数据、分片延迟和并发更新会造成重复或遗漏。深分页还会让每个分片丢弃大量行。

对于高频全局列表，更合理的是维护单独的汇总/搜索读模型。

## 全局唯一 ID 不等于全局唯一约束

常见 ID 方案：

- 中央序列服务：简单一致，但要处理可用性与吞吐。
- 数据库序列分段：每个节点预分配区间，减少中心依赖。
- 时间有序分布式 ID：组合时间、节点与序号，利于索引局部性。
- 随机 UUID：生成独立，但索引局部性、大小和排序语义需评估。
- 复合业务键：`(tenant_id, local_id)`，天然表达归属。

任何生成器都要定义：时钟回拨、节点 ID 重复、序号耗尽、进程重启和跨区域冲突如何处理。

即使 ID 碰撞概率很低，各分片的本地 `UNIQUE(id)` 也只能证明分片内唯一。若业务要求用户名、邮箱或幂等键全局唯一，常见方案包括：

1. 按该唯一字段本身分片，使冲突在同一节点判定。
2. 建立集中或独立分片的唯一性注册表。
3. 预留/确认协议，失败时释放并定期对账。
4. 将约束降为作用域唯一，例如租户内邮箱唯一。

注册表成为正确性关键路径，必须与业务写入处理部分成功。仅“先查不存在，再分别插入”存在并发竞态。

## 本地事务优先于分布式事务

最佳分片设计让绝大多数业务不变量在单分片本地事务内完成。跨分片一致性需求出现时，先判断它属于哪类：

- 必须同步原子提交，任何中间状态都不可见。
- 可以通过状态机和补偿最终一致。
- 其实只需可靠通知，可用本地事务 outbox。
- 只是读模型，可由 CDC 异步构建。

### 2PC/XA 提供什么

两阶段提交由事务协调器驱动：

```text
第一阶段：所有参与者执行并 PREPARE
  ↓ 全部成功
协调器持久记录 COMMIT 决定
  ↓
第二阶段：通知所有参与者 COMMIT
```

它解决多个事务资源“全提交或全回滚”的原子决定，但并不免费：

- prepared 事务可能继续持有锁和旧版本。
- 协调器故障后必须从持久日志恢复并重发决定。
- 参与者在不确定阶段可能阻塞，降低可用性。
- 超时不能擅自解释为回滚；提交响应丢失时结果可能未知。
- 所有参与者、驱动、复制、备份和故障切换都要支持恢复协议。

MySQL 8.4 的 XA 资源管理器支持限于 InnoDB，并有复制等限制；仍需外部 Transaction Manager 管理全局事务和恢复。PostgreSQL 的 `PREPARE TRANSACTION` 同样面向外部事务管理系统。官方建议没有可靠外部管理器时将 `max_prepared_transactions` 保持为 0；prepared 状态不应长期存在，因为它继续持锁并影响 VACUUM。

`postgres_fdw` 能把本地事务对应到远端事务，但当前不支持把远端事务 prepare 为两阶段提交，因此不能把“通过 FDW 写了两个远端”直接理解为跨服务器原子提交。

### Saga 与补偿

Saga 把跨分片流程拆成多个可提交的本地事务，每一步记录状态并产生下一步命令；后续失败时执行语义上的补偿。

例如跨账户转移：

```text
CREATED
  ↓ shard A 预留余额（幂等）
DEBIT_RESERVED
  ↓ shard B 入账（幂等）
CREDIT_APPLIED
  ↓ shard A 确认扣款
COMPLETED

若入账最终失败：释放 shard A 的预留 → COMPENSATED
```

补偿不是数据库 ROLLBACK。已经发出的邮件无法“回滚”，已被消费的权益可能需要反向业务动作。每一步必须：

- 使用全局 workflow ID 和幂等键。
- 持久记录状态、尝试次数、下一次重试时间和错误。
- 允许消息重复、乱序和进程崩溃后恢复。
- 验证状态转换前置条件，拒绝非法倒退。
- 对长期卡住和人工处理状态告警。

### Transactional outbox

在一个分片内，把业务更新和 outbox 事件写入同一事务；异步发布器再把事件投递到消息系统。它避免“数据库提交成功但消息没发出”的窗口。

消息仍可能重复：发布成功但标记已发前崩溃会重发。因此消费者要按 `event_id` 幂等，不能宣称 outbox 自动提供端到端 exactly-once。

## 扩容不是修改取模数

分片迁移应是显式状态机：

```text
STABLE：slot 只在 source
  ↓ 建立 target、schema 和容量检查
COPYING：按稳定水位复制历史数据
  ↓ 捕获增量/受控双写
CATCHING_UP：target 追平
  ↓ 行数、摘要、业务查询验证
VERIFIED
  ↓ 提升 topology epoch，写切换到 target
CUTOVER
  ↓ 旧路由器被 fencing，观察并处理迟到请求
CLEANUP_ELIGIBLE
```

每一步必须可重试、可观察、可暂停。不要复制完成后立刻删除 source；保留有界回退窗口，但要防止两个节点同时成为可写真源。

### 双写为什么危险

应用依次写 source 和 target，会遇到：

- source 成功、target 失败。
- 两边都成功，但响应丢失导致客户端重试。
- 并发更新以不同顺序到达两边。
- 删除、TTL、唯一约束和触发器行为不同。
- 一边 schema 已升级，另一边尚未升级。

需要幂等版本写入、变更顺序、对账与修复机制。数据库日志 CDC 往往比业务双写更容易保持顺序，但仍需处理延迟、DDL 和失败重放。

### topology epoch 与 fencing

路由目录每次切换都提升 epoch：

```text
slot 37 → shard-b, epoch 12
```

写请求携带或由服务端解析 epoch。写入入口同时校验该逻辑槽当前映射的 shard 与 epoch；旧路由器缓存的 `epoch 11` 被拒绝并刷新拓扑。不能只给整个物理 shard 设置一个版本，因为它可能同时承载许多处于不同迁移阶段的槽位。仅依赖缓存 TTL 会形成双写窗口：旧进程在 TTL 到期前仍可能写 source。

路由目录更新、节点写权限和故障切换必须协同。epoch 只是 fencing token 的模型，真正实现可以是租约、数据库角色、代理规则或版本条件写，但必须在存储写入点生效，不能只在调用方自觉检查。

## Schema 变更从一次变成 N 次

分片后，迁移必须处理不同节点处于不同 schema 版本：

1. 先发布向后兼容的扩展变更，例如新增可空列。
2. 在每个分片小批执行，记录开始、成功、失败和版本。
3. 应用同时兼容旧/新 schema，直到所有分片完成。
4. 回填按分片限速，避免所有 primary 同时产生大量日志。
5. 验证后再切读路径，最后删除旧字段或约束。

路由层应把 schema capability 纳入健康检查，不能把需要新列的请求发送到尚未升级的分片。全量“一键并行 DDL”可能同时冲击所有故障域，也会让回滚困难。

## 备份、恢复与一致性快照

每个分片各自成功备份，不代表整套系统处于同一个业务时刻。跨分片工作流、全局目录和唯一性注册表可能需要一致水位。

恢复计划要明确：

- 每个 shard 的 RPO/RTO 和恢复顺序。
- topology 目录恢复到哪个 epoch。
- CDC/outbox 从哪个位置继续，如何避免重复副作用。
- 跨分片 workflow 如何识别未知状态并对账。
- 单租户恢复是否能隔离导入，而不覆盖其他租户。
- 全局 ID/节点租约恢复后如何避免重复发号。

故障演练不能只恢复一个空闲分片，还要测试目录不可用、部分分片落后、切换期间迁移、协调器重启和备份时间不一致。

## 连接池与过载会按分片数放大

若每个应用实例为每个 shard 建 20 条连接：

```text
总连接上限 = 应用实例数 × 分片数 × 每分片池大小
```

20 个应用实例、32 个 shard、每池 10 条就是 6400 条潜在连接，还未计副本和运维。应使用惰性建池、空闲回收、全局连接预算和热点分片独立限流。

scatter 查询如果同时从每个池借一条连接，会形成乘法式放大。fan-out 并发必须在数据库池之外还有全局 semaphore/bulkhead。

## 可观测性必须带路由维度

每个请求至少关联：

- 业务路由键的安全摘要，不直接泄露敏感标识。
- logical slot、physical shard、role 和 topology epoch。
- point/fan-out 类型与目标分片数。
- 每分片连接等待、执行时间、返回行数和错误。
- 是否处于迁移、双读、回退或补偿路径。

核心告警：

- 同一键在不同路由器得到不同 shard/epoch。
- 旧 epoch 写入被拒绝突然上升。
- 某槽位 QPS、数据或锁等待显著偏斜。
- scatter 数量和合并数据量超过预算。
- 迁移 source/target 行版本或摘要不一致。
- prepared/XA 事务停留超过极短阈值。
- Saga 长期处于中间态或补偿失败。
- 分片 schema version 不一致。

## 示例说明

### 路由、fan-out 与拓扑切换

运行：

```bash
node examples/database/20-shard-routing-model.mjs
```

脚本在内存中验证：

- 稳定哈希把业务键映射到逻辑槽，再由版本化拓扑映射物理 shard。
- 缺少 `tenant_id` 的 point query 会被拒绝，而不是静默广播。
- 多 shard 局部有序结果如何做稳定 k-way merge。
- 迁移切换后旧 epoch 写入被 fencing。

### Saga 与幂等状态机

运行：

```bash
node examples/database/20-cross-shard-saga-model.mjs
```

脚本模拟跨账户转移：重复命令不会重复扣款；目标入账失败时通过持久状态进入补偿；非法状态转换被拒绝。它只解释协议模型，不连接数据库或支付系统。

### 数据库只读诊断

- `examples/database/20-mysql-distributed-transaction-diagnostics.sql` 检查版本、binlog 格式和遗留 XA prepared 事务。
- `examples/database/20-postgresql-distributed-transaction-diagnostics.sql` 检查 2PC 配置、prepared 事务与 FDW 元数据。

诊断脚本不会创建、提交或回滚分布式事务。生产环境中的未知 prepared 事务必须由事务协调器依据持久决定恢复，不能凭时间长短人工猜测提交或回滚。

## 上线检查清单

### 分片依据

- 已用容量和故障证据证明单实例成为边界。
- SQL、索引、连接、读副本、缓存、分区和归档已先评估。
- 明确分片目标是写吞吐、存储、维护窗口还是故障隔离。
- 未来至少一个容量周期的增长、成本和恢复时间有模型。

### 路由与数据模型

- 高频接口在 SQL 前即可获得路由键。
- 关联数据同地放置，核心事务保持单分片。
- 哈希算法、编码、槽位和测试向量已版本化。
- 热点按 QPS、字节和 CPU 加权评估，不只看键数量。
- 路由目录高可用、可缓存并有 epoch/fencing。

### 查询与约束

- point query 是在线默认，scatter 有明确预算与 deadline。
- 全局排序使用稳定键和正确 merge，不用无界 OFFSET。
- 全局唯一约束有可并发判定的注册或路由方案。
- 分析、搜索和全局报表有独立读模型。
- 部分分片失败时的 API 语义明确。

### 事务与工作流

- 2PC 只由可恢复的外部协调器使用并监控 prepared 状态。
- Saga 每步幂等，状态和补偿可在崩溃后恢复。
- outbox 与业务写入在同一本地事务，消费者去重。
- 未知提交、重复消息、乱序和人工处理路径经过演练。
- 金额、库存等不变量有定期跨分片对账。

### 迁移与运维

- resharding 有复制、追平、验证、cutover、fencing 和回退状态。
- source 删除与切换分离，并受观察期和审批保护。
- schema 变更兼容滚动发布，记录每 shard 版本。
- 连接总预算按应用实例 × 分片数计算。
- 备份恢复覆盖目录 epoch、CDC/outbox 和中间态工作流。

## 常见误区

### “数据量大就必须分库分表”

总行数不是充分条件。索引可控、历史可归档且单实例仍有容量时，分片可能只增加复杂度。

### “对用户 ID 取模就完成了分片”

直接对物理节点数取模让扩容重映射大量数据，也没有解决热点、迁移、故障切换和拓扑一致性。

### “哈希均匀，所以负载一定均匀”

巨型租户和热点键会让 QPS、CPU 或存储倾斜。要按真实工作量加权。

### “跨分片 SELECT 并行执行就会很快”

总延迟受最慢分片支配，还会放大连接、排序、网络和部分失败。全局报表应尽量退出在线事务库。

### “全局 ID 生成了，所以唯一性已由数据库保证”

ID 算法降低碰撞风险，本地唯一索引只检查单分片。严格全局唯一仍需要可原子判定的边界。

### “XA/2PC 是几条 SQL，打开就能用”

真正的 2PC 需要持久化协调决定、恢复协议、prepared 事务监控和参与者兼容。没有协调器时可能长期持锁并阻塞。

### “Saga 最终都会自动成功”

补偿也可能失败，外部副作用未必可逆。必须有幂等、重试上限、对账和人工处理状态。

### “双写一段时间，对上行数就能切换”

总行数相同不代表内容、版本和删除一致。要比较分片范围、摘要、业务不变量和代表性查询，并 fencing 旧写路径。

## 本课小结

- 分片用于突破单实例容量、吞吐、维护窗口或故障域边界，应先有证据并穷尽更简单方案。
- 路由键要来自接口访问模式，并让需要事务和 JOIN 的关联数据同地放置。
- 稳定逻辑槽将业务哈希与物理节点解耦；拓扑目录必须版本化并在存储侧 fencing 旧写入。
- 哈希键数量均匀不代表负载均匀，必须观察每键 QPS、字节、CPU 和热点。
- 单分片查询应是默认；scatter/gather 会放大尾延迟、连接和部分失败，全局分页需稳定合并。
- 全局 ID、全局唯一约束和 ID 到 shard 的定位是三个不同问题。
- 核心不变量应优先保留在本地事务；2PC 需要可靠协调器，Saga/outbox 需要幂等、补偿和对账。
- 扩容是复制、追平、验证、cutover、fencing 与观察组成的状态机，不是修改取模数。
- schema、连接池、备份和恢复成本都会乘以分片数，必须纳入日常平台能力。
