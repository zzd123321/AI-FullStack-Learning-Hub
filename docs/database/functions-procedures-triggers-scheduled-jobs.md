# 数据库函数、存储过程、触发器与任务调度

一个创建订单接口可能需要校验库存、写订单、扣减库存、记录审计和写入 outbox。应用可以在事务中显式执行这些 SQL，也可以调用存储过程，甚至让触发器自动补写审计。过期订单还可能由数据库定时任务取消。

这些方案都能运行，但故障边界完全不同：调用方能否看见副作用？触发器失败时谁回滚？任务重复运行会不会重复退款？数据库升级后函数计划会不会改变？生产事故中又该从哪里找到日志？

本课不是争论“逻辑应该全在数据库还是全在应用”，而是建立一套可验证的放置标准。

## 四种对象分别解决什么问题

| 对象 | 怎样触发 | 典型返回 | 适合的核心能力 | 主要风险 |
| --- | --- | --- | --- | --- |
| 函数 function | 表达式、查询或显式调用 | 标量、行或集合 | 靠近数据的确定性计算、复用查询 | 优化器语义、逐行调用、权限 |
| 过程 procedure | `CALL` 等显式命令 | OUT 参数、结果集或副作用 | 多条 SQL 的命令边界 | 事务归属、错误契约、版本耦合 |
| 触发器 trigger | 表的行/语句事件自动触发 | 通常不直接返回给调用方 | 强制本库所有写入口执行局部规则 | 隐藏副作用、递归、锁与写放大 |
| 调度任务 scheduled job | 时间或外部调度器触发 | 状态、日志、指标 | 到期处理、刷新、清理、对账 | 重叠、漏跑、重试、所有权漂移 |

数据库约束仍优先于这些对象。`NOT NULL`、`CHECK`、`UNIQUE`、外键或 exclusion constraint 能直接声明的不变量，不应先写成过程中的 `IF`。约束覆盖所有写入口、参与并发仲裁，并向调用方返回标准数据库错误。

## 逻辑放置的五个判断维度

### 是否必须与数据写入原子提交

库存不为负、幂等键唯一等不变量必须在数据库事务/约束内保证。发送邮件、调用支付 HTTP、写 Redis 则不能因为写在触发器或过程中就获得跨系统原子性；应使用 outbox、幂等消费者和补偿。

### 是否需要覆盖所有写入口

若后台脚本、多个服务和 BI 工具都能直接写表，应用层校验可能被绕过。数据库约束最可靠，必要时再评估触发器或只开放受控过程并撤销直接表写权限。

### 是否需要横向扩展与复杂依赖

数据库通常是最昂贵、最难横向扩展的共享资源。复杂 JSON 编排、外部 API、机器学习推断和长时间 CPU 运算放在数据库中，会与核心事务争抢连接、CPU 和内存，更适合应用或异步 worker。

### 是否需要多语言调用者共享

多个不同语言客户端共享一个受控命令时，存储过程可减少重复与网络往返。但数据库过程签名会成为跨团队 API，必须有版本、兼容策略、owner 和测试，不能因为它“不走 HTTP”就不当接口治理。

### 团队能否部署、测试和观测

若 CI 不能创建真实对象、迁移工具无法可靠发布、监控看不到执行时间、值班人员不会定位，数据库端逻辑的理论优势无法转化为生产可靠性。

## 函数：不仅是可复用代码，还是优化器契约

函数适合将纯计算或数据附近的表达式命名，例如货币最小单位换算、稳定规范化、受控集合查询。危险用法是在大查询的每一行调用内部再次查表的函数，形成难以从表面 SQL 看出的 N 次访问。

### PostgreSQL 的 volatility 分类

PostgreSQL 函数必须在 `VOLATILE`、`STABLE`、`IMMUTABLE` 中选择：

- `VOLATILE` 是默认值，可在相同参数下返回不同结果，优化器通常要在需要处重新求值。
- `STABLE` 承诺在一条语句内相同参数结果稳定，不能修改数据库。
- `IMMUTABLE` 承诺相同参数永远得到相同结果，优化器可能预先折叠常量结果。

这是正确性承诺，不是“性能开关”。把读取配置表、当前时间、时区或 collation 的函数错误标成 `IMMUTABLE`，可能让准备语句、表达式索引或缓存计划长期使用过期结果。

例如“把 UTC instant 转成某地区当地日期”会依赖时区规则数据库；即使参数相同，tzdata 升级也可能改变历史/未来规则，不应草率宣称永恒不变。

PostgreSQL 还允许声明并行安全性、成本和预计返回行数。错误的 `PARALLEL SAFE` 或低估成本同样会误导计划器。先保持保守声明，再用真实计划和负载证据调整。

### MySQL 函数属性

MySQL stored function 可声明 `DETERMINISTIC`/`NOT DETERMINISTIC` 和 `NO SQL`、`READS SQL DATA`、`MODIFIES SQL DATA` 等数据访问特征。这些声明与 binary logging、安全和优化有关，必须反映实际行为；服务器不会替开发者完整证明函数真的确定。

函数被嵌入 `WHERE` 或索引相关表达式时尤其谨慎。对索引列逐行包装函数可能破坏可搜索性；应优先把参数边界转换好、使用生成列/表达式索引（符合产品限制时），并查看实际执行计划。

### NULL、异常与精度是函数契约

函数要明确：

- 输入 `NULL` 返回 `NULL`、默认值还是异常？
- 除零、溢出、无效 JSON 和未知枚举如何处理？
- numeric、字符集、collation 和时间精度是否保持？
- 同一函数名重载后，驱动绑定的参数类型会解析到哪个签名？

PostgreSQL 支持函数重载；默认参数还可能造成调用歧义。对外函数应 schema-qualified，并让调用方显式绑定类型。

## 存储过程：把命令放进数据库

过程可以把多条 SQL 封装为显式命令：

```text
place_order_v1(
  tenant_id,
  order_id,
  idempotency_key,
  expected_inventory_version,
  ...
)
```

它可能带来：

- 减少应用与数据库的多次网络往返。
- 将权限收窄为 `EXECUTE`，不授予调用者直接写表。
- 让多个客户端共享同一原子命令与错误契约。
- 在数据附近集中执行集合操作。

但过程不自动解决幂等、并发与外部副作用。内部仍需要唯一约束、条件更新、锁顺序和 outbox。

### 谁拥有事务边界

最重要的问题是：事务由调用方开始和提交，还是过程内部控制？

如果应用已经开启事务，过程通常应参与该事务并把提交权留给应用。否则过程内部提交可能让调用方误以为后续失败会全部回滚。

PostgreSQL procedure 在满足特定 `CALL` 上下文时可执行事务控制，但若 `CALL` 位于显式事务块中则不能随意提交；`SECURITY DEFINER` 或带某些 `SET` 属性的 procedure 也有事务控制限制。MySQL routine 允许的事务语句和 stored function/trigger 限制又不同。不要设计一个靠“可能允许 COMMIT”工作的跨数据库抽象。

推荐在接口契约中明确：

```text
该过程不自行 COMMIT/ROLLBACK；成功返回后由调用方提交。
任何 SQLSTATE/错误码表示当前事务必须如何处理。
死锁、序列化失败和连接中断由最外层命令入口重试。
```

### 错误是接口的一部分

不要只返回 `0/1`。至少区分：

- 业务拒绝：库存不足、状态不允许。
- 幂等重放：返回第一次已提交结果。
- 可重试冲突：deadlock、serialization failure。
- 未知提交结果：连接在提交附近断开，需要用幂等键确认。
- 编程/数据错误：约束、类型、对象不存在。

应用适配层把数据库错误映射成稳定领域错误和 HTTP 状态，同时保存原始 SQLSTATE/错误号用于诊断，不能把敏感 SQL 或内部对象名直接返回给客户端。

### 过程输出也是版本化协议

客户端可能依赖结果集列名、顺序、类型、OUT 参数和多个 result set。直接 `CREATE OR REPLACE` 改返回形状可能破坏旧应用。采用 `place_order_v1`/`v2`、expand/contract 或兼容参数，统计旧版本调用量后再退役。

## 触发器：自动不等于简单

触发器在 `INSERT`、`UPDATE`、`DELETE` 等事件上自动执行。它最大的优势是调用方不必记得调用，最大的风险也是调用方看不到它。

适合候选：

- 不能用声明式约束表达、但必须覆盖本数据库所有写入口的局部派生或审计元数据。
- 兼容迁移期间短期同步新旧列，并有明确删除阶段。
- 数据库原生要求使用 trigger function 的能力。

不适合：

- 调用外部 HTTP、发送邮件或等待消息代理。
- 执行不受界限的大扫描、复杂报表或重 CPU 逻辑。
- 用隐藏级联替代清晰的业务状态机。
- 试图跨数据库获得原子性。

### 行级与语句级、BEFORE 与 AFTER

行级触发器对受影响的每一行执行，批量更新 100 万行可能调用 100 万次；语句级触发器每条语句执行一次，但如何取得变更集合取决于产品能力。`BEFORE` 常用于校验或调整新行，`AFTER` 适合依赖已完成行变化的记录。实际可用组合、transition tables 和限制应按产品版本核对。

触发器与触发它的语句处于数据库事务中：触发器报错通常会使语句失败，未处理错误可使事务进入失败状态。它不会因为“自动执行”就异步，也不会把外部副作用纳入回滚。

### 递归、触发顺序与级联

触发器写入另一张带触发器的表，会形成链；写回本表还可能递归。多个触发器的执行顺序有产品规则，不能依赖创建先后直觉。设计时绘制最大调用链并设置终止条件：

```text
orders UPDATE
  ├── audit trigger -> order_audit INSERT
  └── projection trigger -> shop_totals UPDATE
                              └── totals audit trigger -> ...
```

链越长，锁顺序、死锁、延迟和排障难度越高。不要在 trigger 中吞掉异常后假装主写成功，也不要用 session 变量随意“临时关闭触发器”而绕过业务不变量。

### 批处理和回填的写放大

一条回填 SQL 除了更新主表，还可能：

- 每行插入审计记录。
- 更新热点汇总行，扩大锁竞争。
- 产生额外 WAL/binlog 与复制延迟。
- 写 outbox，导致下游重新发送用户通知。

上线前必须在真实 schema 下测量每输入行产生多少写入，并区分技术回填是否应产生业务事件。如果确需抑制副作用，应通过版本化迁移模式、专用标记和严格权限设计，而不是手工禁用后忘记恢复。

## 调度任务：时间触发的命令仍要幂等

典型任务包括取消超时订单、刷新物化结果、归档历史、重试 outbox 和对账。调度器只负责“尝试在某个时间启动”，不保证业务动作恰好一次。

可靠任务应具备：

1. 稳定任务/业务键，例如 `expire-order:order-42`。
2. 条件写入，例如只有 `status='PENDING' AND expires_at <= now` 才取消。
3. 分批领取、租约和 fencing token，避免旧 worker 恢复后继续写。
4. 断点水位与重叠窗口，重试不漏不重。
5. 超时、停止门槛、退避和死信/人工处置。
6. 最后成功水位、扫描数、命中数、跳过数、失败数和延迟指标。

固定 cron 周期不是新鲜度承诺。上一轮耗时超过周期、主从切换、时钟变化或调度器暂停，都可能重叠或漏跑。

### MySQL Event Scheduler

MySQL 有原生 Event Scheduler，事件按一次性或周期 schedule 执行。它必须启用 `event_scheduler`，对象以 definer 权限运行，并保留创建/修改时相关环境语义。

官方文档明确指出：周期事件若在下一个周期到来前尚未结束，多个实例可能同时执行。必须由任务本身使用锁、租约或幂等条件防重叠，而不是假设调度器串行。

数据库内调度减少外部组件，但任务执行会直接争用数据库资源，输出结果集也不会自动成为可查询运行日志。需要显式运行记录、指标和告警，并在故障切换/副本拓扑中确认究竟哪台实例启用任务。

### PostgreSQL 调度边界

PostgreSQL 核心提供函数、过程和触发器，但没有 MySQL Event Scheduler 的同形内置时间调度器。常见选择是操作系统 cron、Kubernetes CronJob、工作流平台或 `pg_cron` 等扩展。

扩展不是核心语义的一部分：可用版本、运行数据库、权限、升级和高可用行为都要单独验证。外部调度器则需安全管理连接凭证、网络失败和并发实例。无论选择哪种，业务幂等与水位仍在任务自身，而不是调度产品名称中。

## 权限：INVOKER 与 DEFINER

数据库对象可能按调用者权限（invoker）或对象定义者权限（definer）执行。Definer 模式能让低权限应用通过受控入口完成有限操作，也会形成权限提升边界。

MySQL routine、trigger、event 都有 stored object access control 与 definer 生命周期问题。删除/重建账号、跨环境导入对象后，definer 可能不存在或权限漂移。

PostgreSQL `SECURITY DEFINER` 函数/过程应：

- 由专用最小权限 owner 持有，而非 superuser。
- 固定安全 `search_path`，把可信 schema 放前，把可写临时/公共 schema 谨慎处理。
- 对表、函数和 operator 使用必要的 schema qualification。
- 撤销默认 `PUBLIC EXECUTE`，只授权指定角色。
- 防止同名重载、默认参数和用户可创建对象劫持解析。

动态 SQL 的值仍参数化；对象名只能来自白名单并安全引用。调用者输入不能直接拼成表名、排序或 WHERE 片段。

## 复制、恢复与故障切换

数据库端对象属于 schema 和恢复范围：

- 备份必须包含函数、过程、触发器、事件、owner、权限和依赖扩展。
- 逻辑复制是否复制对象定义、触发器是否在订阅端触发，要按产品与复制方式验证。
- MySQL stored programs 与 binary logging 有确定性和权限限制。
- 恢复环境若自动启用任务，可能向真实下游发送 outbox 或重复清理数据。
- 故障切换后必须保证只有预期主节点运行调度任务，同时不丢失应执行水位。

恢复演练默认隔离外部网络并禁用任务，待数据与角色验证后按受控步骤启用。

## 发布与回滚

数据库端代码也要进入迁移版本库。推荐：

1. 创建新版本函数/过程或兼容 trigger，不原地破坏旧签名。
2. 在真实数据库版本上运行契约、权限、并发和执行计划测试。
3. 先部署数据库兼容对象，再部署调用新版本的应用。
4. 用 canary 观察调用量、耗时、锁等待、错误和写放大。
5. 停止旧调用者后移除旧对象和过渡触发器。

回滚应用不等于回滚数据库对象。如果新应用已写入新状态或触发器已产生 outbox，删除函数无法撤销事实。回滚计划要说明数据兼容、补偿和旧应用能否读取新写入。

`CREATE OR REPLACE` 也不是万能原子升级：对象签名、返回类型、依赖、权限和正在执行的 session 可能有额外限制。迁移前后查询目录元数据，并验证实际定义 checksum/版本。

## 测试策略

函数和过程至少测试：

- NULL、边界值、精度、collation、时区和异常映射。
- 真实 runtime role 能调用，但不能越权访问其他对象/租户。
- 并发下唯一性、锁顺序和最外层重试。
- 计划中是否发生意外逐行函数调用。
- 旧调用者与新对象版本兼容。

触发器还要测试：

- 单行、多行语句和零行命中。
- INSERT/UPDATE/DELETE 以及 upsert 的真实触发路径。
- 级联、递归终止、多个触发器顺序与失败回滚。
- 批量回填的写放大、日志量和副本延迟。
- 禁止的外部副作用确实不存在。

调度任务要在多个并发 worker、超时中断、重复运行、跨水位重启和时钟边界下测试。`examples/database/34-database-logic-boundaries.mjs` 演示同一事务中的自动审计失败会整体回滚，以及两个重叠任务依靠条件领取和稳定业务键只处理一次。

## 可观测性

目录中“对象存在”不代表对象健康。生产至少观测：

- 函数/过程调用次数、P95/P99、错误码和调用者身份。
- 触发器增加的事务耗时、每主表行产生的附加写入数。
- 锁等待、deadlock、临时空间、WAL/binlog 和副本延迟。
- 任务计划时间、实际开始时间、完成时间、业务水位和租约 owner。
- 重复领取、条件未命中、重试、死信和最后成功时间。
- 对象版本、owner/definer、权限、配置和定义摘要漂移。

慢查询聚合可能只显示外层 `CALL` 或主语句，排查时还要关联过程内部语句、触发器链和 request/job ID。日志参数脱敏，避免过程输入泄漏密钥或个人数据。

## 选择建议

| 需求 | 优先起点 |
| --- | --- |
| 单表/跨表可声明不变量 | 数据库约束 |
| 纯计算且需在 SQL 中复用 | 谨慎使用函数 |
| 多客户端共享原子数据库命令 | 版本化存储过程 |
| 所有本库写入口都必须执行的局部动作 | 小而有界的触发器 |
| 外部 API、长计算、复杂编排 | 应用服务/异步 worker |
| 到期扫描、刷新、清理 | 幂等批任务 + 明确调度器 |

默认保持逻辑显式：应用事务调用清晰 SQL、约束守住不变量、outbox 传播外部副作用。只有当覆盖所有写入口、减少往返或权限封装的收益被证明时，再引入数据库端代码。

## 常见误区

### “存储过程一次调用，所以一定更快”

它可能减少网络往返，也可能把更多 CPU、锁和逐行逻辑压到数据库。必须测端到端延迟、吞吐和数据库饱和度。

### “触发器在事务里，所以能可靠发消息”

它只能与数据库内写入原子；HTTP 或 broker 发送不能随数据库回滚。应在事务内写 outbox。

### “定时任务每次只会启动一个实例”

超时、故障切换和调度器语义都可能造成重叠。任务必须自身幂等并有租约/fencing。

### “函数标成 IMMUTABLE 会更快”

错误声明会产生错误结果或错误索引，不是安全调优。

### “数据库对象不需要 API 版本”

签名、结果集、错误码、权限和事务语义都是调用契约，同样需要兼容发布。

## 上线检查清单

- 能否先用声明式约束表达不变量？
- 为什么逻辑必须在数据库中，收益是否可测？
- 调用者、对象 owner/definer 和最小权限是否明确？
- 函数 volatility/determinism/data access 声明是否真实？
- 过程是否明确谁负责事务提交、重试和错误映射？
- 触发器链、递归终止、锁顺序和每行写放大是否已测？
- 外部副作用是否只通过 outbox 等可靠边界传播？
- 任务是否可重复、可中断、可续跑，并防止旧 worker 越权写入？
- 对象与应用是否采用 expand/contract 和版本化契约？
- 备份、复制、恢复、故障切换是否包含任务启停规则？
- 调用延迟、错误、锁、日志量、任务水位和定义漂移是否可观测？
- 是否在真实目标版本和 runtime role 下通过并发测试？

## 本课小结

- 函数、过程、触发器和调度任务具有不同触发方式与故障边界，不能互相当作语法变体。
- 数据库约束优先守护可声明不变量，数据库端代码不能替代并发控制和幂等。
- PostgreSQL volatility 与 MySQL determinism/data access 属性是正确性和运行语义承诺。
- 存储过程必须明确事务所有权、错误契约、未知提交结果和版本兼容。
- 触发器与主语句同事务执行，但隐藏的逐行副作用会放大锁、日志和回填风险。
- 调度器只保证尝试启动，任务必须依靠稳定键、条件写、租约、水位和 fencing 实现可靠重试。
- Definer 权限是敏感提升边界，需要固定解析路径、最小 owner 和显式授权。
- 数据库端代码必须纳入迁移、测试、备份、恢复、可观测性和退役流程。
