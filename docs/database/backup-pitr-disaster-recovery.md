# 备份、时间点恢复与灾难演练

数据库每天显示“备份成功”，并不能证明事故发生时能恢复。备份对象可能损坏、日志链中间缺一段、加密密钥不可用、恢复工具版本不兼容，或者恢复一套 5 TB 数据所需时间远超业务能承受的停机窗口。

真正的目标不是“拥有备份文件”，而是：**在明确的数据丢失上限和恢复时间内，把整个业务恢复到一个已验证、可安全开放流量的状态。**

本课从故障场景与 RPO/RTO 开始，解释逻辑备份、物理备份、增量日志和时间点恢复，再分别建立 MySQL binlog 与 PostgreSQL WAL 的恢复链模型，最后落实校验、加密、保留、跨区域复制和灾难演练。

## 先区分副本、备份、归档和高可用

| 能力 | 主要目标 | 会不会同步错误 | 能否回到过去 |
| --- | --- | --- | --- |
| 同步/异步副本 | 节点故障后继续服务、分担读取 | 通常会 | 通常不能任意回退 |
| 高可用切换 | 缩短节点故障停机 | 不负责阻止误操作传播 | 通常不能 |
| 备份 | 从独立副本恢复数据 | 已完成备份不会被在线误删直接改写 | 恢复到备份时刻 |
| 连续日志归档 | 保存备份后的变更序列 | 可在目标点停止重放 | 可以做 PITR |
| 业务归档 | 长期保存低频历史数据 | 取决于归档流程 | 不等同于整库恢复 |

管理员误执行错误更新时，复制会忠实地把错误传播到副本；勒索软件若能访问相同凭据或存储，也可能同时删除在线数据和可变备份。副本提高可用性，但不是备份。

分区归档保存的可能只是部分历史表，没有系统表、schema、权限、序列、事务一致性和其他业务库，也不能自动承担整库灾难恢复。

## 从故障场景定义恢复目标

至少分别分析：

- 单个数据库进程或主机故障。
- 存储损坏、数据页损坏或整个可用区丢失。
- 误删表、错误批量更新、错误 schema 迁移。
- 凭据泄露、勒索软件或备份控制面被破坏。
- 软件缺陷持续写入逻辑坏数据，数小时后才发现。
- 整个区域不可用。
- 分片拓扑、消息位置和数据库恢复点不一致。

不同故障需要不同工具。primary 主机损坏可能直接切换副本；两小时前的逻辑错误需要 PITR 到错误发生前；少数行误改可能更适合在隔离实例恢复后定向导回，而不是让整个生产回退两小时。

## RPO：最多能丢多少数据

Recovery Point Objective 是业务允许的最大数据丢失窗口。例如 RPO 5 分钟意味着灾难恢复后，最多允许缺失故障前约 5 分钟的已确认变更。

它不是“每 5 分钟备份一次”这么简单。实际恢复点受以下链路中最落后者限制：

```text
数据库产生 binlog/WAL
  ↓ 日志离开主机故障域
归档存储收到完整对象
  ↓ 清单与校验成功
恢复系统能够发现并读取
```

若日志每分钟上传，但归档队列已卡住 40 分钟，实际 RPO 暴露约为 40 分钟。指标必须从“最后一个已验证且异地可用的日志位置”计算，而不是从定时任务最后运行时间计算。

RPO 也要按业务区分。可重新计算的搜索索引允许更大 RPO；支付流水可能接近零丢失，并要求额外账务对账。不要用一个系统级口号掩盖不同数据的重要性。

## RTO：多久恢复服务

Recovery Time Objective 是从事故发生到恢复目标服务能力的时间上限。它包括：

```text
发现与确认事故
+ 决策和授权
+ 准备恢复基础设施
+ 下载/读取备份
+ 解压、解密、恢复基础数据
+ 重放日志到目标点
+ 一致性与业务校验
+ 重建缓存/搜索/连接
+ 切换流量与观察
```

只记录“数据库 restore 命令运行 20 分钟”会严重低估 RTO。大备份的对象下载带宽、KMS 限流、DNS/证书、依赖服务和人工审批都可能成为关键路径。

需要同时定义：

- **完整 RTO**：恢复全部功能与容量。
- **最低可用 RTO**：先恢复登录、查询和关键交易，报表等稍后恢复。

演练应测量每个阶段，而不是只判断最终成功或失败。

## 逻辑备份与物理备份

### 逻辑备份

逻辑备份导出 schema 与行，恢复时重新执行 DDL/DML 或使用数据库逻辑装载协议。

优点：

- 易于查看对象和选择性恢复表或数据库。
- 适合迁移、审计 schema 和小规模环境。
- 对物理文件布局依赖较少。

局限：

- 大库导出与重建索引可能很慢。
- 恢复时间随行数和约束成本增长。
- 全局对象、用户、权限、扩展和配置可能需要单独备份。
- 并行导出对象间的一致快照需要工具正确协调。
- 它本身通常不提供备份后的逐事务 PITR 日志链。

MySQL `mysqldump --single-transaction` 可为 InnoDB 获取一致快照而不锁表，但长时间一致性读会带来 MVCC 历史保留压力；非事务表、DDL 并发和全局对象还需单独评估。大型 MySQL 环境也可评估 MySQL Shell 并行 dump/load 或受支持的物理备份工具。

PostgreSQL `pg_dump`/`pg_dumpall` 是逻辑备份。官方明确指出，它们不是文件系统级 base backup，也不包含足以用于 WAL 连续回放的信息，不能作为连续归档 PITR 的物理基础备份。

### 物理备份

物理备份复制数据文件和恢复所需元数据，通常恢复大实例更快，并可作为日志重放起点。

局限：

- 与数据库产品、主版本、平台和文件布局关系更紧。
- 通常恢复整个实例/集群，选择性恢复较困难。
- 直接复制正在变化的数据目录不一定一致；必须使用数据库支持的在线备份协议、快照协调或停库方式。
- 表空间、外部密钥、配置和扩展库仍可能位于数据目录之外。

PostgreSQL `pg_basebackup` 可对运行中的集群获取 full 或 incremental base backup；增量备份不能独立恢复，必须先用 `pg_combinebackup` 与依赖链组合。备份链中的任一依赖丢失都会影响恢复，因此增量策略要换取的是存储/带宽，而不是免费简化。

## 完整恢复链：base backup + 连续日志

时间点恢复的抽象模型：

```text
Base backup B，结束位置 L100
  ↓
日志 L101 → L102 → L103 → L104 → L105
                         ↑
                   目标恢复点
```

恢复过程先还原 B，再严格按顺序重放日志，直到目标时间、日志位置、事务或命名恢复点。以下任一问题都可能让链失效：

- base backup 没记录准确起止位置和时间线。
- L103 缺失、截断、校验失败或使用错误密钥。
- 日志保留早于最新可用 base backup 的起点。
- 只复制了日志文件，却没有历史/时间线元数据。
- 恢复目标位于错误操作之后。
- 备份与日志属于不同实例、系统标识或分片。

“日志对象数量符合预期”不能证明连续；必须按数据库日志身份校验前驱、时间线和范围。

## MySQL：全量备份与 binary log PITR

MySQL 时间点恢复通常是：

1. 恢复一个已验证的 full backup。
2. 从该备份记录的 binlog 文件/位置或 GTID 边界开始。
3. 按顺序应用此后的 binary log。
4. 在错误事件之前的精确位置停止。
5. 验证后再开放流量。

MySQL 8.4 默认启用 binary logging，但不能把“默认”当成审计结果；要检查实际 `log_bin`、日志保留、归档离机延迟和备份中记录的坐标。

### 时间停止点不如事件位置精确

时间戳适合初步缩小范围，但存在时区、秒级边界、同一时刻多个事务和发现时间不准确等问题。更稳妥的流程是先在隔离环境用 `mysqlbinlog` 检查候选区间，找到错误事务的事件位置，再把恢复停止点放在它之前。

恢复命令具有真实写入效果，本课程不提供可直接针对数据库执行的日志重放管道。生产恢复应由版本化 runbook 和受限自动化生成，目标必须是全新隔离实例，且先展示将使用的 backup ID、server UUID、起止位置和对象清单。

### durability 配置影响最后一段日志

`sync_binlog` 与 `innodb_flush_log_at_trx_commit` 会影响操作系统/主机故障时已确认事务的日志和 redo 持久性边界。它们不是备份参数，却会影响故障瞬间能否从源端拿到最新日志。

即便源端持久化正确，RPO 仍取决于 binlog 是否及时复制到独立故障域。只保留在与数据目录同一块盘上的 binlog，整盘损坏时无法帮助 PITR。

### 从副本备份的边界

从 replica 备份可减轻 primary 压力，但要记录：

- 备份对应的 source GTID/binlog 进度。
- receiver 与 applier 是否都追上目标水位。
- replica 是否有延迟、过滤、错误或本地写入。
- 恢复后继续复制所需的元数据。

“备份开始时延迟为 0”不等于整个备份期间快照覆盖了期望位置，工具必须提供一致的恢复坐标。

## PostgreSQL：base backup、WAL archive 与 timeline

PostgreSQL 持续归档 PITR 需要：

1. 一个有效的物理 base backup。
2. 从 base backup 起点开始连续可用的 WAL。
3. 正确的 archive/restore 命令或备份工具。
4. 恢复目标与 `recovery.signal` 等恢复配置。
5. 足够的 timeline history 文件。

WAL 先服务于崩溃恢复；要承受整个实例/存储丢失，必须把需要的 WAL 成功归档到独立存储。官方强调应先建立并测试 WAL 归档，再制作第一份依赖它的 base backup。

### archive_command 成功语义

归档命令只有在对象真正安全保存后才能返回 0；失败必须返回非零，让 PostgreSQL 重试。一个“把文件放到临时目录后异步上传”的脚本若过早返回成功，会让数据库认为 WAL 已归档，后续回收后可能留下永久缺口。

上传应具备：

- 同名对象幂等：重复归档相同内容成功，不覆盖不同内容。
- 完整写入：先写临时对象并校验，再原子发布或使用对象存储等价机制。
- 校验和、大小、加密密钥和来源集群标识。
- 明确失败退出码、超时、重试和积压告警。

### timeline 不是可忽略的编号

PITR 或 standby promotion 会产生新的 timeline。恢复到过去后继续写入不能覆盖原历史；timeline history 描述分叉点，使系统能选择正确祖先日志链。

```text
timeline 1: ───────A────────B（原生产继续）
                    \
timeline 2:          C────D（从 A 恢复后产生的新历史）
```

同一个时间可能在不同 timeline 上对应不同数据状态。恢复清单必须记录 system identifier、timeline、base backup 和目标；不能只按文件时间选择 WAL。

### 恢复到目标后先暂停验证

恢复配置可指定目标时间、事务、LSN 或命名 restore point，并决定到达后 pause、promote 或 shutdown。面向误操作恢复时，先 pause/shutdown 验证通常比自动立即接受写入安全：一旦新实例开放写入，就产生新时间线和新的回退成本。

## 备份一致性不只包含业务表

恢复清单还应覆盖：

- schema、函数、触发器、扩展和字符集/collation。
- 数据库用户、角色、权限和密码轮换后的访问方式。
- 表空间、外部文件、证书和密钥引用。
- 配置参数、插件/共享库及其兼容版本。
- 序列/自增状态和全局 ID 节点租约。
- 分片 topology epoch、迁移状态和全局唯一注册表。
- outbox/CDC/消息消费位置与幂等记录。
- 对象存储或搜索索引等数据库外部依赖。

只恢复数据库却让消息消费者从旧位置重放，可能重复发送通知或扣减外部资源；从新位置开始又可能漏掉恢复点之后应重放的事件。数据库恢复与事件恢复必须共享清晰水位协议。

## 校验分为四层

### 1. 对象完整性

- 备份对象存在、大小合理、校验和匹配。
- 清单签名有效，所有分片/表空间/日志段齐全。
- 加密密钥仍可获得，但未与备份使用相同权限无隔离保存。

### 2. 数据库可启动

- 使用目标版本能够完成 restore/recovery。
- 日志回放无缺口，到达指定 target/timeline。
- 没有未解释的损坏、恢复警告或扩展加载失败。

### 3. 数据不变量

- 核心表行数与分区边界合理。
- 主外键、唯一性、余额守恒、库存非负等约束成立。
- 序列值不低于现有最大键。
- 跨分片账务与 workflow 状态可以对账。

### 4. 应用可用性

- 使用生产同版本应用执行代表性只读接口。
- 权限、时区、字符集和查询计划符合预期。
- 登录、订单、支付查询等关键旅程通过。
- 缓存和读模型能从明确水位安全重建。

校验查询默认只读。任何“修复”都应在确认恢复副本身份后由单独流程执行，避免验证脚本误连生产并修改数据。

## 恢复到隔离环境，而不是覆盖现场

事故发生后，优先保存证据并恢复到新环境：

```text
冻结/隔离受影响系统
  ↓ 保存日志、坐标、配置和未归档尾部
选择 backup + timeline + target
  ↓
在隔离网络恢复
  ↓ 完整性、数据库、业务、应用四层验证
决定整库切换或选择性导回
  ↓
fencing 旧 primary，切换流量
```

直接清空原数据目录并覆盖，会失去尚未归档日志、取证材料和第二次恢复机会。即使空间紧张，也应优先保存数据库最新日志尾部和恢复所需元数据；具体操作必须遵循数据库官方 runbook。

隔离环境默认禁止对外发送邮件、支付、Webhook 和消息，避免恢复后的 outbox/定时任务产生真实副作用。

## 加密、权限与不可变性

备份通常包含生产全部敏感数据，安全等级不能低于生产：

- 传输与静态加密，密钥由独立 KMS/HSM 管理。
- 最小权限：备份写入者不必拥有删除历史备份权限。
- 恢复身份与日常应用身份分离，并定期验证凭据。
- 对关键备份使用对象锁/WORM、版本控制或隔离账户降低篡改风险。
- 记录读取、删除、保留变更和恢复操作审计日志。
- 定期轮换密钥，但保证保留期内旧备份仍可解密。

不可变备份不是永不删除。它应有合规保留期、法务冻结和到期销毁流程，防止无限保存个人数据。

常见的“3-2-1”只是启发式：至少多份副本、不同介质/系统、至少一份异地或隔离。真正设计仍要从故障域、攻击者权限、RPO/RTO 和恢复带宽验证。

## 保留策略必须保证链可恢复

示例策略可能包含：

- 每日 base backup 保留 14 天。
- 每周备份保留 8 周。
- 每月备份保留 12 个月。
- 连续日志至少保留到“最老仍承诺可恢复的 base backup”之前，并留安全余量。

不能独立删除增量链中间对象。过期任务应从恢复点集合反推依赖图，只删除不再被任何受保留恢复点引用的对象。

还要监控归档增长。异常批处理、索引重建或全表更新可能产生大量 binlog/WAL，填满本地磁盘或超出上传带宽，使 RPO 突然恶化。容量按峰值日志生成率而非日均值规划。

## 分片系统的恢复是一致性编排

每个 shard 单独恢复成功，不代表业务一致。需要一个恢复 manifest：

```text
recovery_set_id
topology_epoch
每个 shard: backup_id、system/server id、timeline/GTID、target position
目录库与唯一注册表位置
消息/CDC 水位
加密密钥版本
验证结果与审批
```

如果错误事务只影响 shard-7，可单独恢复并导回修复数据；若要整体回退，必须选择一个业务上一致的全局 cut，而不是让 32 个 shard 各自恢复到“最接近 10:00”的位置。

跨分片 Saga 在目标点可能处于中间态。恢复后应由状态机重试/补偿和账务对账决定，不要直接把所有 `PROCESSING` 改为失败。

## 灾难演练如何设计

### 演练不是桌面朗读 runbook

至少定期执行：

1. 从备份目录自动选择一个符合策略的恢复点。
2. 在全新、隔离、无预置缓存的环境获取对象与密钥。
3. 恢复 base backup 并回放到指定目标。
4. 运行四层验证与应用 smoke test。
5. 记录实际数据丢失窗口和每阶段用时。
6. 销毁隔离环境及临时明文，保留审计结果。
7. 为每个偏差建立负责人和完成期限。

### 注入真实失败

- 随机缺失一个日志段，确认系统 fail closed。
- 使用已轮换密钥，验证旧版本密钥获取路径。
- 让默认区域不可用，从异地副本恢复。
- 模拟错误操作发现延迟 90 分钟，寻找正确 stop point。
- 恢复一个版本较旧的扩展或 schema，验证兼容检查。
- 在多分片恢复中加入一个慢 shard，测量最低可用服务策略。

演练不能修改生产，也不能把真实用户通知发送出去。所有凭据、网络出口和目标实例身份应由自动化硬性限制。

## 可观测性与告警

### 备份生成

- 最近成功 base/logical backup 的时间、位置、大小和耗时。
- 备份期间数据库负载、锁、复制延迟和日志增长。
- 增量链深度、依赖对象和下次 full backup 时间。

### 日志归档

- 最后已验证归档的 binlog/GTID 或 WAL/LSN。
- 数据库当前位置与归档位置的字节/时间差。
- 归档失败次数、连续失败时长、队列和本地磁盘余量。
- 日志链连续性、对象校验和与 timeline history 完整性。

### 可恢复性

- 最近一次完整恢复演练时间与结果。
- 实测 RPO/RTO 对目标的偏差。
- 备份对象、密钥和恢复身份的可用性。
- 四层验证的失败项和未关闭整改。
- 每个 shard 的恢复覆盖和全局 manifest 完整性。

“过去 24 小时备份任务 100% 成功”只是生成指标；管理层真正需要的是“最近一次随机恢复在 43 分钟内到达目标，丢失窗口 52 秒，关键旅程全部通过”。

## 上线检查清单

### 目标与范围

- 每类数据和故障场景都有明确 RPO/RTO。
- 副本、高可用、备份、PITR 和业务归档职责分离。
- 恢复范围包括 schema、权限、配置、密钥引用和外部水位。
- 最低可用服务与完整恢复目标分别定义。

### 备份与日志链

- 逻辑/物理备份选择与库规模、恢复粒度和 RTO 匹配。
- base backup 记录实例身份、时间线/GTID 和准确日志边界。
- binary log/WAL 在独立故障域连续归档并校验。
- 保留策略从可恢复点依赖反推，不会删断增量链。
- 峰值日志生成和上传/存储容量有余量。

### 安全

- 备份加密，密钥权限与备份对象隔离。
- 备份写入身份无权任意删除不可变历史。
- 恢复凭据定期验证并受到强审计。
- 法务冻结、合规保留和到期销毁同时落实。
- 隔离恢复环境禁止真实外部副作用。

### 恢复

- 默认恢复到全新隔离环境并保存事故现场。
- 恢复点使用精确日志位置/命名点复核，而非只猜时间。
- 对象、数据库、不变量和应用四层校验自动化。
- 旧 primary 在切流前被 fencing，连接和任务重新发现角色。
- CDC、outbox、消息和缓存从明确水位恢复。

### 演练

- 定期随机选择备份做完整恢复，不只校验文件存在。
- 演练包含缺日志、密钥轮换、区域故障和发现延迟。
- 实际 RPO/RTO 按阶段记录并与目标比较。
- 分片系统使用全局 recovery manifest 和业务一致 cut。
- 演练缺陷有负责人、期限并在下次演练复验。

## 常见误区

### “有副本就不需要备份”

误删、逻辑损坏和攻击通常会传播到副本。副本主要提高可用性，不保存任意历史恢复点。

### “备份文件校验和正确，所以一定能恢复”

校验和只证明对象字节未变化，不能证明日志链完整、密钥可用、版本兼容或应用不变量成立。

### “每天全量备份，所以 RPO 是一天”

若没有连续日志，实际数据丢失可能接近 24 小时；有日志归档时 RPO 取决于最后已安全归档水位，而不是 full backup 频率。

### “恢复到 09:59:59 就能避开 10:00 的误操作”

事务事件、时区和时间精度可能让目标仍包含错误。应检查日志，使用精确位置或命名恢复点。

### “归档命令先返回成功，后台慢慢上传更快”

数据库可能在真正上传前回收日志，制造永久缺口。只有对象安全持久化后才能报告成功。

### “恢复数据库后直接开放流量”

权限、序列、扩展、外部消息水位和业务不变量仍可能错误。必须通过分层验证，并 fencing 旧写入口。

### “增量备份节省空间，也一定恢复更快”

增量链需要组合与验证，链越深依赖越多。应以实测恢复时间和失败面决定 full/incremental 周期。

## 本课小结

- 备份的目标是满足可验证的 RPO/RTO，不是产生一个成功任务记录。
- 副本、高可用、备份、连续日志和业务归档解决不同问题，不能互相替代。
- PITR 由有效 base backup 与无缺口的 binlog/WAL 链组成，日志必须离开主机故障域。
- 逻辑备份便于选择性恢复，物理备份通常更适合大库快速整库恢复；选择取决于恢复目标。
- MySQL 使用 full backup 后的 binary log，PostgreSQL 使用物理 base backup、WAL archive 和 timeline history。
- 恢复目标应通过日志精确定位，并默认恢复到隔离环境保存现场。
- 校验必须覆盖对象、数据库启动、业务不变量和应用关键旅程四层。
- 加密密钥、权限、不可变性与到期销毁共同构成备份安全。
- 分片恢复需要全局 manifest、业务一致水位和中间态工作流对账。
- 只有定期注入失败的完整恢复演练，才能证明实际 RPO/RTO。
