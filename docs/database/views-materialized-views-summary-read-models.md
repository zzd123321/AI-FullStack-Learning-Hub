# 视图、物化视图、汇总表与派生读模型

一个运营接口需要返回“每个店铺今天的已支付订单数、成交金额和最近成交时间”。直接连接订单、支付和退款表再聚合，结果最接近当前事实，却可能在高并发大表上很慢。把 SQL 包成视图能统一口径，但不等于缓存结果；把结果预先存起来能加速读取，却必须回答它多久更新、重复事件会不会多算、失败后怎样重建。

这一课讨论的本质是：**读取成本、写入成本、新鲜度和维护复杂度应该放在哪里**。选择某种数据库对象之前，先把接口契约写成可度量要求：

```text
GET /shops/:id/daily-sales
P95 < 100 ms
允许结果最多落后 60 秒
退款在 5 分钟内反映
重复消息不得重复计入
读模型可在 30 分钟内从事实数据重建
```

没有新鲜度与恢复目标，“使用物化视图”只是技术名词，不是完整设计。

## 先区分事实、查询与派生状态

假设 `orders` 和 `refunds` 是权威事实，日销售额是从事实计算出的派生状态：

```text
权威事实 orders/refunds
        │
        ├── 查询时 JOIN + GROUP BY ──> 当前结果
        ├── 普通视图 ───────────────> 统一查询定义，仍在查询时计算
        ├── 物化视图 ───────────────> 某次刷新时刻的结果快照
        └── 事件/批任务 ─────────────> 应用维护的汇总表或读模型
```

接口响应不是天然的表，表也不必逐字段复制接口 JSON。事实模型适合保证写入不变量；读模型适合某种稳定查询形状。派生数据可以删除后重算，权威事实则不能因为重建方便而丢失。

## 四种能力不要混为一谈

| 能力 | 存储什么 | 读取时是否重算 | 新鲜度 | 主要代价 |
| --- | --- | --- | --- | --- |
| 普通视图 | 查询定义 | 通常是 | 接近基础表当前可见状态 | 查询成本、依赖与权限语义 |
| 生成列 | 同一行表达式的结果或定义 | 取决于 virtual/stored | 随该行写入变化 | 写成本、表达式限制 |
| PostgreSQL 物化视图 | 查询结果快照 | 否 | 到上次刷新为止 | 刷新、空间与陈旧治理 |
| 汇总表/读模型 | 应用定义的派生结果 | 否 | 由维护链路决定 | 幂等、顺序、重建与校验 |

普通视图解决“如何表达和复用查询”；后三者把某些计算从读取路径移走。预计算没有消灭成本，只是把成本移到写入、刷新、存储和一致性治理。

## 普通视图：保存查询，不保存结果

```sql
CREATE VIEW paid_order_view AS
SELECT id, shop_id, paid_at, total_cents
FROM orders
WHERE status = 'PAID';
```

普通视图可以：

- 隐藏重复的连接、筛选和字段命名，统一业务词汇。
- 只暴露调用方需要的列，配合权限形成稳定读取边界。
- 在底层 schema 演进时，暂时维持旧接口所需的列形状。

但它通常不会保存查询结果。调用视图时，优化器仍要把定义与外层条件组合并访问基础表。因此“把慢 SQL 建成视图”通常只是给慢 SQL 起了名字。是否更快必须看实际执行计划、扫描行数、排序、临时结果与等待。

### 视图能否被展开

MySQL 的视图算法包括 `MERGE`、`TEMPTABLE` 和 `UNDEFINED`。能合并时，优化器可把视图定义与外层语句合并；需要临时结果时，条件下推和可更新性会受影响。不要为了强制某个算法而跳过执行计划验证。

PostgreSQL 会通过重写系统展开普通视图，再由优化器规划。聚合、窗口函数、`DISTINCT`、集合运算、安全屏障等结构可能形成优化边界。多层嵌套视图还会让基数估计和问题定位更困难。

结论不是“视图一定没有性能损失”，而是：**视图是逻辑抽象，性能仍由展开后的真实计划决定**。

### 可更新视图不是普通表

简单的单表视图可能支持 `INSERT`、`UPDATE` 或 `DELETE`；聚合、分组、去重、多表组合等通常不可自动更新。可更新性还会随数据库产品和视图定义变化，不应由应用猜测。

`WITH CHECK OPTION` 可以要求经视图写入后的行仍满足视图条件。例如只允许通过该视图写入 `status = 'DRAFT'` 的订单。它是数据边界的一部分，不是完整授权系统；角色权限、租户隔离和审计仍需独立设计。

稳定接口视图应显式列名：

```sql
CREATE VIEW order_api_v1 (order_id, shop_id, amount_cents) AS
SELECT id, shop_id, total_cents
FROM orders
WHERE deleted_at IS NULL;
```

长期使用 `SELECT *` 会让新增列意外扩大暴露面，也可能使字段顺序、类型和依赖关系难以演进。对外契约发生破坏性变化时，可以新增 `v2`，迁移消费者后再退役旧版。

### 视图与安全边界

MySQL 视图有 `DEFINER` 和 `SQL SECURITY DEFINER/INVOKER` 语义；PostgreSQL 视图可使用默认所有者权限或 `security_invoker`，`security_barrier` 会影响谓词执行顺序。它们会改变“用谁的权限访问基础对象”，必须使用明确、最小权限的角色并测试真实运行身份。

不要把 `security_barrier` 理解为万能防泄漏开关。官方文档明确说明其保护范围有限，而且可能牺牲性能。敏感数据还需列级暴露控制、RLS/租户谓词、函数安全性与审计共同保证。

## 生成列：保存同一行的确定性推导

生成列适合把同一行的稳定表达式集中定义，例如标准化邮箱、从 JSON 提取可索引字段或计算未税金额。它不适合跨行聚合，也不应承担订单总额这种需要读取明细集合的逻辑。

虚拟生成列通常在读取时计算；存储生成列会占空间并在写入时计算。不同产品、版本对表达式、索引、函数易变性和复制都有具体限制。判断是否使用时要问：

- 表达式是否只依赖当前行，并且语义确定？
- 读取或索引收益能否覆盖写放大和存储？
- 更改表达式是否需要昂贵重写？
- 时区、collation、JSON 类型转换是否固定？

生成列保持了“基础字段变更与派生值同一行提交”的强一致性，但并没有解决跨表报表问题。

## PostgreSQL 物化视图：数据库内的结果快照

```sql
CREATE MATERIALIZED VIEW daily_shop_sales AS
SELECT shop_id,
       paid_at::date AS sale_date,
       COUNT(*) AS order_count,
       SUM(total_cents) AS revenue_cents,
       MAX(paid_at) AS latest_paid_at
FROM orders
WHERE status = 'PAID'
GROUP BY shop_id, paid_at::date;

CREATE UNIQUE INDEX daily_shop_sales_identity
ON daily_shop_sales (shop_id, sale_date);
```

物化视图保存某次执行查询所得的行，读取可以像普通关系一样使用索引，因此适合计算昂贵、读取频繁、允许陈旧的结果。但基础表提交后，物化结果不会自动同步：

```sql
REFRESH MATERIALIZED VIEW daily_shop_sales;
```

所以它表达的是“上次成功刷新时看到的事实”，不是“此刻事实”。接口若不能说明刷新水位，就无法解释用户看到的数据版本。

### 普通刷新与并发刷新

普通 `REFRESH MATERIALIZED VIEW` 替换全部内容，刷新期间可能阻塞并发读取。`CONCURRENTLY` 允许读取继续，但有严格前提：物化视图必须已经填充，并且至少有一个只由普通列组成、覆盖所有行的 `UNIQUE` 索引；表达式索引或部分唯一索引不满足该要求。

此外，同一个物化视图同一时刻只能运行一个 refresh。即使使用 `CONCURRENTLY`，刷新也不是免费的：它会消耗 CPU、I/O、临时空间和 WAL，并可能与在线流量争抢资源。

`REFRESH` 不保证保留物化视图定义中的排序。调用方需要稳定顺序时，查询仍必须显式 `ORDER BY`。

### 刷新计划必须是一等生产任务

一个可运营的刷新任务至少包含：

1. 单实例互斥，避免调度器重叠执行。
2. statement/lock timeout 与资源上限，防止无限拖延。
3. 成功水位、开始/结束时间、耗时和错误记录。
4. 失败重试与告警，但不把旧结果伪装成新结果。
5. 大刷新前容量评估，关注临时空间、WAL 和副本延迟。
6. schema 变更时验证视图、索引、刷新任务和消费者的依赖顺序。

MySQL 8.4 没有与 PostgreSQL 该能力等价的通用原生物化视图。需要类似效果时，通常显式创建汇总表，由事务、事件或批任务维护。此时刷新正确性全部属于应用设计，不能继续借用“物化视图”一词掩盖实现差异。

## 汇总表与派生读模型：更灵活，也更需要纪律

汇总表可能形如：

```text
daily_shop_sales(
  shop_id,
  sale_date,
  order_count,
  revenue_cents,
  latest_paid_at,
  source_watermark,
  model_version,
  refreshed_at
)
```

`source_watermark` 说明已处理到哪个源位置，`model_version` 说明使用哪版聚合规则，`refreshed_at` 只说明任务何时写表，三者不能互相替代。任务今天运行成功，也可能只处理到昨天的消息。

### 同步事务维护

订单写入事务中同时更新汇总行，可以获得很强的提交一致性。代价是热门店铺或日期形成热点行，写事务更长，死锁和锁等待增加；任何汇总故障还会阻塞核心下单路径。

适合规则简单、写量有限且必须立即一致的计数。更新多个汇总键时要固定锁顺序，并把 deadlock/serialization failure 的整个事务作为重试单元。

### 触发器维护

触发器也能保证同一数据库事务内更新，但逻辑隐蔽在数据库中，容易造成递归、锁顺序复杂、迁移耦合和线上写放大。使用时必须版本化、测试、监控执行时间，并确保批量回填不会无意触发巨大副作用。

### CDC 或 Outbox 增量维护

写事务只记录权威事实与 outbox，消费者异步更新读模型，可隔离读取形状并水平扩展。代价是最终一致性，以及重复、乱序、缺口、毒消息和重放治理。

“消息系统承诺至少一次”意味着消费者必须幂等。最稳妥的做法通常是在同一读模型数据库事务中：

1. 以稳定 `event_id` 写入 inbox/消费记录，唯一约束去重。
2. 只有首次插入成功才应用金额 delta。
3. 更新分区水位和汇总行。
4. 一起提交。

不能先更新汇总再异步记录去重标记，否则两步之间崩溃会重复累计。也不要用“当前总额是否相同”判断重复，因为两笔合法业务可能金额相同。

### 周期批处理

按时间窗或递增主键扫描事实数据，适合分钟级、小时级的新鲜度目标，实现相对简单。它必须处理边界重叠、迟到数据、已发生更正和断点续跑。

只查询 `updated_at > last_time` 容易因相同时间戳或事务可见性漏行。应使用稳定复合游标、重叠窗口加幂等，或数据库日志水位。事件时间决定业务归属日期，处理时间决定何时看见事件，两者必须分开保存。

## 冲正不能靠覆盖历史事件

已支付订单 `+1000` 后发生退款 `-200`，推荐记录一个新的补偿事实，而不是静默把原事件从 `1000` 改成 `800`。派生模型应用 delta 后得到 `800`，并保留审计链路。

如果源系统允许更正原行，维护器必须能识别 before/after 差异并应用 `new - old`，或周期性重算受影响分区。简单地再次读取新值并相加会多算。

乱序也不能只靠“忽略较小时间戳”解决。退款可能晚到，却属于更早的销售日。可选择：

- 按订单/租户分区保证有界顺序，并检测序号缺口。
- 让事件携带可交换的 delta，使不同顺序收敛。
- 对迟到窗口重新计算，并在窗口关闭后仍允许补偿。

所谓序号通常只在某个分区内有意义。强造全局严格递增序号可能成为写瓶颈。

## 重建能力决定读模型是否真的可靠

只会增量更新、不能从事实重建的“派生表”，最终会变成第二套无法证明来源的事实。可靠重建采用影子模型：

1. 选择源快照及其上界水位 `W`。
2. 用快照中的事实构建新版本影子表。
3. 捕获并重放 `W` 之后的增量，追到目标水位。
4. 比较行数、关键总额、分桶摘要和业务不变量。
5. 暂停或栅栏旧写者，在明确边界完成原子/受控切换。
6. 保留旧版本一段时间，观察后再安全退役。

如果先全表扫描、扫描期间又持续写入，却没有快照和水位衔接，重建结果既不是开始时状态，也不是结束时状态。影子表还应有独立的 `model_version`，避免新旧消费者同时写同一行。

校验不能只比较总行数。两行一正一负的错误可能互相抵消。应按租户、日期等业务键分桶比较 `COUNT`、`SUM`、最小/最大边界和稳定摘要，并抽样回查原始事实。

配套模型 `examples/database/32-derived-read-model.mjs` 演示了单分区序号缺口检测、事件 ID 去重、退款 delta、影子重建和校验后切换。

## 物化结果、汇总表与缓存的边界

三者都可能“让读变快”，但生命周期不同：

- 缓存通常按 key 保存可丢弃副本，通过 TTL/失效维持近似新鲜度；缓存未命中仍应能回源。
- 物化视图是数据库管理的查询结果关系，可建索引、参与 SQL，但依赖显式刷新。
- 汇总表是应用管理的读模型，可增量、可跨源、可定制，但应用必须证明正确性和可重建性。

不要让 Redis 缓存成为唯一读模型后又无法回源，也不要在物化视图外再加缓存却不定义两层陈旧度。若物化视图最多落后 5 分钟、缓存 TTL 又是 10 分钟，最坏新鲜度不是直觉中的 10 分钟，必须按刷新与填充时序计算并测试。

## 如何做选择

先测直接查询。合适的索引、分区裁剪和 SQL 改写足够时，保持单一事实来源通常最简单。确需派生时按问题选择：

| 问题 | 更可能的起点 |
| --- | --- |
| 只想统一字段与查询口径 | 普通视图 |
| 同一行确定性表达式需索引 | 生成列/表达式索引 |
| PostgreSQL 中昂贵全量查询、允许周期陈旧 | 物化视图 |
| 需秒级增量、跨源或自定义补偿 | 汇总表/事件驱动读模型 |
| 可回源、允许短期丢失的热点读取 | 缓存 |

最终决策至少量化：

- 查询频率、扫描规模、P95/P99 和并发。
- 源写入速率、热点键与可接受写放大。
- 新鲜度预算，以及强制读到自己写入的场景。
- 全量刷新/重建耗时、额外空间与故障窗口。
- 重复、乱序、删除、退款、迟到数据的语义。
- 谁拥有 schema、刷新任务、告警和事故处置。

## 新鲜度应该进入接口语义

异步读模型不应假装强一致。可在响应中提供：

```json
{
  "shopId": "shop-42",
  "saleDate": "2026-07-16",
  "revenueCents": 128000,
  "sourceWatermark": "shop-42:1842",
  "refreshedAt": "2026-07-16T08:30:12Z",
  "isStale": false
}
```

是否暴露水位取决于客户端能力，但服务端必须能观测它。刚完成写入的用户若必须立即看到结果，可在有限时间内读主事实、等待水位越过写入位置，或返回“处理中”；不能依靠固定 `sleep` 猜复制或消费已完成。

## 生产监控与告警

至少监控：

- 当前源水位与已应用水位之差，而不只是队列条数。
- 最后成功刷新时间、数据水位年龄、刷新耗时和失败次数。
- 每批读取、去重、应用、跳过、迟到和死信数量。
- 序号缺口、未知事件版本、约束冲突和负数等不变量违反。
- 汇总表/索引空间、增长率、刷新产生的 WAL 与副本延迟。
- 定期事实—派生分桶对账的差异规模。

告警应围绕用户可感知的新鲜度 SLO，例如“99% 店铺水位延迟小于 60 秒”，而不是笼统的“定时任务失败一次”。任务成功但源输入停滞，同样可能是事故。

## Schema 演进和发布顺序

视图、物化视图、生成列和汇总模型都有依赖。安全演进一般采用 expand/contract：

1. 新增兼容字段、视图版本或影子读模型。
2. 让新写者同时提供旧、新消费者需要的信息。
3. 回填/重建并校验新模型。
4. 逐步切换读取，监控结果与性能。
5. 确认没有旧消费者后再删除旧对象。

修改基础列类型前，应查询依赖对象并在测试环境执行完整迁移。直接替换视图定义不代表所有 ORM、BI 工具和长期运行语句都会自动兼容。

## 常见误区

### “视图会把结果缓存起来”

普通视图只保存定义。要证明性能改善，必须查看展开后的计划和实际指标。

### “定时任务每分钟跑，所以数据最多延迟一分钟”

任务可能重叠、排队、失败，源也可能停滞。新鲜度由源事件时间、水位、处理进度和刷新完成时间共同决定。

### “消息有唯一 ID，消费者自然幂等”

只有在去重记录与汇总更新原子提交时，唯一 ID 才形成完整幂等边界。

### “总金额一致，重建就正确”

不同租户或日期的误差可以抵消。需要分桶、不变量和抽样明细校验。

### “派生表很快，可以删掉历史事实”

删除事实会失去审计、更正规则升级和重建能力。数据生命周期应基于合规、恢复与模型需求共同设计。

## 上线检查清单

- 普通视图与物化结果是否被准确区分？
- 直接查询是否已用真实计划验证，确实需要预计算？
- 新鲜度、读取延迟、重建时间和空间目标是否量化？
- 重复、乱序、缺口、删除、退款和迟到事件是否定义语义？
- 去重与汇总更新是否在同一原子边界？
- 水位、模型版本和刷新时间是否分别记录？
- 全量刷新是否会阻塞读取或耗尽临时空间/WAL？
- 是否能用影子模型重建、分桶校验并安全切换？
- 权限身份、租户隔离和敏感列暴露是否实测？
- schema 演进是否采用兼容发布并验证所有依赖？
- 新鲜度、差异和重建能力是否有监控与演练？

## 本课小结

- 普通视图保存查询定义，提供抽象与权限边界，但不会天然缓存结果或加速查询。
- 生成列适合同一行的确定性推导，不负责跨行、跨表聚合。
- PostgreSQL 物化视图保存结果快照，读取快但需要显式刷新，并发刷新有唯一索引等前提。
- MySQL 没有同等通用原生物化视图，通常由应用维护汇总表。
- 同步、触发器、CDC 和批处理分别交换一致性、写路径成本、新鲜度与维护复杂度。
- 增量汇总必须处理稳定事件 ID、原子去重、分区顺序、缺口、补偿和迟到事件。
- 派生模型必须可从权威事实按水位影子重建，经分桶校验后再切换。
- 新鲜度是接口契约与 SLO，不是后台任务的内部细节。
