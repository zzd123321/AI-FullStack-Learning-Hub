# 评估、幻觉发现与质量回归：建立可持续的质量门禁

> 适用环境：Node.js 22 或更高版本。文中的 OpenAI Evals、Graders、Structured Outputs 与 Responses API 资料已于 2026 年 7 月 20 日按官方文档核对；平台能力和接口形态可能变化，生产项目应再次查阅官方资料。

## 1. 为什么“看起来不错”不是评估

前五课已经建立请求链路、上下文预算、结构化输出、检索和 RAG。系统现在可以回答问题并给出引用，但仍无法回答三个关键工程问题：

1. 新模型、新 Prompt 或新索引真的更好吗？
2. 好了多少，代价是什么，哪些用户反而变差？
3. 如何在发布前发现幻觉、越权、无答案失败和长尾回归？

大模型输出具有概率性。几个演示问题通过，只说明那几个样本在那几次调用中表现不错。可靠应用需要把“好”翻译为数据、标注规范、指标、阈值和发布决策。

评估不是上线前的一次考试，而是持续循环：

```text
产品目标 → 失败定义 → 评估集 → 评分器 → 基线
    ↑                                  ↓
线上事件 ← 人工复核 ← 切片分析 ← 回归门禁
```

## 2. 学习目标

完成本节后，你应该能够：

- 把业务目标拆成可观察的质量、安全、延迟和成本指标。
- 区分模型评估、组件评估、端到端评估与线上实验。
- 为 RAG 建立包含证据、权限、时间点和无答案标签的评估集。
- 识别事实错误、无证据扩写、引用幻觉、时效错误和遗漏等失败类型。
- 正确使用确定性规则、人工专家与 LLM-as-a-judge。
- 校准模型裁判，处理位置、长度、自我偏好和 grader hacking。
- 用配对实验、切片指标、置信区间和重复采样理解变化。
- 建立硬性安全门槛、非劣化门槛与持续评估流程。
- 在保护隐私并控制成本的前提下，从线上失败扩充评估集。

## 3. 先区分“模型好”与“系统好”

通用排行榜测量的是特定数据集上的模型能力，不能代表你的生产系统。用户看到的结果还受到以下因素影响：

- Prompt、Schema、工具描述和输出解析。
- 查询改写、Embedding、索引、权限过滤与重排序。
- 上下文选择、文档版本、引用校验和无答案策略。
- 超时、重试、缓存、降级和前端渲染。

因此要建立多层评估：

| 层级 | 典型问题 | 示例指标 |
| --- | --- | --- |
| 模型/Prompt | 给定相同证据能否正确回答 | 忠实度、格式通过率 |
| 检索 | 关键证据能否进入候选 | Recall@K、MRR、nDCG |
| 上下文 | 候选能否被正确组装 | 证据覆盖、重复率、截断率 |
| 生成与引用 | 声明是否受证据支持 | 声明正确率、引用完整性 |
| 系统 | 用户请求能否可靠完成 | 任务成功率、P95、成本 |
| 安全 | 是否出现不可接受行为 | 越权率、敏感信息泄漏率 |

只测最终答案会掩盖根因；只测模型会漏掉生产链路。

## 4. “幻觉”不是单一错误

为了可诊断，应把模糊的“模型胡说”拆成可标注类别：

- **与证据矛盾**：资料说 304，回答写成 200。
- **无证据扩写**：资料未说明适用范围，回答擅自加入“所有企业账户”。
- **模型记忆替代私有资料**：知识库没答案，却用通用知识猜测内部政策。
- **引用幻觉**：引用不存在、未检索、无权限或链接伪造。
- **引用错配**：来源存在，但不支持紧邻声明。
- **时效幻觉**：把旧版本或未来生效政策当作当前规则。
- **归因错误**：事实正确，却归给错误文档、人物或机构。
- **过度确定**：把“可能”“建议”“部分情况”改成“必然”。
- **遗漏**：漏掉关键条件、例外、风险或用户子问题。
- **冲突抹平**：多个来源冲突，模型自行合并为一个确定结论。

并非所有错误都是模型生成造成的。关键证据从未被召回时，首先是检索失败；引用链接映射错时，是应用验证或渲染失败。

## 5. 从业务风险定义评估目标

不要从“选什么指标”开始，而要先写失败声明：

```text
对于有权限访问且知识库存在答案的缓存问题，
系统应在前 10 个候选中召回至少一个权威证据，
最终每个事实声明都应由本次授权片段支持，
并在 P95 5 秒内完成。

对于知识库没有答案的问题，
系统不得使用模型记忆猜测内部政策，
应返回 insufficient_evidence。
```

再将它们变成可执行门槛。安全指标与普通质量指标不同：平均准确率提升不能抵消一次跨租户泄漏。

## 6. 一条评估样本应包含什么

RAG 样本不应只有 `question` 和 `referenceAnswer`：

```json
{
  "id": "rag-cache-017",
  "input": {
    "question": "缓存过期后如何使用 ETag？",
    "conversation": [],
    "tenant_id": "tenant-a",
    "allowed_group_ids": ["backend"],
    "as_of": "2026-07-20T00:00:00Z"
  },
  "expected": {
    "status": "answered",
    "relevant_chunk_ids": ["cache-etag-1", "cache-etag-2"],
    "required_facts": ["If-None-Match", "304"],
    "forbidden_facts": ["一定返回 200"],
    "forbidden_chunk_ids": ["tenant-b-secret"]
  },
  "metadata": {
    "language": "zh-CN",
    "query_type": "exact_identifier",
    "risk": "medium",
    "source": "sanitized_production_failure",
    "dataset_version": "2026-07-20.1"
  }
}
```

还应固定或可恢复文档快照、索引版本和权限策略。否则同一测试明天可能因为资料变化而得到不同“正确答案”。

## 7. 评估数据从哪里来

### 7.1 真实业务数据

经过授权和脱敏的线上请求、客服升级、用户纠错和失败追踪最接近真实分布。不能直接把生产日志复制到评估平台；先执行数据最小化、访问控制、保留期限和删除传播。

### 7.2 专家构造数据

领域专家适合编写关键政策、例外、冲突与高风险案例。专家应同时标注为什么正确、证据在哪里、哪些错误不可接受。

### 7.3 历史事故与红队数据

每个确认的严重事故都应转为回归样本，包括跨租户、Prompt Injection、旧版本命中、错误工具参数和缓存泄漏。

### 7.4 合成数据

模型可生成改写、错别字、多语言和边界案例，但合成数据容易继承生成模型偏好，不能代表真实流量，也不能由同一个模型同时生成样本、参考答案和最终评分后宣称客观。

合理组合通常是：真实流量决定主体分布，专家补高风险事实，事故补回归，合成数据扩展表达形式。

## 8. 数据集划分与污染防护

- **开发集**：开发者可以反复查看，用于快速调试。
- **验证集**：用于选择 Prompt、模型和阈值。
- **测试/保留集**：发布前评估，尽量限制查看与修改。
- **事故回归集**：确认过的严重失败，通常作为硬门禁。

同一文档的近似问题或同一会话片段不能跨集合随机泄漏，否则结果虚高。应按用户、文档、时间或问题簇分组切分。

如果 Prompt 中塞入保留集参考答案，或者不断针对测试样本手调，测试集就变成了训练集。此时应建立新的未见保留集。

## 9. 覆盖主流分布，也覆盖高风险长尾

评估集至少按以下维度切片：

- 语言、长度、错别字、结构化输入和多轮指代。
- 事实问答、精确标识符、比较、多跳、总结和无答案。
- 文档类型、版本、新旧程度与权威级别。
- 租户、角色、权限组和撤权后的访问。
- 单一证据、多个证据、冲突证据与噪声证据。
- Prompt Injection、越狱、敏感信息和工具诱导。
- 正常延迟、部分源超时、重排失败和降级路径。

总体平均值必须与切片结果同时报告。占比很小的高风险切片不应被大量简单样本稀释。

## 10. 人工标注规范

评分指南应定义每个标签并提供正例、边界例和反例。以“证据忠实度”为例：

| 标签 | 定义 |
| --- | --- |
| 通过 | 每个外部可验证声明均由授权证据直接支持 |
| 轻微问题 | 不影响结论的措辞扩大或次要遗漏 |
| 严重失败 | 核心事实矛盾、无证据扩写、引用错配或越权 |

先让多名标注者独立标注重叠样本，再检查一致率或 Cohen's kappa 等一致性指标。低一致性常说明任务本身含糊或评分规范不清，而不是简单地“标注员不认真”。

争议样本应由领域专家裁决并更新指南。记录标注者角色和规范版本，但避免在训练或普通日志中暴露个人身份。

## 11. 确定性评分器优先

能用代码准确判断的条件，不要先交给另一个模型：

- JSON/Schema 是否有效。
- 必填字段、枚举和数值范围。
- 引用 ID 是否在本次授权白名单。
- 原文摘录是否存在。
- 工具名称、参数和调用次数是否合法。
- 禁止词、敏感字段或越权片段是否出现。
- 延迟、Token、成本和重试是否超预算。

确定性评分便宜、快速、可重放，但字符串包含并不能判断语义蕴含。应明确每个自动指标能证明什么、不能证明什么。

## 12. 检索与排序指标

### Recall@K

前 K 是否覆盖相关片段。RAG 候选生成阶段通常优先关注召回，因为证据一旦丢失，生成模型无法恢复私有事实。

```text
Recall@K = 前 K 中相关片段数 / 所有相关片段数
```

### Precision@K

前 K 中相关片段比例。它反映噪声，但在相关片段数量未知或标注不完整时要谨慎解释。

### MRR

第一个相关结果排名倒数的平均值，适合用户只需要一个正确入口的任务。

```text
RR = 1 / 第一个相关结果的排名
```

### nDCG

同时考虑结果位置和多级相关性，适合“权威政策比相关讨论更有价值”的情况。

权限违规率、失效文档命中率和跨时间点错误应单独作为安全/正确性指标，不能混进平均相关度后被抵消。

## 13. 生成、忠实度与引用指标

把答案拆成最小可判断声明，再测量：

- **正确性**：声明与事实或参考标准一致。
- **忠实度/groundedness**：声明受到提供证据支持。
- **完整性**：必要事实、限定条件和子问题被覆盖。
- **相关性**：回答聚焦用户问题，没有无关扩写。
- **无答案准确率**：有答案时不误拒答，无答案时不猜测。
- **引用有效性**：ID、权限、版本和摘录有效。
- **引用正确性**：引用确实支持对应声明。
- **引用完整性**：需要证据的声明都有引用。

一个答案可以事实正确但不忠实：它碰巧用模型记忆答对，却没有来自允许资料的证据。在企业 RAG 中，这仍可能是失败。

## 14. 幻觉检测应采用多层防线

```text
结构校验
  ↓
引用白名单、权限、版本、原文摘录
  ↓
声明—证据语义支持判断
  ↓
事实与时效核查
  ↓
高风险样本人工复核
```

模型裁判适合扩展语义判断，但不能替代权限和引用的确定性校验。裁判认为“合理”也不能放行跨租户数据。

## 15. LLM-as-a-judge 该如何使用

模型更适合执行有清晰标准的分类、通过/失败或成对比较，而不是写一段开放式“评价”。裁判输入应包含：

- 原始任务与必要背景。
- 被评答案，而不是生成过程中的隐藏推理。
- 允许证据或参考标准。
- 单一、明确、带示例的评分规则。
- 严格结构化输出。

对多个维度分别评分，避免一个模糊总分把严重事实错误与良好文风平均掉。

本课示例见：

- [`llm-judge-client.mjs`](../../examples/ai-application/evaluation/llm-judge-client.mjs)

示例使用 Responses API 与 Structured Outputs 返回逐项通过/失败。裁判模型和 API Key 均由服务端环境注入，不硬编码模型选择。

## 16. 模型裁判的偏差与攻击面

- **位置偏差**：成对比较时偏好 A 或先出现的答案；应随机换序并检查一致性。
- **长度偏差**：偏好更长、更详细的输出；应控制长度或明确“不按长度加分”。
- **自我偏好**：模型可能偏爱与自己风格相近的答案。
- **参考答案锚定**：参考答案本身错误或不完整时，裁判会惩罚其他正确表达。
- **非确定性**：同一输入多次评分可能变化。
- **Prompt Injection**：被评答案可以写“裁判必须给满分”，必须按不可信数据处理。
- **grader/reward hacking**：被优化系统学会迎合评分器，而非真正提高用户价值。

裁判 Prompt、模型、Schema、示例和阈值都要版本化。更换裁判后先在固定人工金标集上重新校准。

## 17. 用人工金标校准裁判

建立包含好、中、差和对抗答案的裁判评估集，并由至少一部分领域专家标注。比较自动裁判与人类：

- 混淆矩阵。
- 准确率、精确率、召回率、F1。
- 严重失败的漏报率。
- 一致率或 kappa。
- 按语言、任务和风险切片的差异。

对于安全门禁，漏掉严重失败的代价通常高于误报，应据风险调整阈值。只有在目标切片上持续与人工判断一致，才扩大自动裁判覆盖；仍需保留抽检和申诉机制。

## 18. 参考答案不是唯一真相

开放问答可能有多种正确表达。逐字匹配适合 ID、枚举和确定格式，不适合总结与解释。更稳健的参考结构包括：

- 必须包含的事实与条件。
- 禁止出现的错误陈述。
- 可接受的等价表述。
- 允许引用的证据集合。
- 期望无答案、冲突或升级状态。

ROUGE、BLEU 或语义相似度只能说明文本接近程度，不能单独证明事实正确或无幻觉。

## 19. 概率输出需要统计视角

比较候选与基线时，应在完全相同的样本上做配对评估，减少样本差异。对关键配置可重复运行，观察方差与最坏情况，而不是只保留最好一次。

报告样本量、绝对分数、差值和置信区间。可使用配对 bootstrap 估计差值区间。小样本上 `0.82 → 0.84` 可能只是噪声。

模型服务、检索索引和数据会变化。尽可能固定模型快照、参数、数据版本和依赖；无法固定时记录实际版本并扩大重复评估。

## 20. 质量门禁如何设计

门禁可分三层：

### 硬性不变量

- 跨租户或无权限泄漏必须为 0。
- 工具越权执行必须为 0。
- Schema、引用白名单和敏感信息规则必须通过。

### 非劣化指标

- 关键切片 Recall@K 不低于基线容忍范围。
- 忠实度、引用正确性和无答案准确率不能显著下降。
- P95 延迟和平均成本不能超过预算。

### 改进目标

- 候选必须在预先声明的主要指标上获得有意义提升。
- 不能以严重切片退化换取总体均值小幅上升。

阈值应来自业务风险和历史分布，不是为了让当前版本刚好通过而倒推。

## 21. 离线评估、影子流量、灰度与 A/B

| 阶段 | 作用 | 主要风险 |
| --- | --- | --- |
| 离线回放 | 快速、可重复比较 | 数据分布与线上不一致 |
| 影子流量 | 用真实请求比较，不影响用户 | 隐私、额外成本、不能观察真实交互 |
| 小流量灰度 | 验证完整生产行为 | 少量用户会受到影响 |
| A/B 实验 | 测量真实用户价值 | 实验干扰、样本选择和安全风险 |

安全红线必须在进入线上实验前通过。A/B 不能用来“看看是否会泄漏数据”。高风险系统还需要人工审批和快速回滚。

## 22. 线上信号如何使用

引用点击、追问、重新生成、复制、人工升级和明确纠错都可提供线索，但它们带有位置、界面和用户选择偏差。用户点赞也不等于答案事实正确。

建议把线上信号用于：

1. 发现新的失败簇和流量漂移。
2. 抽样进入人工复核。
3. 经脱敏和标注后加入评估集。
4. 监控离线指标与真实业务指标是否脱节。

不能把未经审核的用户反馈直接当金标，攻击者可能故意污染反馈数据。

## 23. 失败定位与修复闭环

```text
输入/路由错误？
  → 独立问题和来源选择是否正确
检索错误？
  → 关键证据是否进入候选
上下文错误？
  → 证据是否被去重、截断或冲突覆盖
生成错误？
  → 给定正确上下文仍是否不忠实
验证错误？
  → 伪造引用为何未被拦截
展示错误？
  → 后端可信元数据是否映射错误
```

修复后把最小复现加入对应组件测试和端到端回归集。不要只加一句更长的 Prompt；有些问题应修索引、权限、Schema、工具或产品交互。

## 24. 持续评估与发布流程

```text
代码/Prompt/模型/索引变更
          ↓
快速确定性测试与事故回归集
          ↓
完整离线评估 + 配对差值 + 切片报告
          ↓
人工抽检高风险与裁判分歧样本
          ↓
影子或小流量灰度
          ↓
线上监控、回滚、失败入库
```

每次运行保存不可变制品：候选提交、模型与 Prompt 版本、数据集版本、索引快照、每样本输出、评分器版本、聚合结果、耗时和成本。没有逐样本结果的单个总分无法审计。

## 25. 成本、限流与隐私

大规模评估会放大模型调用量。应设置：

- 每次运行的样本、Token、金额和最长耗时上限。
- 并发信号量、请求超时与对 `429/5xx` 的有限重试。
- 可恢复检查点，避免失败后重跑全部样本。
- 基于输入、模型、Prompt 和评分器版本的安全缓存。
- 先运行便宜的确定性评分，再对剩余样本调用模型裁判。

评估集通常比普通日志更敏感，因为它集中保存失败、用户数据和内部证据。必须执行脱敏、加密、最小权限、审计、保留和删除策略。外部裁判服务接收的是被评答案与证据，同样要完成供应商与数据地域评估。

## 26. 可运行的确定性评测器

示例文件：

- `examples/ai-application/evaluation/eval-cases.jsonl`
- [`deterministic-eval.mjs`](../../examples/ai-application/evaluation/deterministic-eval.mjs)

运行：

```bash
node examples/ai-application/evaluation/deterministic-eval.mjs
```

它会计算 Recall、MRR、必要事实覆盖、无答案准确率、引用有效性、P95 延迟与平均成本，并验证伪造引用会触发硬门禁。字符串事实覆盖只是教学用确定性代理，不替代语义忠实度评估。

模型裁判请求 dry-run：

```bash
DRY_RUN=1 OPENAI_EVAL_MODEL='由部署配置指定的受支持模型 ID' \
node examples/ai-application/evaluation/llm-judge-client.mjs
```

真实执行前由服务端密钥管理系统注入 `OPENAI_API_KEY`。不要把真实密钥写入源码、数据集或 CI 日志。

## 27. 常见误区

| 误区 | 后果 | 修正方向 |
| --- | --- | --- |
| 凭感觉试几个问题 | 无法量化或复现 | 版本化评估集与成功标准 |
| 只看总体平均分 | 小语种和高风险切片被掩盖 | 分层、分切片报告 |
| 用一个模型包办生成与评分 | 共享偏差、自我偏好 | 确定性规则、人工校准、多样裁判 |
| 把语义相似度当事实正确 | 文本相近仍可能矛盾 | 声明—证据忠实度判断 |
| 每次修改测试集直到通过 | 门禁失去独立性 | 开发、验证与保留集分离 |
| 线上点赞就是金标 | 行为信号有偏且可污染 | 抽样人工标注后入库 |
| 新版本均值更高就发布 | 严重安全或长尾回归 | 硬门槛与非劣化门槛 |
| 裁判给高分就可信 | 裁判也会漂移和被攻击 | 人工金标校准与持续抽检 |

## 28. 上线检查清单

- [ ] 业务失败已拆成组件级、端到端和安全目标。
- [ ] 评估集有权限、时点、证据、无答案和切片标签。
- [ ] 开发、验证、保留和事故回归集相互隔离。
- [ ] 真实数据已经授权、脱敏，并支持删除传播。
- [ ] 能确定判断的规则全部使用代码评分。
- [ ] 模型裁判使用明确 rubric 和结构化输出。
- [ ] 裁判已在人工金标上校准并报告严重失败漏报率。
- [ ] 配对比较报告样本量、差值、方差或置信区间。
- [ ] 总体与语言、权限、风险、无答案等切片同时报告。
- [ ] 安全不变量不会被质量均值抵消。
- [ ] 每次运行保存逐样本结果和所有版本标识。
- [ ] 评估调用有超时、限流、重试、预算和隐私控制。
- [ ] 灰度具备实时监控、停止条件和快速回滚。
- [ ] 新的线上事故会进入长期回归集。
