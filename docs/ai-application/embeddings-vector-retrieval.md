# Embedding、文档切分、向量检索与重排序：构建可靠的召回层

> 适用环境：Node.js 22 或更高版本。文中的 OpenAI Embeddings API 参数与模型限制已于 2026 年 7 月 13 日按官方文档核对；模型、限额和 SDK 形态可能变化，生产项目应再次查阅供应商文档。

## 1. 学习目标

完成本节后，你应该能够：

- 解释 Embedding、相似度、向量索引、召回与重排序各自解决的问题。
- 建立一条可重放、可增量更新、可删除的文档摄取链路。
- 根据标题、段落、代码、表格与 Token 预算切分文档，而不是机械截断字符。
- 正确处理模型、维度、归一化和向量迁移契约。
- 组合语义检索、关键词检索、元数据过滤与重排序。
- 把租户和访问权限约束放在服务端检索边界内。
- 用 Recall@K、MRR、nDCG、延迟和成本评价检索系统。
- 处理 Embedding 请求的批量、超时、限流、重试、隐私与成本问题。

## 2. Embedding 不是“让模型记住文档”

Embedding 模型把一段文本映射为固定长度的数值数组：

```text
“缓存过期后使用 ETag 重新验证”
                    ↓ Embedding 模型
[0.018, -0.027, 0.004, ...]
```

在模型训练形成的表示空间里，语义或用法相近的文本通常更接近。应用可以据此搜索与问题相近的文档片段、聚类、去重或分类。

但向量不是：

- 原文的无损压缩，无法可靠地从向量还原全文。
- 事实数据库，距离接近不代表内容真实或适用于当前用户。
- 权限系统，相似度不会自动隔离租户或机密文档。
- 生成模型，它只负责表示和比较，不负责组织最终回答。
- 校准后的概率，`0.82` 的余弦相似度不等于“82% 正确”。

检索系统的目标也不是“找到看起来最像的一段”，而是：在权限、时效、延迟和成本约束下，尽可能召回能支持后续任务的证据。

## 3. 一条完整的检索链路

向量数据库只是链路中间的一环。生产系统通常分为离线摄取和在线查询两条路径。

```text
离线摄取
数据源 → 解析 → 清洗/去重 → 结构化切分 → 权限与版本元数据
     → 批量 Embedding → 向量索引 + 关键词索引 → 发布索引版本

在线查询
身份/租户 → 查询规范化 → 查询 Embedding ─┐
                         关键词检索 ───────┼→ 候选融合 → 重排序
              权限/时间/来源过滤 ─────────┘          ↓
                                             去重与上下文组装
```

这条链路包含三个不同目标：

1. **候选生成**追求高召回：相关证据不要在第一步就丢失。
2. **重排序**追求高精度：把真正有用的片段推到有限上下文的前面。
3. **上下文组装**追求可用性：去重、保留来源、控制 Token，并让后续模型能引用。

把三者揉成一次 `topK: 5` 向量查询，会让问题很难定位：究竟是没有索引、召回方式不匹配、权限过滤过严，还是排序质量不足？

## 4. 先定义文档与片段的数据契约

不要只把 `{ text, embedding }` 写入数据库。一个可运营的片段至少需要：

```js
const chunk = {
  chunkId: 'sha256:...',          // 稳定、可幂等写入
  documentId: 'handbook/http-cache',
  documentVersion: '2026-07-13T08:00:00Z',
  sourceUri: 'docs://handbook/http-cache',
  title: 'HTTP 缓存重新验证',
  headingPath: ['HTTP 缓存', 'ETag 与条件请求'],
  text: '缓存过期后，客户端可发送 If-None-Match ...',
  startOffset: 428,
  endOffset: 716,
  contentHash: 'sha256:...',
  tenantId: 'tenant-a',
  visibility: 'team',
  allowedGroupIds: ['backend'],
  language: 'zh-CN',
  updatedAt: '2026-07-13T08:00:00Z',
  embeddingModel: 'text-embedding-3-small',
  embeddingDimensions: 1536,
  pipelineVersion: 'chunker-v3'
}
```

这些字段分别服务于：

- `documentId + documentVersion`：判断新旧版本，删除已失效片段。
- `sourceUri + offsets + headingPath`：生成引用并回到原文。
- `contentHash`：跳过未变化片段，复用 Embedding。
- 权限字段：在搜索执行期间进行强制过滤。
- 模型、维度与流水线版本：定位召回回归并支持索引迁移。

`chunkId` 应由稳定输入生成，例如 `documentId + version + headingPath + offsets + contentHash`。重复执行摄取任务必须得到相同 ID，这样重试不会制造重复记录。

## 5. 文档解析：先恢复结构，再考虑长度

PDF、网页、Markdown、Word 和工单导出的原始文本往往带有噪声。摄取阶段应先恢复语义结构：

- 标题层级、段落、列表、引用块。
- 代码块的语言、函数或类边界。
- 表格的表头与行列关系。
- 页码、章节和原始字符位置。
- 附件、脚注、图片 OCR 与来源链接。

常见清洗包括移除重复页眉页脚、导航栏、版权尾注和不可见控制字符；但不能为了“干净”而删除否定词、单位、代码缩进或表头。原始文件应保留，以便新解析器上线时重放。

解析失败不能悄悄变成空文档。至少记录来源 ID、解析器版本、失败类型、页数或字节数，并让摄取作业进入可重试或人工检查队列。

## 6. 文档切分的核心权衡

### 6.1 片段太小与太大分别会怎样

片段过小：

- 指代词、标题和前提丢失，片段本身无法解释。
- 一个事实散落在多个向量中，检索结果碎片化。
- 片段数量、索引空间和请求数量增加。

片段过大：

- 同一向量混合多个主题，表示被稀释。
- 命中后占用大量生成上下文，挤掉其他证据。
- 更新一小段文字也需要重新嵌入整个大块。
- 重排序模型可能因输入过长而截断关键句。

不存在适用于所有语料的固定大小。起点可以按段落和标题合并，再用 Token 上限约束；最终大小应由真实查询集上的 Recall@K 和端到端回答质量决定。

### 6.2 按结构递归切分

推荐优先级通常是：

1. 文档或章节边界。
2. 标题、小节、段落和列表边界。
3. 句子边界。
4. 最后才按 Token 硬切。

每个片段应携带祖先标题，必要时把标题文本加入待嵌入内容：

```text
HTTP 缓存 > ETag 与条件请求

缓存过期后，客户端可发送 If-None-Match ...
```

标题为短段落提供主题，但不要把整份目录和冗长路径复制到每个片段，避免公共词淹没真正内容。

### 6.3 重叠不是默认答案

滑动窗口重叠能降低边界切断风险，也会：

- 增加 Embedding 与存储成本。
- 让搜索结果出现几乎相同的片段。
- 挤占候选集和最终上下文。

优先保留自然结构。只有文本连续、句子可能跨边界或评估显示边界漏召回时才加入少量重叠，并在结果阶段按 `documentId`、位置和内容相似度去重。

### 6.4 代码与表格需要专门策略

代码应按文件、类、函数或方法切分，并附带符号名、语言、导入关系和文件路径。错误码、API 路径和标识符依赖精确词法匹配，因此通常要同时建立关键词索引。

表格不能只逐行转文本而丢掉表头。可以把每一行表示为：

```text
表：模型限额
列：模型=text-embedding-3-small；最大输入=8192 tokens；默认维度=1536
```

特别宽的表可按行组拆分，但每块都要重复必要的表名和表头。数值比较、聚合等问题更适合结构化查询工具，不应强行依赖向量搜索。

### 6.5 父子片段

一种实用设计是：用较小的“子片段”进行精确检索，用较大的“父片段”提供生成上下文。

```text
父片段：完整小节（用于阅读）
├── 子片段 A（用于检索）
├── 子片段 B（用于检索）
└── 子片段 C（用于检索）
```

子片段命中后回取父片段或相邻片段，但仍要受总 Token 预算约束。不要无条件把整篇文档加入上下文。

## 7. Token 计数与模型输入限制

字符数不等于 Token 数，中英文、代码、JSON 和不同 tokenizer 的比例不同。切分器应使用与目标模型相符的 tokenizer 或服务端认可的计数方式，并保留安全余量。

当前 OpenAI 官方模型页列出 `text-embedding-3-small` 和 `text-embedding-3-large` 的单条输入上限均为 8192 Token。Embeddings API 参考还说明：

- 单次请求的数组最多包含 2048 条输入。
- 一个请求中所有输入合计最多 300,000 Token。
- 输入不能为空字符串。

这些是 API 上限，不是推荐批次大小。生产批次还应考虑请求体大小、超时时间、失败重试成本和服务等级；通常采用远小于上限的批次，并用队列限制并发。

## 8. Embedding 模型是一份索引契约

当前官方文档给出的默认维度是：

| 模型 | 默认维度 | 单条最大输入 |
| --- | ---: | ---: |
| `text-embedding-3-small` | 1536 | 8192 Token |
| `text-embedding-3-large` | 3072 | 8192 Token |

`text-embedding-3` 系列支持通过请求中的 `dimensions` 参数缩短向量。更少维度通常降低存储、内存和距离计算成本，但可能影响检索质量，必须在自己的评估集上比较。

以下配置必须视为一个整体并写入索引元数据：

```text
模型名称 + 模型版本策略 + dimensions + 文本预处理版本 + 距离度量
```

查询向量与文档向量必须使用兼容的模型、相同维度和相同预处理。不能把不同模型产生的向量放进同一个空间直接比较。

### 模型迁移

模型或维度变化通常意味着重新嵌入全部有效片段。安全迁移流程是：

1. 创建新索引版本，不覆盖旧索引。
2. 对现有语料回填新向量，持续同步增量更新。
3. 用固定评估集比较质量、延迟和成本。
4. 小流量影子查询或灰度切换。
5. 验证删除、权限和引用一致性后切换别名。
6. 保留回滚窗口，再按保留策略清理旧索引。

双写期间要监控两边的文档数、缺失率和更新时间水位，不能只检查作业是否“成功”。

## 9. 相似度：余弦、点积与欧氏距离

余弦相似度比较方向：

```text
cosine(a, b) = (a · b) / (||a|| × ||b||)
```

OpenAI 官方指南建议使用余弦相似度，并说明其 Embedding 已归一化为长度 1。因此在这些向量上：

- 点积可以更快地计算余弦相似度。
- 余弦距离和欧氏距离会得到相同的排序。

但不要把这个性质外推到其他模型。切换供应商或自托管模型时，要核对是否归一化、索引使用的是距离还是相似度，以及升序还是降序。

阈值也不能照搬。相似度分布会随模型、语言、片段长度和领域变化；应从标注数据上选择阈值，并为“没有足够相关证据”保留明确分支。

## 10. 调用 Embeddings API

一个最小请求如下：

```js
const response = await fetch('https://api.openai.com/v1/embeddings', {
  method: 'POST',
  headers: {
    Authorization: `Bearer ${process.env.OPENAI_API_KEY}`,
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    model: process.env.OPENAI_EMBEDDING_MODEL ?? 'text-embedding-3-small',
    input: ['第一个片段', '第二个片段'],
    encoding_format: 'float'
  })
})
```

密钥只能存在于服务端环境或密钥管理系统，不能写进源代码、浏览器包、移动端应用和日志。完整实现见：

- [`openai-embedding-client.mjs`](../../examples/ai-application/vector-retrieval/openai-embedding-client.mjs)

该示例还处理了：

- 有界批次与空输入校验。
- 可选 `dimensions` 参数。
- 15 秒超时。
- 对网络错误、`429` 和部分 `5xx` 的有限指数退避。
- 按响应 `index` 恢复输入顺序。
- 响应数量、维度和有限数值校验。
- 仅记录数量、模型、维度和 Token 用量，不记录原文与密钥。

API 返回的 `data` 项带有 `index`。批量处理时应按 `index` 建立输入与输出映射，而不是依赖数组恰好按预期排序。还要记录 `usage.prompt_tokens`，用于成本核算和异常监控。

### 超时与重试

Embedding 请求没有外部写副作用，但重试仍可能重复计费。建议：

- 为连接和整体请求设超时，超时后中止网络请求。
- 仅重试网络暂时性错误、`429`、`500`、`502`、`503`、`504`。
- 尊重服务端 `Retry-After`，否则采用带随机抖动的指数退避。
- 对 `400`、`401`、`403` 等确定性错误直接失败并告警。
- 限制最大次数和总耗时；批量作业把失败批次放入可追踪的重试队列。

如果一次批次超时，客户端通常不知道服务端是否已经计费，所以“越多重试越可靠”并不成立。

## 11. 批处理、缓存与增量索引

大规模摄取不应逐片段串行请求：

1. 计算规范化文本和 `contentHash`。
2. 查找 `(contentHash, model, dimensions, preprocessingVersion)` 缓存。
3. 仅将未命中项组成有界批次。
4. 用并发信号量控制在途请求。
5. 保存向量后再更新索引发布水位。

不要只用文本哈希作为缓存键；模型或预处理变化时，旧向量不再兼容。也不要把失败批次标记为已索引。

文档更新时，应对比新旧 `chunkId/contentHash`：新增和变更片段重新嵌入，未变化片段复用，已删除片段从向量索引、关键词索引、缓存和引用存储中同步删除。

## 12. 精确搜索与近似最近邻索引

少量向量可以逐个计算相似度，结果精确、实现简单。数据规模增长后，通常使用近似最近邻（ANN）索引，以少量召回损失换取显著的延迟和内存收益。

常见类别包括 HNSW 和 IVF，但参数含义依数据库而异。一般要关注：

- 建索引时间和增量写入能力。
- 查询时可调的搜索深度。
- 召回率、P95/P99 延迟和内存。
- 元数据过滤如何与 ANN 协作。
- 删除、更新、压缩和备份行为。

调参必须在代表性数据规模上比较 ANN 结果与精确搜索的真值。只观察平均延迟，可能把小语种、长查询或强过滤条件下的召回损失藏起来。

## 13. 权限过滤必须进入检索边界

错误模式是先全库搜索，再从返回结果中删除无权片段：

```text
全库 top 10 → 删除 8 条无权限结果 → 只剩 2 条
```

这既可能泄漏标题、分数和时序信息，也让合法候选被无权限结果挤出 top K。正确做法是从可信会话解析租户和主体，在搜索条件中强制加入：

```text
tenantId = 当前租户
AND (visibility = public
     OR ownerId = 当前用户
     OR allowedGroupIds 与当前用户组有交集)
AND validFrom <= now
AND (validTo 为空 OR validTo > now)
```

过滤条件不能由 Prompt 或用户传来的 JSON 决定。数据库若采用“先 ANN 后过滤”，要验证强过滤下是否仍能获得足够候选；必要时使用支持过滤感知搜索的索引、分区索引或提高内部候选数。

离线评估数据也不能绕过权限规则。测试中至少包含跨租户同名文档、权限撤销、组成员变更和已删除文档。

## 14. 为什么只做向量搜索不够

语义向量擅长处理改写和近义表达，例如：

```text
查询：缓存过期后怎样避免重新下载整个响应？
文档：客户端携带 ETag 进行条件请求，未修改时服务端返回 304。
```

关键词搜索更擅长精确符号：

- `If-None-Match`、`ERR_CONNECTION_RESET`。
- 产品型号、订单号、版本号和人名。
- 代码标识符、API 路径和缩写。
- 用户明确加引号的短语。

因此工程系统常采用混合检索：分别获得向量和词法排序，再融合候选。

### Reciprocal Rank Fusion

RRF 不要求两种分数处于同一量纲，只使用名次：

```text
RRF(d) = Σ 1 / (k + rank_i(d))
```

`k` 是平滑常数，常以 60 作为起点，但仍应评估。某文档同时在两个列表靠前时会得到更高融合分。

不要直接把 BM25 分数和余弦相似度相加：它们的范围和分布不同，换一批查询就可能失衡。若使用加权归一化，应在稳定评估集上拟合并持续监控分布漂移。

完整的本地演示见：

- [`local-retrieval.mjs`](../../examples/ai-application/vector-retrieval/local-retrieval.mjs)

示例使用透明的规则型“玩具向量”展示链路，因此无需 API Key。它适合理解算法和测试权限，不代表生产语义质量。

## 15. 候选数、top K 与“无答案”

候选生成阶段可以取较大的 K，例如向量和关键词各取几十条，再让重排序选出少量片段。数值取决于语料和延迟预算，不能照抄。

K 太小会在第一阶段永久丢失相关片段；K 太大会增加数据库读取、重排序成本和尾延迟。最终放进生成上下文的片段数还受 Token 预算、重复度和来源覆盖限制。

系统应允许返回零条结果。可组合以下信号判断证据不足：

- 最佳候选低于按评估集确定的阈值。
- 重排序分数低，或前几名都只命中宽泛主题。
- 多种召回方式没有一致候选。
- 关键过滤条件使有效候选为空。

“总要返回五条”会把最不相关的文档也包装成证据，增加后续幻觉风险。

## 16. 重排序：把高召回候选变成高精度上下文

向量搜索通常分别编码查询和文档，速度快，适合大规模召回。重排序模型同时查看查询与候选文本，能判断更细的对应关系，但计算更贵，因此只处理初筛后的候选。

一条典型链路是：

```text
向量 top 50 + 关键词 top 50
          ↓ 合并、权限过滤、去重
      约 60 个候选
          ↓ 重排序模型
      前 8～12 个候选
          ↓ 多样性与 Token 预算
      3～6 个上下文片段
```

重排序输入要保留标题和必要上下文，但遵守模型长度限制。发送给第三方重排序服务前，还要确认数据地域、保留策略和隐私协议。

### MMR：相关性与多样性

如果多个候选是同一段落的重叠副本，可以使用 Maximal Marginal Relevance：

```text
MMR(d) = λ × relevance(d, query)
       - (1 - λ) × max similarity(d, selected)
```

`λ` 越大越重视相关性，越小越鼓励多样性。MMR 不能替代语义重排序，但能减少重复内容霸占上下文。另一种简单办法是为每个 `documentId` 或章节设置数量上限。

重排序后仍要执行最终权限复核，因为候选从召回到生成之间，用户权限或文档状态可能变化。

## 17. 上下文组装为下一课留下正确接口

检索结果不要变成无来源的字符串数组。建议输出：

```js
{
  queryId: 'q_...',
  indexVersion: 'knowledge-v12',
  noAnswer: false,
  results: [
    {
      chunkId: '...',
      documentId: '...',
      sourceUri: '...',
      title: '...',
      text: '...',
      startOffset: 428,
      endOffset: 716,
      retrieval: {
        vectorRank: 2,
        lexicalRank: 1,
        fusionScore: 0.0325,
        rerankScore: 0.91
      }
    }
  ]
}
```

面向模型的文本与面向引用验证的元数据应同时存在。应用只能允许模型引用这次结果中的 `chunkId`，最终展示引用时再从可信元数据解析标题和链接，不能让模型自行编造 URL。

## 18. 离线评估：先测召回，再测回答

只凭几个手工问题观察结果，很容易过拟合。应建立版本化评估集：

```json
{
  "query": "缓存过期后如何用 ETag 验证？",
  "relevantChunkIds": ["cache-etag-1", "cache-etag-2"],
  "requiredFacts": ["If-None-Match", "304"],
  "tenantId": "tenant-a",
  "allowedGroupIds": ["backend"],
  "asOf": "2026-07-13T00:00:00Z",
  "tags": ["zh", "exact-token", "multi-hop"]
}
```

### 关键指标

- **Recall@K**：所有相关查询中，前 K 是否召回了至少一个或足够多相关片段。
- **Precision@K**：前 K 中相关片段的比例。
- **MRR**：第一个相关结果排名倒数的平均值，适合“找到一个正确入口”的任务。
- **nDCG@K**：同时考虑多级相关性和排序位置。
- **无答案准确率**：不存在相关文档时，系统是否正确返回空。
- **权限违规率**：任何越权结果都应视为严重失败，而不是普通质量误差。

指标要按查询类型切片：中文/英文、精确标识符/自然语言、短/长查询、新旧文档、强权限过滤、表格、代码和多跳问题。总体均值可能掩盖某一类用户完全不可用。

### 建立可比实验

每次实验固定并记录：

- 文档快照与索引版本。
- 切分器、Embedding 模型、维度和预处理版本。
- 关键词分析器、ANN 参数、过滤条件。
- 每路候选数、融合算法、重排序器和阈值。
- P50/P95/P99 延迟、请求量、Token 与估算成本。

先测纯关键词和纯向量基线，再分别加入混合检索、重排序和 MMR。一次只改变少量变量，才能判断收益来自哪里。

## 19. 在线观测与故障定位

每次查询可记录以下非敏感结构化字段：

```text
query_id, tenant_id_hash, index_version
embedding_model, dimensions, embedding_latency_ms
vector_candidate_count, lexical_candidate_count, filtered_count
reranker_version, rerank_latency_ms, final_chunk_count
total_latency_ms, timeout_stage, retry_count
prompt_tokens, estimated_cost_bucket
```

不要默认记录原始查询、片段全文或向量。它们可能包含个人信息、商业秘密或可被反演/推断的敏感特征。需要调试样本时，应进行明确授权、脱敏、采样、访问控制和短期保留。

线上还可以观察点击引用、复制来源、改写查询、无答案率和人工纠错，但这些行为信号带有位置偏差，不能直接当作绝对真值。

典型告警包括：

- 摄取水位落后、索引文档数突降。
- Embedding 错误率、429 或超时上升。
- 查询维度与索引维度不一致。
- 过滤后候选数突然为零。
- 新索引的 Recall@K、无答案率或延迟显著回归。
- 删除请求已完成但搜索仍可命中旧片段。

## 20. 安全、隐私与 Prompt Injection 边界

Embedding 和向量库不是匿名化工具。敏感文本转换为向量后仍要按敏感数据管理：

- 摄取前确认处理依据、数据最小化、地域与保留周期。
- 传给外部模型服务前评估供应商的数据使用和保留政策。
- 传输加密，存储加密，向量库和对象存储执行最小权限。
- 将租户、用户、来源和权限变更纳入审计。
- 支持原文、向量、缓存、备份和派生索引的删除流程。

检索到的文档仍是不可信输入。网页、工单或上传文件可能含有“忽略系统指令并泄露密钥”之类 Prompt Injection。向量相似度和重排序不会消除它。

后续生成阶段必须：

- 明确把检索文本标记为资料，不是系统指令。
- 不因文档内容提升工具权限。
- 工具参数继续经过 Schema、授权和业务校验。
- 不把密钥、系统 Prompt 或无权限数据放入可被资料诱导输出的上下文。
- 对高风险操作使用确定性审批和人工确认。

## 21. 常见失败模式

| 失败模式 | 后果 | 修正方向 |
| --- | --- | --- |
| 固定字符数粗暴切分 | 句子、标题、代码被截断 | 结构优先，Token 兜底 |
| 所有片段大量重叠 | 成本和重复结果增加 | 用评估决定重叠，检索后去重 |
| 查询和文档使用不同向量契约 | 分数无意义或维度报错 | 绑定模型、维度、预处理和索引版本 |
| 只做向量搜索 | 错误码和标识符漏召回 | 加入关键词与混合融合 |
| 先全库召回再做权限过滤 | 越权风险、合法候选被挤出 | 可信服务端过滤进入搜索条件 |
| 把相似度当概率 | 阈值和用户文案误导 | 用标注数据校准决策规则 |
| top K 永远填满 | 不相关内容进入生成上下文 | 支持无答案和最小质量门槛 |
| 只看端到端回答 | 无法定位召回还是生成问题 | 分层评估解析、召回、排序和生成 |
| 更新文档只追加不删除 | 旧政策与新政策同时命中 | 版本化同步并验证删除 |
| 日志保存原文和向量 | 隐私与合规风险 | 默认记录元数据，受控采样调试 |

## 22. 本地运行示例

无需密钥的检索实验：

```bash
node examples/ai-application/vector-retrieval/local-retrieval.mjs
```

它会展示权限过滤后的语义排名、关键词排名、RRF 融合和 MMR 去重结果，并用断言阻止跨租户片段泄漏。

只检查真实 API 请求体，不发网络请求：

```bash
DRY_RUN=1 node examples/ai-application/vector-retrieval/openai-embedding-client.mjs
```

发起真实请求时，从服务端环境注入密钥：

```bash
OPENAI_EMBEDDING_MODEL='text-embedding-3-small' \
node examples/ai-application/vector-retrieval/openai-embedding-client.mjs
```

执行前应由部署环境或密钥管理系统设置 `OPENAI_API_KEY`，不要把真实值写进命令历史、源码或配置样例。

可用 `OPENAI_EMBEDDING_DIMENSIONS` 指定正整数维度。修改维度后必须使用匹配的新索引，不能与既有向量混用。

## 23. 上线检查清单

- [ ] 原始文档可追溯、解析失败可见，摄取任务可重放。
- [ ] 切分保留标题、来源、偏移、版本和稳定 ID。
- [ ] Token 计数符合 Embedding 模型，批次远离硬上限。
- [ ] 模型、维度、预处理和距离度量绑定到索引版本。
- [ ] 密钥只在服务端，调用设置超时、限流、有界重试和成本监控。
- [ ] 权限过滤来自可信身份，并在召回阶段强制执行。
- [ ] 精确词法与语义召回按真实查询需要融合。
- [ ] 候选生成、重排序、去重和上下文组装分层可观测。
- [ ] 支持无答案，不把最低分候选强塞入上下文。
- [ ] 评估集覆盖语言、标识符、权限、时效、代码和表格。
- [ ] 文档变更与删除能传播到所有派生索引和缓存。
- [ ] 日志、向量、原文和第三方重排序满足隐私与保留要求。
- [ ] 检索资料仍按不可信输入处理，不能改变工具授权。
