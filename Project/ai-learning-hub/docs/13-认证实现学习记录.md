# 步骤 4：认证实现学习记录

> 状态：已完成并通过本地 MySQL 端到端验证。

## 本步目标

实现一个最小但完整的认证闭环：用户注册后得到短期 access token 和浏览器不可读的 refresh cookie；access token 用于访问 `/me`；refresh cookie 用于续期和登出时撤销。前端只在内存保留 access token，刷新页面后应通过 `/auth/refresh` 恢复会话（恢复交互将在下一步补全）。

## 本步实现

| 部分 | 做法 | 原因 |
| --- | --- | --- |
| 密码 | Spring Security 的 Argon2id 编码器 | 数据库只保存不可逆哈希。 |
| access token | JJWT 0.13、HS256、默认 15 分钟 | Bearer 请求无需服务端查询；暴露窗口较短。JJWT 官方发布页列出 0.13.0 为最新版本。 |
| refresh token | 32 字节随机值、SHA-256 哈希入库、默认 30 天 | 数据库泄露不能直接得到可用 token；可撤销、可轮换。 |
| Cookie | `HttpOnly`、`SameSite=Strict`、`Path=/api/v1/auth` | JavaScript 无法读取；仅认证端点携带。生产 HTTPS 时将 `JWT_SECURE_COOKIE=true`。 |
| 授权 | Spring Security 无状态过滤器 | `/me` 必须有有效 Bearer token；健康检查和认证端点公开。 |

## 文件导览

- `identity/`：用户、角色、刷新令牌实体和 Repository；它们对应 V1 的身份表。
- `security/`：JWT 配置、签发/解析服务、Bearer 过滤器和 Security 配置。
- `auth/`：输入校验、注册/登录/刷新/登出 API、当前用户 API、统一错误响应。
- `web/src/App.vue`：最小注册和登录页面，用于看见前后端契约如何被使用；它不是最终产品界面。

## 如何验证

在两个终端分别启动数据库和服务：

```bash
cd Project/ai-learning-hub
docker compose --env-file infra/.env -f infra/compose.yaml up -d

cd server
set -a; . ../infra/.env; set +a
DB_PASSWORD="$MYSQL_PASSWORD" mvn spring-boot:run
```

另开终端：

```bash
cd Project/ai-learning-hub/web
npm run dev
```

打开 `http://localhost:5174`，注册一个新邮箱后点击“验证当前身份”，再点击“登出”。也可用 `curl` 保存 Cookie：

```bash
curl -i -c cookies.txt -H 'Content-Type: application/json' \
  -d '{"email":"learner@example.com","password":"LearningHub2026!","displayName":"学习者"}' \
  http://localhost:8080/api/v1/auth/register
```

不要提交 `cookies.txt`、`.env` 或任何真实密钥。

## 边界与风险

本步不做邮箱验证、密码找回、限流、设备管理、管理角色界面或前端自动刷新。CSRF 在本地同域 API 场景依靠 `SameSite=Strict` 控制；如果将来改为跨站前后端部署，必须重新引入明确的 CSRF 方案，不能沿用当前配置。

JWT 密钥的默认值只为本地学习便捷；任何共享、测试或生产环境都必须用 `JWT_SECRET` 覆盖。数据库中仍可能保留用户和 refresh token 哈希，因此开发测试结束后可用 Compose 的受控数据卷清理流程重置本地数据。

## 验收标准

- 注册返回 201、access token 和 `HttpOnly` refresh Cookie；重复邮箱返回 409。
- 登录错误返回统一 401，不区分邮箱和密码。
- `/me` 无 token 返回 401，有效 access token 返回当前用户。
- refresh 会撤销旧 token 并设置新 Cookie；logout 撤销当前 token 并清除 Cookie。
- 前端构建、后端测试和至少一次浏览器或 curl 端到端验证通过。

## 手把手代码导览

建议按“数据→令牌→HTTP”阅读，不要一开始就钻进 Security 配置：

1. 打开 `identity/UserEntity.java`。构造方法为新用户自动加入 `LEARNER`；密码字段保存的是 `passwordHash`，不是明文密码。
2. 打开 `security/JwtService.java`。`createAccessToken` 把用户 ID 放到 `subject`、把角色放进 `roles` claim；`parseAccessToken` 用同一签名密钥验证 JWT。它只负责令牌，不处理 HTTP Header。
3. 打开 `security/JwtAuthenticationFilter.java`。它从 `Authorization: Bearer ...` 取 token，解析角色并建立 Spring Security 的 `Authentication`。从这一刻起，Controller 可以通过 `@AuthenticationPrincipal Long userId` 得到当前用户。
4. 打开 `auth/AuthService.java`。注册/登录先用 Argon2 校验密码，再生成随机 refresh token；数据库只写 `SHA-256` 哈希，响应只经 HttpOnly Cookie 发送原 token。刷新时先创建新记录，再撤销旧记录，称为轮换。
5. 最后读 `SecurityConfiguration.java`。它定义哪些路径公开、哪些默认需要登录，并把 JWT 过滤器放在用户名密码过滤器之前。尝试无 token 请求 `/me`，预期 401；携带登录返回的 access token 重试，预期 200。
