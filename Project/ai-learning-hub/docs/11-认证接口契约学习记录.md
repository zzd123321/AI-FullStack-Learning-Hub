# 步骤 2：认证接口契约学习记录

> 状态：已完成契约设计；尚未实现认证代码或数据库表。

## 本步学习目标

先把前端和后端之间的“约定”写清楚，再写控制器和页面。OpenAPI 是机器可读的接口说明：它约束 URL、请求字段、响应字段、状态码和认证方式，使 Vue 与 Spring Boot 可以独立实现而不靠猜测联调。

## 已确定的简化范围

- 使用邮箱、密码和显示名注册；密码长度为 8–72 个字符。
- 使用邮箱和密码登录；失败时统一提示“邮箱或密码错误”，避免暴露账号是否存在。
- 不做邮箱验证、密码找回、邮件服务、OAuth 社交登录或多因素认证。
- `LEARNER` 是注册用户的默认角色；管理角色只能由后续系统管理员授予。

## 为什么仍保留 refresh/logout 契约

访问令牌很短（建议 15 分钟），泄露后的暴露时间较小；刷新令牌可用于续期和主动登出。即使第一版简单，也不应把一个长寿命 JWT 长期存入浏览器。刷新令牌的 Cookie 属性、数据库存储和撤销逻辑将在认证实现步骤中完成。

## 阅读与验证

1. 打开 `contracts/openapi.yaml`，从 `/auth/register` 到 `/me` 阅读每个端点。
2. 对照 `components/schemas`：请求对象是客户端提交的数据，响应对象是服务器承诺返回的数据。
3. 注意 `401` 与 `409` 的区别：前者是认证失败，后者是邮箱已存在的资源冲突。
4. 本步可用任意 OpenAPI 编辑器检查 YAML；暂不需要启动数据库或编写 Java 认证代码。

## 已确认的令牌策略

采用短期 access token（建议 15 分钟）+ HttpOnly refresh Cookie。前者只放在前端内存中，通过 `Authorization: Bearer` 请求 API；后者由浏览器以 Cookie 发送，服务端仅保存其 SHA-256 哈希并可撤销。这个策略避免长期 JWT 留在 JavaScript 可读存储中，也让登出和风险失效成为可能。

## 范围、非目标、风险与验收

**范围**：定义认证 API 契约与错误语义。

**非目标**：不实现 Spring Security、密码哈希、数据库迁移、登录页面或邮件能力。

**风险**：若前端把 refresh token 放进 JavaScript 可读存储，XSS 风险会扩大；若登录错误区分邮箱不存在和密码错误，会泄露账号枚举信息。

**验收**：契约含注册、登录、刷新、登出、当前用户五个端点；注册/登录字段、成功响应、401/409 和 Bearer 安全方案可在文档中找到；范围明确排除邮箱相关功能。

## 手把手代码导览

本步的“代码”是 YAML 契约。打开 `contracts/openapi.yaml`，按下面顺序读：

1. 从 `paths: /auth/register: post:` 开始。`requestBody` 描述浏览器要提交的 JSON，`201` 描述创建成功而不是普通读取成功。
2. 跳到 `components.schemas.RegisterRequest`。`required: [email, password, displayName]` 是前后端共同承诺；后端用 Bean Validation 再做一次强制校验，前端的 `required` 只改善体验。
3. 查看 `AuthResponse`。它把 `accessToken`、有效秒数和当前用户放入 JSON；refresh token 故意不在 schema 中，因为它由 `Set-Cookie` 且带 `HttpOnly` 返回。
4. 查看 `/me` 的 `security: - bearerAuth: []` 与文件末尾的 `securitySchemes`。这就是“这个端点要带 `Authorization: Bearer <token>`”的机器可读表达。
5. 尝试故意把某个字段名改错，再对照后端 `AuthController` 的 `RegisterRequest`。你会发现契约与 Java record 的字段必须完全一致；因此先写契约能更早发现沟通错误。
