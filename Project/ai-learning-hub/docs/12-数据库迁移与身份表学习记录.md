# 步骤 3：数据库迁移与身份表学习记录

> 状态：已完成。Flyway 已在本地 MySQL 8.4 执行 V1、V2，并通过二次启动验证。

## 本步学习目标

理解为什么数据库结构不能依靠 JPA 自动生成。Flyway 把每次结构变化作为不可修改、按顺序执行的 SQL 文件保存；空数据库和团队成员的数据库都会得到相同结构，并能记录当前版本。

## 本步交付

| 文件 | 内容 |
| --- | --- |
| `V1__create_identity_tables.sql` | `users`、`roles`、`user_roles`、`refresh_tokens` 与外键、唯一约束、索引。 |
| `V2__seed_system_roles.sql` | 三个固定角色：学习者、内容管理员、系统管理员。 |
| `application.yaml` | MySQL 数据源、Flyway 启用、JPA 只做 schema 校验，不自动建表。 |

## 表为何这样设计

- `users.email` 唯一且使用大小写不敏感的 MySQL 排序规则，防止 `A@x.com` 与 `a@x.com` 成为两个账号；应用层仍要在写入前转小写。
- 角色使用连接表 `user_roles`，所以一个人可以同时是学习者和内容管理员。
- `refresh_tokens` 只保存随机令牌的 SHA-256 哈希，不保存原始令牌；泄露数据库时不会直接获得可用 Cookie。
- `replaced_by_id` 形成刷新轮换链。旧令牌被替换后立即撤销；第一版会拒绝旧令牌的重复使用。若未来需要把它升级为“检测重放后撤销该用户全部会话”，应增加专门的服务和测试，而不是假设当前实现已具备该能力。
- `ddl-auto: validate` 表示 JPA 只能验证实体和表是否匹配，不能悄悄修改生产结构；真正的结构变化只能通过新 Flyway 迁移完成。

## 如何验证

1. 在项目根目录复制 `infra/.env.example` 到 `infra/.env`，设置仅本机使用的密码。
2. 运行 `docker compose --env-file infra/.env -f infra/compose.yaml up -d` 并等待 MySQL healthy。
3. 运行 `cd server && DB_PASSWORD=<与 .env 相同> mvn spring-boot:run`。Flyway 会先执行 V1、V2，随后服务启动。
4. 再次停止并启动服务。Flyway 应报告数据库已是最新版本，而不是再次执行迁移。
5. 可进入容器执行 `SHOW TABLES;`，应看到四张身份表和 `flyway_schema_history`。

## 本步边界

本步不创建 JPA 实体、不存密码、不签 JWT，也不创建路线、任务、笔记等业务表。身份相关的表先行，是因为下一步登录代码需要稳定的数据边界。

## 风险与验收

**风险**：修改已执行的 Flyway 文件会导致校验失败；把真实密码提交到 `.env` 会泄露本地凭据；把原始 refresh token 存入表会扩大泄露风险。

**验收**：空 MySQL 能执行 V1、V2；二次启动不重复执行；角色种子只有三条；表结构包含邮箱唯一约束、用户—角色外键和 refresh token 哈希唯一约束。完成后再进入 Spring Security 与 JWT 的代码实现。

认证实体接入后，JPA 校验发现 `refresh_tokens.token_hash` 是 SQL 的固定长度 `CHAR(64)`，而 Java 默认推断为 `VARCHAR(64)`。实体已显式声明 `CHAR(64)`；没有修改已执行的 V1 文件，也不需要额外迁移。这是“迁移不可变、应用映射适配事实表结构”的实例。

## 手把手代码导览

1. 打开 `V1__create_identity_tables.sql`，先读 `users`：`email` 的唯一索引是数据库最后一道重复注册防线。再读 `user_roles`，它用 `user_id + role_code` 复合主键表达“同一角色不能重复授予同一人”。
2. 接着读 `refresh_tokens`：`token_hash` 有唯一约束，而原始 token 从不落库；`revoked_at` 为非空时代表不可再用。
3. 打开 `V2__seed_system_roles.sql`。这是数据迁移而非 Java `if` 语句，优点是每个新数据库都会有相同角色集合。
4. 打开 `server/src/main/resources/application.yaml`。`spring.flyway.enabled: true` 开启迁移；`ddl-auto: validate` 让 Hibernate 只能验证，不得自动改表。
5. 最后看 `RefreshTokenEntity` 的 `@Column(columnDefinition = "CHAR(64)")`。它解释了为什么 Java 映射要适配已经执行的 SQL，而不是反过来偷偷改 V1。
