# M0：工程基线实现记录

> 状态：核心链路已验证（2026-07-21）；MySQL 容器留待步骤 3 的数据库迁移练习。

## 本步学习目标

理解一个最小全栈应用的组成：Vue 只负责界面；Spring Boot 提供 API；MySQL 是后续数据的唯一事实来源；浏览器通过同域路径 `/api` 调用后端，而不直接处理跨域。

## 本步创建的内容

| 目录/文件 | 用途 |
| --- | --- |
| `web/` | Vite + Vue 3 + TypeScript 的最小页面；启动后请求 `/api/v1/health`。 |
| `server/` | Spring Boot 最小服务，提供 `/api/v1/health` 和 Actuator 健康检查。 |
| `contracts/openapi.yaml` | 健康检查的第一份接口契约。 |
| `infra/compose.yaml` | 仅运行 MySQL 8.4 的本地开发依赖。 |
| `infra/.env.example` | 本地变量样例，真实 `.env` 不提交。 |

## 为什么先做健康检查

健康检查没有业务规则，因此能单独验证端口、前端代理、后端启动和接口契约的基本链路。若这一链路不通，直接做登录会把网络、框架和认证问题混在一起，难以学习和定位。

## 按顺序运行

1. 进入项目目录：`cd Project/ai-learning-hub`。
2. 复制本地变量：`cp infra/.env.example infra/.env`，并把其中演示密码改成仅本机使用的值。
3. 启动 MySQL：`docker compose --env-file infra/.env -f infra/compose.yaml up -d`。
4. 新开终端启动后端：`cd server && mvn test && mvn spring-boot:run`。
5. 再新开终端启动前端：`cd web && npm install && npm run dev`。
6. 打开 Vite 输出的地址（默认 `http://localhost:5174`），页面状态应显示 `ai-learning-hub-api: UP`。

## 验收与排错

- 后端测试通过，且 `curl http://localhost:8080/api/v1/health` 返回 `status: UP`。
- 浏览器页面显示 API 正常，证明 Vite 的 `/api` 代理有效。
- `docker compose ... ps` 显示 MySQL healthy；本步后端尚未连接数据库，这是有意保留到步骤 3。
- 如果 Maven 在本机 JDK 25/26 下运行异常，安装并显式使用 Java 21；项目的编译目标固定为 Java 21。

## 本步边界与下一步

本步没有用户表、数据库迁移、JWT、路由、组件库或任何业务页面。完成上述验证后，下一步不是立刻写登录页面，而是扩展 OpenAPI，学习认证接口的请求/响应契约；随后在步骤 3 引入 Flyway 和用户/角色表。

## 手把手代码导览

按这个顺序打开文件并亲自改动一次，再运行命令观察结果：

1. 看 `server/src/main/java/com/ailearninghub/health/HealthController.java`。`@RestController` 表示方法的返回对象会自动转 JSON；`@GetMapping("/api/v1/health")` 把 Java 方法绑定到 HTTP GET。先只返回固定的 `status` 和 `service`，避免把数据库问题带入第一步。
2. 看 `web/vite.config.ts`。其中的 `server.proxy` 把浏览器请求的 `/api` 转发到 `http://localhost:8080`。因此 Vue 代码只写 `fetch('/api/v1/health')`，不写后端主机名。
3. 看 `web/src/App.vue` 的 `onMounted`。它在组件首次显示后调用健康接口；成功时更新 `apiStatus`，失败时显示“API 未启动”。`ref` 是 Vue 的响应式容器，赋值 `apiStatus.value = ...` 后模板会更新。
4. 在后端运行 `mvn spring-boot:run`，前端运行 `npm run dev`；然后暂时停止后端并刷新浏览器，观察页面从 UP 变为失败提示。这个练习说明前端状态来自真实 API，而不是写死的文字。
