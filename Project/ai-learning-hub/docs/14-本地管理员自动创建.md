# 本地管理员自动创建

## 目的

内容管理与用户管理需要管理员角色。学习项目不应把固定管理员密码写进 Java 文件或 Git，因此采用“仅本地 `.env` 提供凭据时才创建”的方式。

## 配置

在 `infra/.env` 增加以下三项，并使用只限本机的强密码：

```dotenv
ADMIN_EMAIL=admin@local.test
ADMIN_PASSWORD=请替换为本机强密码
ADMIN_DISPLAY_NAME=本地管理员
```

启动后端时，应用会：

1. 若邮箱不存在，创建用户并使用 Argon2id 哈希密码；
2. 若邮箱已存在，保留原密码，不覆盖用户资料；
3. 确保该用户拥有 `CONTENT_ADMIN` 和 `SYSTEM_ADMIN`；
4. 仅记录管理员邮箱，不记录密码或哈希。

若 `ADMIN_EMAIL` 与 `ADMIN_PASSWORD` 均为空，应用不创建任何管理员；若只设置其中一项，应用会停止启动，避免意外的半配置状态。

## 验收

- 设置三项后重启后端，日志显示管理员邮箱且不含密码。
- 使用该邮箱登录后，`/me` 中的角色包含 `LEARNER`、`CONTENT_ADMIN`、`SYSTEM_ADMIN`。
- 再次启动不会创建重复用户，也不会重置现有密码。

## 本次验证结果

使用一次性环境变量启动后端，已自动创建 `bootstrap-admin@example.test`，并验证：

- 登录返回 200；
- `/me` 返回 `LEARNER`、`CONTENT_ADMIN`、`SYSTEM_ADMIN`；
- 数据库中该邮箱只有一条用户记录。

该临时测试管理员仍在本地 MySQL 中，但没有写入 `infra/.env`、源代码或文档中的真实密码。你自己的管理员应通过 `.env` 的 `ADMIN_*` 配置创建；无需使用该测试邮箱。

## 边界与风险

这是开发环境引导能力，不是生产用户管理方案。生产环境应采用受控的首次管理员流程与密钥管理，不能把 `ADMIN_PASSWORD` 作为普通环境变量长期保存。

## 手把手代码导览

1. 打开 `bootstrap/BootstrapAdminProperties.java`。`@ConfigurationProperties` 会把环境变量 `ADMIN_EMAIL`、`ADMIN_PASSWORD`、`ADMIN_DISPLAY_NAME` 绑定到 Java 配置对象；属性类不写任何密码逻辑。
2. 打开 `bootstrap/BootstrapAdminConfiguration.java`。`ApplicationRunner` 表示应用成功启动、Flyway 已完成后再运行引导逻辑，因此不会在表还不存在时创建用户。
3. 代码先判断邮箱和密码是否同时为空；只设置一个时直接抛出异常。这是防止“以为配置成功，其实没有可登录密码”的 fail-fast 规则。
4. 若用户不存在，使用已配置的 `PasswordEncoder` 将密码转换为 Argon2 哈希后创建 `UserEntity`；若已存在，则不覆盖密码。随后只调用 `grantRole(CONTENT_ADMIN)` 和 `grantRole(SYSTEM_ADMIN)` 补齐权限。
5. 在 `.env` 写入自己的 `ADMIN_*` 值后启动服务，再调用 `/api/v1/me` 验证角色。不要把 `.env` 内容复制到聊天、Git 或实现记录。
