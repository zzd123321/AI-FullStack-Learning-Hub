# 项目章程与边界

## 1. 目标与成功定义

AI Learning Hub 的目标是让学习者把“想学什么—今天做什么—完成了什么—掌握得怎样”留在同一个可持续使用的系统中；让管理员可以维护结构化学习内容，而不依赖开发人员改页面。

第一版成功并不等同于堆积功能。它应当让一名注册用户能选择或浏览学习路线、安排当天任务、完成学习、写下可检索笔记，并在仪表盘看到与记录一致的学习数据；管理员能安全地维护内容。上线前以以下结果验收：核心路径可端到端通过、未授权访问被拒绝、关键指标可由原始记录复算。

## 2. 利益相关者与角色

| 角色 | 关心的问题 | 第一版能力 |
| --- | --- | --- |
| 学习者 `LEARNER` | 下一步学什么、是否坚持、内容是否沉淀 | 浏览、计划、完成、笔记、仪表盘 |
| 内容管理员 `CONTENT_ADMIN` | 内容结构与发布是否可控 | 路线、课程、知识点的草稿/发布管理 |
| 系统管理员 `SYSTEM_ADMIN` | 用户安全、权限与运行状态 | 用户、角色、账号状态管理 |

一个用户可同时拥有多个角色。最小权限优先：内容管理员不自动拥有系统管理员权限。

## 3. 第一版范围

1. 注册、登录、登出、JWT 认证、角色授权与账号禁用。
2. 学习路线、课程、知识点的层级管理、发布与浏览。
3. 学习任务、每日计划、完成记录、学习进度与手工学习时长记录。
4. 富文本以外的轻量 Markdown 笔记，以及用户私有标签。
5. 管理端的用户、内容管理和基础审计可追踪性。
6. 学习仪表盘：学习时长、连续学习天数、完成率、掌握度及其统计口径。

## 4. 非目标

- 不实现 AI 个性化计划、RAG 问答、自动总结、复习卡片或 Agent；仅预留可扩展的数据和边界。
- 不将既有 VitePress 文档迁移、抓取或同步到本平台；第一版内容由管理员独立录入。
- 不处理支付订阅、公开社区、消息通知、直播/视频托管、原生 App、离线优先和多语言。

## 5. 现有学习站边界与目录决策

现有仓库是可构建的 VitePress 学习知识库，`docs/` 中存放站点内容，根 `package.json` 的 `build` 脚本以它为输入。AI Learning Hub 是一套有账号、数据和服务端状态的应用，技术形态不同。

因此，项目整体放在仓库根目录的 `Project/ai-learning-hub/`，而不是 `docs/projects/`：前者不会进入导航、不会触发学习站的内容规范，也让未来的运行时工程能独立演进。备选方案是把规划页面加入 VitePress，优点是在线阅读，代价是需要修改导航并混淆“学习材料”和“产品决策”；本阶段不采用。

## 6. 约束、假设与风险

**约束**：前端使用 Vue 3 + TypeScript；后端使用 Java；本阶段仅产出文档；所有内容保留在当前仓库内且不覆盖现有文件。

**待验证假设**：目标用户先按个人学习者设计；一个知识点的完成可被任务记录驱动；手工时长在第一版是可信且可接受的；管理员数量较少，内容协同不需要工作流。

**风险与缓解**：掌握度若被误解为考试分数，将破坏信任，故必须标注为规则评分并展示构成；JWT 一旦泄露难以即时失效，故采用短访问令牌与可撤销刷新令牌；指标统计在数据量增长后可能变慢，故以原始事件为准、汇总可重建。

## 7. 验收方式与决策门

本章程评审通过的条件是：角色定义无歧义、第一版范围得到确认、AI 被明确排除、目录策略得到认可。以下事项未确认前不得开始业务实现：

1. 内容是否允许多个管理员协同编辑；
2. 任务是否仅由用户创建，还是管理员也可下发模板；
3. 是否接受第一版仅记录手工学习时长。

已确认：第一版仅支持邮箱和密码注册/登录；不提供邮箱验证、密码找回或邮件服务。该决定降低了学习项目的外部依赖，但不适合直接作为生产账号恢复方案。
