# ADR-028：订单事实源保持 MySQL，暂不引入第二写库

- 状态：Accepted
- 日期：2026-07-15
- Owner：订单平台组
- 复审触发器：预测峰值达到单节点实测安全容量的 70%，或出现当前版本无法表达的硬约束

## 上下文

订单接口当前使用托管 MySQL 8.4。未来 18 个月峰值预计从 2,800 QPS 增长到 6,000 QPS。团队希望评估迁移 PostgreSQL、使用 Redis 保存订单，或保持现状。

关键写事务同时写订单、支付幂等记录和 outbox；要求唯一约束、跨表事务、PITR、RPO 60 秒、RTO 15 分钟和指定地域部署。列表接口按 `(tenant_id, status, created_at, id)` 过滤与游标分页。

## 候选

1. 保持托管 MySQL 8.4，并完成索引与连接治理。
2. 迁移至托管 PostgreSQL 18。
3. 使用 Redis 作为订单事实源。

候选比较的是本次 PoC 的精确托管规格、驱动和参数，不代表产品的普遍排名。

## 硬约束结果

| 约束 | MySQL 方案 | PostgreSQL 方案 | Redis 方案 |
| --- | --- | --- | --- |
| 所需关系约束与跨表事务 | 通过 | 通过 | 不通过当前数据模型门禁 |
| 指定地域与加密 | 通过 | 通过 | 通过 |
| PITR/RPO 恢复演练 | 11 分钟，RPO 32 秒 | 13 分钟，RPO 41 秒 | 当前方案未达到 |
| Node.js 官方/受支持驱动 | 通过 | 通过 | 通过 |

Redis 方案因硬约束失败而淘汰，低延迟不能抵消该结果。

## PoC 证据

使用脱敏后的数据量、热点租户分布、top 12 查询与生产连接池参数进行 90 分钟峰值和 8 小时 soak 测试。两种关系型方案都达到接口 SLO。

MySQL 方案在补充联合索引后达到 12,000 QPS 的实测边界；以 1.5 安全系数计算，18 个月预测峰值仍在边界内。团队已有 MySQL failover、PITR 和升级自动化。PostgreSQL 方案的部分聚合查询更快，但需要迁移、重新建立值班能力和双栈观察，当前没有对应硬需求。

这些数字只用于示例，真实 ADR 必须附上数据集版本、压测脚本、原始指标和执行计划。

## 决定

保持 MySQL 8.4 为唯一订单事实源，完成索引、连接池和容量告警。暂不迁移 PostgreSQL，不使用 Redis 保存权威订单。

允许对订单详情热点开展 Redis cache-aside canary，但必须满足：

- MySQL 始终是事实源，缓存可整体删除和重建。
- 缓存超时后限流回源，具备请求合并和随机 TTL。
- 写库成功后执行可重试失效，接受短暂旧读的接口必须明确标注。
- 监控命中率、回源 QPS、旧读窗口、内存和大 key。

## 后果与新风险

优点是保留现有事务边界、恢复自动化和团队经验，避免迁移与双写。代价是部分 PostgreSQL 特有能力暂不可用，并继续承担 MySQL 容量治理。

若启用缓存，将新增缓存击穿、旧读、连接和 Redis 故障面；缓存故障不得阻止订单写入。

## 迁移、回退与退出

本次没有事实源迁移。缓存 canary 按 1% → 10% → 50% 流量推进；任何正确性差异、回源过载或错误率超限都关闭缓存读取，直接回到 MySQL。

每季度验证逻辑导出、PITR 和恢复时间。若未来触发迁移复审，新方案必须包含 snapshot + CDC 水位、shadow read、业务不变量对账、旧写入口 fencing 和回退资格关闭条件。

## 未决项

- 平台组在两周内补充 Redis 冷启动回源压测。
- 订单组在一个月内为峰值容量达到 70% 设置复审告警。
- 安全组确认缓存字段最小化和审计保留策略。
