# 前端发布事故 Runbook

## 触发条件

- 新 Release 的错误率、关键性能或业务成功率越过门禁；
- 静态资源、配置、Remote 或核心 API 出现区域性失败；
- 告警包含 Release、环境、路由、Cohort 和影响规模。

## 前五分钟

1. 指定 Incident Commander 和记录人，建立统一事件时间线；
2. 冻结相关发布，确认告警是否与最新 Release 或 Flag Cohort 对齐；
3. 优先止损：关闭 Flag、停止灰度、回滚静态入口或切换降级路径；
4. 验证 CDN 与 HTML 缓存是否真的让用户回到兼容版本；
5. 用用户影响指标验证恢复，而不是只看部署命令成功。

## 诊断上下文

- Release ID、Commit、部署时间与 Source Map 是否匹配；
- 错误指纹、首次出现时间、路由、浏览器和区域分布；
- Trace 是否显示前端、网关或下游服务失败；
- 新旧 Cohort 的错误率、性能和业务漏斗差异；
- 遥测采样、SDK 或 Collector 是否自身故障。

## 恢复以后

- 保留时间线、决策依据和未知项；
- 创建有 Owner 与截止日期的修复项；
- 补充能更早发现或自动止损的测试、门禁和 Runbook；
- 复盘系统条件与防线缺口，不把结论停留在个人失误。
